## 2.4.1 分数二进制数

> **Ch2 §2.4** · [章导读](../README.md) · 上节 [§2.3 整数算术](./section-2.3-整数算术.md) · 下节 [§2.4.2 IEEE 754](./section-2.4.2-IEEE754浮点表示.md)

> **核心主线：** 很多十进制小数 **无法** 用二进制精确存储 → IEEE 754 只能存 **近似值** → 运算产生 **舍入误差**。  
> （对照 §2.3：整数在固定位宽内可精确表示；小数从这里开始丢精度。）

---

### 二进制小数表示规则

小数点后第 \(i\) 位权重为 \(2^{-i}\)：

| 位 | 权重 |
|----|------|
| 第 1 位 | \(2^{-1} = 1/2\) |
| 第 2 位 | \(2^{-2} = 1/4\) |
| 第 3 位 | \(2^{-3} = 1/8\) |
| … | \(2^{-n}\) |

\[
0.b_1b_2b_3\ldots b_n = b_1\cdot\tfrac12 + b_2\cdot\tfrac14 + b_3\cdot\tfrac18 + \cdots + b_n\cdot\tfrac{1}{2^n}
\]

即：有限二进制小数 = **分母为 \(2^n\)** 的分数。

---

### 精确可表示 / 不可表示

#### 能精确：`0.625`

\[
0.625 = \tfrac58 = \tfrac12 + \tfrac18 = 1\cdot 2^{-1} + 0\cdot 2^{-2} + 1\cdot 2^{-3} = 0.101_{(2)}
\]

有限二进制、无循环 → **可精确存储**。

#### 不能精确：十进制 `0.1`

类比：十进制里 \(1/3 = 0.\dot{3}\) 无限循环。  
\(0.1 = 1/10\)，分母 \(10 = 2\times 5\) 含质因子 **5**；二进制分母只能是 \(2^n\)，**消不掉 5** → 转成二进制是 **无限循环**：

\[
0.1_{(10)} = 0.0\overline{0011}_{(2)}
\]

（反复 ×2 取整：`0.0001100110011…`）  
机器尾数位数有限 → 只能 **截断/舍入** 存近似值。

---

### 经典现象：`0.1 + 0.2 ≠ 0.3`

```c
/* Python / C 同理 */
0.1 + 0.2 == 0.3;   /* 常为假（float/double） */
```

1. `0.1`、`0.2` 各自进 IEEE 浮点时已被舍入成近似  
2. 两近似相加仍是近似，与 `0.3` 的近似存储 **未必相等**  
3. **不是** 编译器/语言 bug — 是二进制浮点的 **表示极限**

**HFT：** 价格/金额用整数 tick（定点数）或十进制库；热路径别拿 `double` 做「必须精确相等」的金额判断。

---

### 口述巩固 · 自测

1. 为什么 `0.1` 不能精确存成 binary float？  
   - \(0.1_{10}=1/10\)，分母含质因子 **5**  
   - 二进制小数 = 分母 \(2^n\) 的分数，构造不出含因子 5 的有限精确表示  
   - 转换后无限循环，尾数有限 → 只能截断/舍入近似
