## 2.4.4 舍入 (Rounding)

> **Ch2 §2.4** · [章导读](../README.md) · 上节 [§2.4.3 数字示例](./section-2.4.3-浮点数字示例.md) · 下节 [§2.4.5 浮点运算](./section-2.4.5-浮点运算.md)

---

### 为什么需要舍入

浮点数的尾数字段有限（float 23 位、double 52 位），但运算结果往往 **需要更多位** 才能精确表示：

- **对阶：** 加减法时小指数的数右移对齐，低位被移出
- **乘除：** 两个 23 位尾数相乘得 46 位，超出字段宽度
- **十进制→二进制：** `0.1` 这种数本身就是无限循环（→ [§2.4.1](./section-2.4.1-分数二进制数.md)）

超出字段宽度的多余位必须丢掉——但 **怎么丢** 有讲究，直接影响精度和误差累积。

---

### IEEE 754 四种舍入模式

| 模式 | 英文 | 规则 | 用途 |
|------|------|------|------|
| **向偶数舍入** | Round to Nearest, Ties to Even | 找最近的可表示值；**正中间** 时往 **偶数**（最低位 0）靠 | **默认**，数学计算 |
| 向零舍入 | Round toward Zero (Truncate) | 直接截断多余位 | `int` 转 float、C 的 `(int)` |
| 向下舍入 | Round toward −Inf (Floor) | 往更小的方向 | 区间算术下界 |
| 向上舍入 | Round toward +Inf (Ceil) | 往更大的方向 | 区间算术上界 |

> C99 提供 `fenv.h` 的 `fesetround()` 切换模式；默认是 `FE_TONEAREST`（向偶数）。

---

### 向偶数舍入详解（默认模式 · 重点）

**基本规则：**

1. 多余位 < 半个 ULP（最低精度单位的一半）→ **舍去**（向近的靠）
2. 多余位 > 半个 ULP → **进一**（向近的靠）
3. 多余位 **正好** = 半个 ULP（正中间）→ 往 **尾数最低位为 0**（偶数）的方向靠

> **为什么要「向偶数」？** 如果正中间总是向上进一，会产生 **系统性正偏差**（所有 .5 都变大）。向偶数舍入让一半的 .5 向上、一半向下，**长期统计上偏差抵消**——这对科学计算和金融累计至关重要。

**十进制类比（先理解原理，再推二进制）：**

| 原值 | 向偶数舍入（保留 1 位小数） | 解释 |
|------|---------------------------|------|
| 1.4 | 1.4 | 第二位 < 5，舍去 |
| 1.6 | 1.6 | 第二位 > 5，不变 |
| 1.5 | **2** | 正中间，2 是偶数 → 向上 |
| 2.5 | **2** | 正中间，2 是偶数 → 向下 |
| 3.5 | **4** | 正中间，4 是偶数 → 向上 |
| 4.5 | **4** | 正中间，4 是偶数 → 向下 |

> 对比「四舍五入」：1.5→2, 2.5→3, 3.5→4, 4.5→5——全部向上，**累积偏差为正**。向偶数把 2.5 和 4.5 拉下来，偏差抵消。

### 二进制向偶数舍入实例

**场景：** float 尾数 23 位，运算后多出 2 位 guard/round 位（`GR`），需要决定怎么处理。

**规则二进制版：** 看被丢弃部分的最高位（round bit `R`）和其余位（sticky bit `S`）：

| R | S | 动作 |
|---|---|------|
| 0 | × | **截断**（不到一半，舍去） |
| 1 | 0（其余位全 0） | **正中间** → 看保留的最低位：偶数(0)就不动，奇数(1)就进一 |
| 1 | 1（其余位非全 0） | **超过一半** → 进一 |

**例 1：** 尾数 `1.0110 1101...`，保留 4 位，多余 `1101...`

- 被丢弃部分 = `1101...`，R=1, S=1（非全0）→ 超过一半 → **进一**
- `1.0110` + 1 = `1.0111`

**例 2：** 尾数 `1.0110 1000...`（精确的中间值），保留 4 位，多余 `1000...`

- 被丢弃部分 = `1000...`，R=1, S=0 → **正中间** → 看保留最低位 `0`（偶数）→ **不进**
- 结果 = `1.0110`

**例 3：** 尾数 `1.0111 1000...`（精确的中间值），保留 4 位，多余 `1000...`

- 被丢弃部分 = `1000...`，R=1, S=0 → **正中间** → 看保留最低位 `1`（奇数）→ **进一**
- `1.0111` + 1 = `1.1000`

> 这三个例子覆盖了「舍去 / 中间偶数不变 / 中间奇数进一 / 超过一半进一」四种情况。Data Lab 的 `floatFloat2Int` 等题会考这个判断。

### 舍入在运算流程中的位置

| 运算 | 舍入发生时机 | 典型丢位原因 |
|------|-------------|-------------|
| 加/减 | 对阶后尾数加减、规格化后 | 小指数数右移对齐，低位被挤出 |
| 乘 | 尾数相乘后规格化 | 23×23=46 位，砍回 23 位 |
| 除 | 尾数相除后 | 商可能是无限循环 |
| int↔float | 类型转换 | `int` 的有效位 > float 的 23 位 |

**例：大 int 转 float 丢精度**

```c
int   x = 0x1000001;   /* 16777217 = 2^24 + 1，需要 25 位 */
float f = (float)x;    /* float 只有 24 位有效（1+23），最后那个 1 被舍掉 */
int   y = (int)f;      /* y = 16777216，丢了 1 */
```

> 这就是为什么 `2^24 + 1` 存不进 float——正好在精度边界上（→ [§2.4.3 量级与精度](./section-2.4.3-浮点数字示例.md#量级与精度记数量级)）。

### x87 扩展精度陷阱

| 寄存器 | 宽度 | 尾数位 |
|--------|------|--------|
| x87 FPU | **80 bit** | 64 位 |
| SSE (float) | 32 bit | 23 位 |
| SSE (double) | 64 bit | 52 位 |

**问题：** x87 用 80 bit 寄存器做中间计算，精度比声明的 `float`/`double` 高。编译器如果把中间值留在寄存器里 vs 落到内存（截断成 32/64 bit），**同一个表达式在不同优化级别下结果可能不同**：

```c
float a = 0.1f, b = 0.2f;
/* -O0: a+b 先存内存（32bit）再比较 → 可能 != 0.3f */
/* -O2: a+b 留在 80bit 寄存器 → 可能 == 0.3f */
if (a + b == 0.3f) { ... }   /* 结果依赖编译选项！ */
```

**缓解：**

| 方法 | 做法 | 代价 |
|------|------|------|
| 统一 SSE/AVX | `-mfpmath=sse`（x86-64 默认） | 无，推荐 |
| `-ffloat-store` | 强制中间结果落内存 | **慢**，仅调试用 |
| 不用 float 做账本 | 整数 tick / decimal 库 | 最佳实践 |

> x86-64 默认用 SSE，这个问题基本消失了；但跨平台或老代码仍要注意。GCC 加 `-fexcess-precision=standard` 可强制 IEEE 严格语义。

### HFT 铁律（常见）

| 场景 | 建议 | 原因 |
|------|------|------|
| 价格、PnL、风控阈值 | **整数 tick** 或 decimal 库 / `rust_decimal` | 舍入误差会累积成真金白银 |
| 指标、统计、信号 | double 可接受 | 知悉误差，不涉及资金精确性 |
| 相等比较 | **禁止** `a == b`；用 epsilon 或整数 tick 差 | 舍入让「应该相等」的值差 1 ULP |
| 累加大量小数 | Kahan 求和 / 整数定点 | 普通连加会「大数吞小数」 |
| 跨平台一致性 | 固定 SSE/AVX + `-fexcess-precision=standard` | 避免 x87 扩展精度导致结果不一致 |
