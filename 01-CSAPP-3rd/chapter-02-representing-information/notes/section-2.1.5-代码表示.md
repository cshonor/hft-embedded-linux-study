### 指令也是字节序列

**核心观念：** 机器指令在内存里 **和数据没有本质区别**——都是 0/1 的字节序列。同一段 `0xFF 0xE8 ...` 到底是指令还是数据，完全取决于 **谁来解释它**：

- 当 **PC（程序计数器）指向** 它，CPU 译码器把它当 **指令** 来执行
- 当程序用 `mov` / `load` 读它，就是 **数据**

这就是 **冯·诺依曼架构 (von Neumann)** 的根本特征：**存储程序**——指令和数据放在同一个内存里，靠 PC 和访问方式区分，而不是靠物理位置。

```
内存
┌──────────┐
│ .text    │ ← 机器码（PC 从这里取指）   E.g. 55 48 89 e5 ...
├──────────┤
│ .rodata  │ ← 只读数据（字符串常量等）
├──────────┤
│ .data    │ ← 已初始化全局变量
├──────────┤
│ .bss     │ ← 未初始化全局变量（运行时清零）
├──────────┤
│ heap ↑   │
│          │
│ stack ↓  │
└──────────┘
```

> **关键：** 段 (section) 划分是 **链接器/加载器** 贴的标签，内存里本质都是字节。理论上你可以 `jmp` 到 `.data` 里的字节——如果那些字节恰好是合法指令编码，CPU 会照执行（这就是 shellcode 注入的原理）。

---

### CPU 怎么区分指令 vs 数据：页权限

现代 CPU 靠 **MMU + 页表权限位** 在硬件层控制：

| 权限位 | 含义 | 典型段 |
|--------|------|--------|
| **R (Read)** | 可读 | .text, .rodata, .data |
| **W (Write)** | 可写 | .data, heap, stack |
| **X (Execute)** | 可执行 | **只有 .text** |

- 试图 **执行** 没有 X 权限的页 → CPU 触发 **页错误 (page fault)** → OS 杀进程（segfault / SIGSEGV）
- 试图 **写** 没有 W 权限的页 → 同样 segfault

> **查看进程的页权限：** `cat /proc/<pid>/maps`，每行末尾有 `r-xp`（只读可执行）、`rw-p`（读写不可执行）等标记。

---

### NX / DEP / W^X：阻止代码注入

早期 x86 没有页级 X 权限位（只有 R/W），所以 **任何可读的内存都可以被执行**——包括栈和堆。攻击者把 shellcode 塞进栈/堆的缓冲区溢出处，然后跳过去执行。

| 机制 | 全称 | 做法 |
|------|------|------|
| **NX** | No-eXecute | AMD64 / Intel EM64T 新增页表 X 位，标记不可执行页 |
| **DEP** | Data Execution Prevention | Windows 的 NX 实现（软件 + 硬件配合） |
| **W^X** | Write XOR Execute | **同一页要么可写、要么可执行，不能同时**——BSD/macOS 的策略 |

**W^X 的意义：** 攻击者就算能把 shellcode 写进内存（需要 W），那块内存也不能被执行（没有 X）；想让它可执行就得改成 X 权限，但那它就不可写了——没法动态填充 payload。

```
攻击链（无 NX）：              攻击链（有 W^X）：
溢出缓冲区写 shellcode    →    溢出缓冲区写 shellcode    ✓（W 页可写）
跳到 shellcode 执行       →    跳到 shellcode 执行       ✗（W 页无 X → segfault）
```

> 现代 JIT 编译器（V8、JVM）需要「写完机器码再执行」，所以用 `mprotect()` 临时切换权限：先 W 写入，再改成 X 执行（**不能同时 W+X**）。

---

### 代码注入防御的演进（了解）

| 代次 | 攻击 | 防御 |
|------|------|------|
| 1 | 栈溢出跳 shellcode | NX/DEP/W^X |
| 2 | ROP（跳到已有代码片段拼凑） | ASLR（地址随机化）+ 栈金丝雀 |
| 3 | 信息泄露 + ROP | CFI（控制流完整性）|

HFT 服务器一般全开 NX + ASLR + 栈金丝雀，编译加 `-fstack-protector-strong -fPIE`。

---

### HFT 视角

- **策略代码保护：** 交易策略是核心资产，常做成 `.so` 动态库，文件权限严格控制；运行时加载到只读+可执行的页
- **内存安全：** 热路径避免动态内存操作（减少 UAF/溢出风险）；用 `mlock()` 锁页防 swap
- **反调试/反注入：** 生产环境限制 `ptrace`、关掉不需要的 `dlopen` 路径
- **内核 bypass：** DPDK/AF_XDP 把网卡数据直接映射到用户态内存——这块内存是 **数据** 不是代码，但需要大页 (hugepage) + pinned memory
