## 2.1.3 寻址和字节顺序 (Byte Ordering)

> **Ch2 §2.1 一条线：** [§2.1.1 十六进制](./section-2.1.1-十六进制表示法.md) → [§2.1.2 数据大小](./section-2.1.2-数据大小与sizeof.md) → **§2.1.3 本节**

> **一句话先分清：** 大端/小端只规定 **「多字节数字在内存（或字节流）里的存放顺序」**；  
> 和「数据进来的顺序」**不能直接划等号** — 要分两层：**① 内存怎么排** · **② 网络怎么发/收**。

---

### 寻址 (addressing)

- CPU 通过 **地址总线** 访问内存；x86-64 为 **按字节寻址 (byte-addressable)** — **每个字节** 有唯一地址
- 变量 `int x` 占 **4 个连续字节** 的地址，例如 `0x1000`–`0x1003`
- `&x` = 对象 **起始（最低）字节** 的地址
- 指针算术：`char* p; p+1` 前进 1 字节；`int* q; q+1` 前进 `sizeof(int)` 字节 — 见 [§3.8 指针步长](../../chapter-03-machine-level-programs/notes/section-3.8-指针步长详解.md)

**寻址回答：** 数据在 **哪几个字节地址** 上。

### 字节顺序 (byte order / endianness)

多字节对象占多个连续地址时，**各字节在地址上的排列顺序**。

| | 英文 | 规则 | 典型 |
|--|------|------|------|
| **小端** | Little-Endian | **最低有效字节** 在 **低地址** | **x86-64 原生**（不可切换） |
| **大端** | Big-Endian | **最高有效字节** 在 **低地址** | **网络字节序**、多数 wire 协议 |

> **术语：** byte order / **endianness**

**字节序回答：** 同一数值的多个字节，**谁放在低地址**。

> **易混：** 寻址 ≠ 字节序。先确定占哪段地址，再谈 `0x78` 和 `0x12` 谁先谁后。

### 示例：`int x = 0x12345678`

拆成：高字节 `0x12` · `0x34` · `0x56` · 低字节 `0x78`；地址低→高：`addr0 … addr3`。

| | addr0 | addr1 | addr2 | addr3 |
|--|-------|-------|-------|-------|
| **大端**（网络序） | `0x12` | `0x34` | `0x56` | `0x78` |
| **小端**（x86） | `0x78` | `0x56` | `0x34` | `0x12` |

- 大端：书写顺序 ≈ 地址递增顺序，人读直观。  
- 小端：低位先放低地址 — x86 全系；对硬件加法器友好。

`0x11223344`：**小端** `44 33 22 11`；**大端** `11 22 33 44`。

---

### 「大端 = 数据进来的顺序？」—— 分场景

#### 场景 1：网络包（HFT 重点）— **这里可以这么说**

网络传输 **统一大端（network byte order）**：

- 发送：多字节数字 **先发高字节、后发低字节**  
- 接收：字节流到达顺序 = 大端内存布局  

例：发 `0x12345678` → 线上顺序 `12 → 34 → 56 → 78`。  
→ **在网络层，「数据流进来的顺序」= 大端字节序。**

#### 场景 2：x86 本地内存 — **不能划等号**

x86 **永远小端**。网卡/用户态缓冲里若仍是 **大端字节布局**，用 `*(uint32_t*)buf` 直接当本机 `int` → **数值错**。  
必须 `ntohl` / 显式解码： **wire 大端字节流 ≠ CPU 小端解析顺序**。

#### 场景 3：文件 / 自定义二进制

文件里字节顺序 = **写入方约定**，与「进来」无固定绑定；协议文档写死大端或小端即可。

| 概念 | 固定规则（HFT） |
|------|-----------------|
| **字节流收发顺序** | 网络：高字节先到 = **大端流入** |
| **CPU 内存解析** | x86：低字节在低地址 = **小端** → 收发数字要转换 |

---

### 为什么 x86 原生小端？（硬件 · 兼容 · HFT）

> 核心不是「好看」，而是 **加法器从最低位算起 + 少一层字节反转电路**；历史定型后 **不可切换**。

#### 1. 硬件：加减从最低字节开始进位

算 `0x12345678`：先碰低字节 `0x78`，进位再碰 `0x56`… 往高走。

| 小端内存 | 与加法器 |
|----------|----------|
| 低地址 = `0x78`（最先算） | 读出顺序 ≈ 运算顺序，**少做字节重排** |
| 高地址 = `0x12`（最后算） | 进位流水自然对齐 |

若大端：低地址先是高字节 → 进 ALU 前常要 **反转**，多电路、多延迟。HFT 共置机全是 x86，本机侧吃的就是这类「少一层」的收益。

#### 2. 历史：8086 定型，全系兼容

Intel **8086** 起即小端（便于与 8 位时代低半字用法衔接：16 位 `0x1234` 小端存 `34 12`，读低地址即低 8 位）。  
80386 → x86-64 **继承**，二进制兼容优先 → **永远小端、不可切**。

#### 3. 短类型：低地址即有效低半

如数值 `0x…ABCD`，有效 short 在低 16 位：

- **小端：** 低地址就是 `CD AB` → `movw (%rax), %dx` **无需额外偏移**  
- **大端：** 有效半字常在高地址 → 多偏移/多指令  

热路径里大量 `uint8_t`/`uint16_t` 状态、字段 → 少指令、少气泡。

#### 4. 乘除结果与内存

`imul`/`idiv` 等常把积/商的 **低位** 放在 `%rax`（高位 `%rdx`）。小端下把 `%rax` 直接 `movq` 到缓冲区起始，**低位自然落在低地址**；大端则更常要搅顺序再存。

#### 5. 和大端网络不冲突（取舍）

| | 目标 | 选端 |
|--|------|------|
| **本机 x86** | 运算延迟、电路简单 | **小端** |
| **网络** | 跨 x86/ARM/RISC-V 统一 | **大端**（高字节先发） |

行情价 `0x000186A0` 大端进缓冲 → 小端内存字节成 `A0 86 01 00`；直接 `movl` 进 `%eax` 会错 → **`ntohl` 一次**。  
本机一辈子省的反转 vs 收发包各转一次：**边界付费、内核路径免费**。

**极简：** 小端贴加法器与 8086 遗产；网络大端贴跨平台；HFT 在边界做 `ntoh*`。

### 字符串没有大小端（只有多字节整数才有）

> **口诀：单字节无大小端；多字节数值才有大小端。**

大小端前提：一个 **数值** 拆成多个字节后，谁放低地址。  
普通 ASCII / UTF-8 文本里每个 `char` 是 **独立 1 字节** — 没有「高低有效字节」，**不受 LE/BE 影响**。

```
"AB" → 'A'=0x41, 'B'=0x42
低→高地址永远：41 42（书写顺序）
x86 小端 / 某 ARM 大端：字符串字节序相同，不会颠倒
```

| | 有无 endian？ | 说明 |
|--|---------------|------|
| `char` 串 / FIX 标的代码等 ASCII | **无** | 按书写顺序存；**不要** `hton*` |
| `short`/`int`/`long`/指针 | **有** | 本机小端会翻字节；wire 大端要 `ntoh*` |
| `wchar_t` / UTF-16 码元 | **有（按「宽字符数值」）** | 如 `L'A'=0x0041` 在 x86 小端内存常为 `41 00` — 是 **宽整数** 被小端化，不是「字符串这个概念」有端序 |

**HFT：** FIX 订单号、symbol 等字符串字段 — 收发与本机顺序一致，**只拷贝字节**；价格/数量/时间戳等 4/8 字节整数才转字节序。

→ 字符串专节：[§2.1.4](./section-2.1.4-2.1.9-字符串布尔与C位操作.md)

### 和字符串的直觉对比（类比，不是定义）

| | 内存低→高 | 别误读成 |
|--|-----------|----------|
| `"1234"` | `31 32 33 34` | 「字符串是大端」— **错**；只是单字节序列按书写排 |
| 小端 `0x12345678` | `78 56 34 12` | 多字节 **整数** 才翻 |

### 网络序转换

```c
#include <arpa/inet.h>   /* Linux */
uint16_t n16 = htons(h16);   /* host → network（小端机上会翻字节） */
uint32_t n32 = htonl(h32);
uint16_t h16 = ntohs(n16);   /* network → host */
uint32_t h32 = ntohl(n32);
```

### HFT 必记

1. FIX / TCP 二进制行情等：**wire 大端**；本机 x86 **小端** → `hton*`/`ntoh*` 或 schema 生成代码  
2. ARM 可配端序，但 **网络开发一律按大端协议**；x86 **不可切换**  
3. **禁止** `write(fd, &m, sizeof m)` 裸发本地 struct  
4. 同机 mmap：须同 endian、同 ABI、同 padding  

→ [Ch 11 网络编程](../../chapter-11-network-programming/) · [11 UNP](../../../11-UNP-Vol1/)

**极简：** 网络层大端 ≈「流入字节顺序」；本机 x86 小端解析，**中间必须转换**。
