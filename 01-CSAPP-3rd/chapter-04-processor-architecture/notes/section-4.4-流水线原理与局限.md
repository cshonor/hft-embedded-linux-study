## 4.4 流水线的通用原理（4.4.1–4.4.4）

> [章导读](../README.md) · 上节 [§4.3 SEQ](./section-4.3-SEQ顺序处理器.md) · 下节 [§4.5 PIPE 与冒险](./section-4.5-PIPE流水线与冒险.md)

---

### 本节核心定位

| | |
|--|--|
| **§4.3 SEQ** | 单周期串行：一条指令走完五级才下一条；硬件大量闲置、极慢 |
| **§4.4** | 引入 **流水线 (Pipeline)** — 现代 x86/ARM 提速的核心思路之一 |
| **§4.5** | 三类 **冒险** 如何产生、如何缓解（本节只铺概念） |

> **一句话：** 把 SEQ 的串行五级改成 **多指令重叠并行**，吞吐大涨；同时抛出「冲突会 stall / 冲刷」→ 交给 §4.5。

---

### 核心思想：工厂流水线类比

指令仍拆成固定五级：**F → D → E → M → W**。  
每级有独立硬件；**同一时钟周期内，不同指令处在不同级，同时干活**：

```
时钟1：  I1[F]
时钟2：  I1[D]  I2[F]
时钟3：  I1[E]  I2[D]  I3[F]
时钟4：  I1[M]  I2[E]  I3[D]  I4[F]
时钟5：  I1[W]  I2[M]  I3[E]  I4[D]  …
```

流水线 **填满** 后，理想情况 **每个时钟周期完成 1 条指令**（吞吐拉满）。  
对比 SEQ：一条指令独占整机一整段长周期。

#### 关键硬件：流水线寄存器（锁存器）

级与级之间插入时序寄存器：`F/D`、`D/E`、`E/M`、`M/W` — 锁存本级临时数据与控制信号。

**没有它们，前后指令信号会搅在一起，无法分开并行。** → [§4.2 时序电路](./section-4.2-HCL逻辑与组合电路.md) 的延伸。

---

### 流水线带来的优势

| 优势 | 说明 |
|------|------|
| **吞吐↑** | SEQ 相当于「很久才出一条」；填满后理想 **每拍一条**（相对五级，吞吐上限可接近 **×段数**） |
| **时钟可更短 / 易提频** | SEQ 时钟须盖住「最慢的那一整条路径」；流水线每级组合路径更短，周期可压短 |
| **硬件少闲置** | 取指、译码、ALU、内存口可同时服务不同指令 |

**吞吐 vs 延迟（别混）：**

- **吞吐：** 单位时间完成多少条 — 流水线主攻这个  
- **单条延迟：** 仍 ≈ 各段之和（甚至略增，因锁存开销）；只是多条重叠掩盖

```
示意（原书数量级）：无流水线 3 条 × 300ps = 900ps
三段流水线填满后：开头填管 + 每拍出货 → 总时间远短于串行累加
```

**加速比上界：** 段数 N、指令很多时，理想加速 → N；头尾气泡浪费。

---

### 局限：天然会有冲突 → **冒险 (hazard)**（§4.5 详解）

并行一开，就会「踩脚」→ CPU **stall（停顿）** 或 **flush（冲刷）**，吞吐暴跌。§4.4 只立概念：

| 冒险 | 直觉 | 例子 / HFT 味道 |
|------|------|-----------------|
| **数据冒险** | 后指令要用前指令结果，前指令还没写回 | `addq %rax,%rbx` 后立刻 `subq %rbx,%rcx` |
| **控制冒险** | `jmp`/`jXX`/`call`/`ret` 改 PC；已预取的顺序指令可能作废 | **冲刷流水线**；HFT 写无分支 / 可预测分支的硬件根因 |
| **结构冒险** | 同拍争抢同一端口（如数据内存） | 硬件口不够 → 暂停一条 |

带反馈的系统更难流水：PC 依赖上条结果、写回值下条要用 → 必须 **检测 + 转发 + 预测**（§4.5）。

其他工程限制：段延迟不均（最慢段定拍）、流水线寄存器面积与延迟开销。

---

### SEQ vs 流水线（对照）

| | **SEQ** | **流水线** |
|--|---------|------------|
| 执行 | 串行，一条走完再下一条 | 多指令重叠 |
| 流水线寄存器 | 无（级间） | **有**（F/D…M/W） |
| 停顿 / 冲刷 | 基本无此概念 | **有** 冒险 |
| 速度 | 慢、资源闲置 | 更快，但有损耗场景 |

分段硬件本身仍是 §4.2 的 MUX、ALU、寄存器、内存 — 只是 **复制/分段 + 锁存** 后重叠使用。

### 与 HFT、全书链路

1. **过渡章：** 讲清「为何快」+ 抛出「冒险会慢」→ [§4.5](./section-4.5-PIPE流水线与冒险.md)  
2. **HFT 优化目标 ≈ 少让流水线停 / 被冲刷：**  
   - `inline` 少 `call`/`ret` → 减控制冒险  
   - 拆数据依赖、避免紧挨 load-use → 减数据冒险  
   - 查表 / `cmov` 代乱 `if` → 少冲刷  
3. **观测：** `perf` 里 IPC&lt;1、branch-misses — 不全是分支，还有 cache / 资源冲突  

→ 软件侧减冒险：[Ch 5](../../chapter-05-optimizing-performance/)
