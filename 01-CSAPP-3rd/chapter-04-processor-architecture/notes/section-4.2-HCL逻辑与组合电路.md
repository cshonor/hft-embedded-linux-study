## 4.2 逻辑设计与 HCL（4.2.1–4.2.5）

### 本节核心定位

| 节 | 视角 |
|----|------|
| **§4.1** | **指令**（软件：CPU 能识别什么命令） |
| **§4.2** | **数字电路**（硬件：内部怎么算出数据与控制信号） |

用教材自定义 **HCL（Hardware Control Language，硬件控制语言）** 描述电路 — **工业界不用**，纯教具（类似 Y86）。  
目标：看懂组合/时序积木，为 **§4.3 SEQ**、后面流水线 CPU 打电路基础。

**教具 ≠ 没用（与 §4.1 同理）：**

| | 载体（可丢） | 原理（要留） |
|--|--------------|--------------|
| **Y86** | 虚构 ISA / 模拟器 | 寄存器、栈帧、五级流程、CC、访存 → 套 **x86-64** |
| **HCL** | 书本语法 | **MUX、ALU、时序寄存器** → 工业用 **Verilog/VHDL** 写同类电路 |

别只看「语法没用」就忽略底层知识。

---

### §4.2 零件总清单（别只记住「全加器 + MUX」）

两大类：**组合**（无记忆）+ **时序**（有时钟、能存）+ 描述工具 **HCL**。

#### 组合逻辑

| 零件 | 作用 |
|------|------|
| **MUX** | 2/4/8 选 1；`sel` 选哪路输出 — CPU 到处都是 |
| **ALU** | 多位全加器拼成 64 位加减 + 与/或/异或；**内部再有 MUX** 选结果；更新 **ZF/SF/OF**（Y86 **无 CF**） |
| **位截取 / 拼接** | 拆变长指令：icode、寄存器号、立即数 |
| **与/或/非门** | MUX、加法器的底层砖块 |

#### 时序逻辑（拼 SEQ **必不可少** — 光靠 MUX+加法器不够）

| 零件 | 作用 |
|------|------|
| **寄存器堆** | **15**×64 位（`%rax`～`%r14`）；读/写程序变量 |
| **PC** | 下一条指令地址 |
| **CC** | 存 ALU 算出的标志，供分支 |
| **数据存储器** | `push`/`pop`/`mrmovq`/`rmmovq` 等访存 |

#### HCL

画电路用的教学伪代码；工业不用。

---

### 加法器 + MUX ≠ SEQ（核心问答）

**问：** 只有加法器和 MUX，能不能拼出 SEQ？  
**答：不能。** 缺三大块：

| 缺什么 | 为什么致命 |
|--------|------------|
| **时序存储**（寄存器堆、PC、内存、CC） | MUX/加法器是「纯算开关」，算完不留。没有 PC/寄存器/内存，**跑一步数据就丢**，无法连续执行多条指令 |
| **指令字段拆分电路** | 变长指令要拆操作码/寄存器/立即数 — 大量位拼接，不只是 ALU |
| **全链路译码控制** | 每条 `add`/`jmp`/`push`/`call` 要生成多根控制线，分别管 MUX、ALU、读写使能 — 不是一个 ALU 搞定 |

**正确等式：**

```
MUX 集群
+ ALU（全加器 + 逻辑门 + 结果 MUX）
+ 位拆分组合电路
+ 译码控制信号
+ 时序存储（PC / 寄存器堆 / 内存 / CC）
= SEQ 完整 CPU
```

**通俗类比：**

| | 像什么 |
|--|--------|
| **MUX + 加法器** | 简易计算器：输入两数算一下，**算完不记账** |
| **SEQ** | 计算器 + **笔记本**（寄存器/内存）+ **记事本页码**（PC）+ **判断开关**（CC 做 if） |

计算器只是 CPU 里负责算的一小块，**不是整机**。

**学习逻辑：** §4.2 拆零件 → [§4.3](./section-4.3-SEQ顺序处理器.md) 全部接线拼成能跑 Y86 的串行 CPU。

---

### 两大电路分类（展开细节）

#### 1. 组合逻辑（无记忆：输入一变，输出随之变）

没有存储元件，只靠逻辑门（与/或/非）与 **MUX** 等运算，**不保存状态**。教材重点：

| 部件 | 干什么 |
|------|--------|
| **多路选择器 mux** | 按控制信号从多路输入选 **1** 路输出。例：2 选 1 — 控制=0 出 A，=1 出 B |
| **ALU** | 加法器 + 逻辑门 + mux；按操作码做 **加/减/与/异或**（Y86 仅这四类） |
| **位提取 / 拼接** | 把指令字节拆成 icode、寄存器号、立即数等 — **变长 Y86 取指译码**靠组合逻辑拆字段 |

---

### MUX：最基础、无处不在 — 但不能单独说「最重要」

**结论：** MUX 是 CPU 里 **数量最多、贯穿全通路** 的开关/分流器；但 CPU 是组合系统，**MUX / ALU / 寄存器（含 PC）平级核心，缺一不可**。

| 部件 | 角色 | 没了会怎样 |
|------|------|------------|
| **MUX** | 数据/控制信号的 **开关、分流** | 无法切换数据源、难做分支/可编程路径，只能死板跑固定线路 |
| **ALU** | **真正计算**（加减、位运算） | 做不了运算，失去核心算力 |
| **寄存器堆 / PC** | **保存运行状态** | 算完就丢，无法连续执行 |

学 §4.2 **先吃透 MUX** — 后面 SEQ/PIPE 图上满是选择框，看懂开关就能读通数据通路。

**为何到处都是（典型场景）：**

1. **ALU 输入 MUX** — 选寄存器 / 立即数 / 转发值等送入运算  
2. **ALU 功能选择** — 按操作码选加 / 与 / 异或…（内部也是 mux 一类选择）  
3. **写回 MUX** — 选 ALU 结果还是内存读值写回寄存器堆  
4. **PC MUX** — 顺序下一条 `valP`，还是跳转 / `call` / `ret` 目标地址  

五级流水线每一级都有大量 MUX 切换信号；**没有 MUX，电路无法按指令切换数据来源**。

**HFT 补充：** 真 x86 数据通路上也是海量 MUX；每次选择有 **传播延迟**，叠在关键路径上。流水线里争用同一通路/端口（结构冒险、转发路径）时，你看到的是「等一拍」— 硬件侧往往就是 **共享资源 + 选择网络** 在打架。低延迟优化（少依赖、少畸形争用）对应的就是少踩这些路径。

### MUX 靠什么决定输出？—— **控制信号 `sel`**

拿最简 **2 选 1**：

| 端口 | 含义 |
|------|------|
| `in0` / `in1` | 两路数据输入 |
| `sel` | **1 根选择线**（来自指令 **译码** 产生的控制信号） |
| `out` | 选中的那一路 |

| `sel` | `out` |
|-------|-------|
| 0 | `in0` |
| 1 | `in1` |

电路直觉：`sel` 控制与门开关 — 一路导通、一路截断，再经或门合并。  
**一句话：** MUX 输出由外部送来的 **`sel` 唯一确定**；`sel` 来自对指令的 **译码解析**。

**多路扩展：**

| MUX | 控制线根数 | 例 |
|-----|------------|-----|
| 2 选 1 | 1 | `sel=0/1` |
| 4 选 1 | 2 | `00/01/10/11` → 四路 |
| 8 选 1 | 3 | … |

CPU 里大量 4 选 1、8 选 1，靠多根译码线组合精准挑通路。

```hcl
bool out = (a && b) || !c;
word out = a ? b : c;   // 字级 2 选 1：条件即 sel
```

### ALU：不只是加法器 + 内部也有 MUX

**纠正：** ALU ≠ 单纯加法器。

| 内部积木 | 作用 |
|----------|------|
| **多位加法器**（教材常讲超前进位等） | 加减（减 = 加补码一类技巧） |
| **逻辑门** | 与 / 或 / 异或等 |
| **结果 MUX** | 按 **运算控制信号** 放出哪一种结果 |

**Y86 四运算场景（示意）：** 译码生成控制码 → ALU 内 MUX 选路：

| 指令 | 控制直觉 | MUX 放行 |
|------|----------|----------|
| `addq` / `subq` | 控制选「加法器通路」 | 算术结果；其他逻辑结果被屏蔽 |
| `andq` | 切换到「与」通路 | 与运算结果；加法器输出丢弃（不接到 out） |
| `xorq` | 切换到「异或」通路 | 异或结果 |

（真机控制编码是多 bit；这里用「信号 0/1/2…」帮助建立直觉。）

**条件码（衔接 §4.1）：** 算术/逻辑结果再生成 **ZF / SF / OF**（Y86 **无 CF**）。→ [§4.1](./section-4.1-Y86-64-ISA.md)

一位全加器：`Sum = A⊕B⊕Cin`，`Cout = (A&B)|(Cin&(A⊕B))` — 串联成多位加法器 ⊂ ALU。→ [§2.3](../../chapter-02-representing-information/notes/section-2.3-整数算术.md)

**整条链路串起来：**

```
Y86 指令
  → 组合电路译码产生 sel / ALU 控制
  → 输入 MUX 选操作数
  → 加法器与逻辑单元并行算出多种候选
  → 结果 MUX 按控制放出对应一路
  → 写回寄存器 / 内存（再经写回 MUX 等）
```

#### 2. 时序逻辑（有记忆：时钟边沿才更新）

带存储单元；**时钟触发**才改值。CPU 里「会保存数据」的硬件属这类：

| 部件 | 作用 |
|------|------|
| **寄存器堆** | Y86：**15** 个 GPR（`%rax`～`%r14`，**无 `%r15`**）；按编号读/写 64 位 |
| **PC、CC、Stat** 等 | 指令推进后锁存新 PC、新条件码、状态 |
| **内存** | 模型里作时序器件：读写控制 +（写）时钟才更新存储内容 |

> **纠正口述易混：** 寄存器堆不是「16 个到 `%r15`」——与 [§4.1](./section-4.1-Y86-64-ISA.md) 一致为 **15**。

| | 组合逻辑 | 时序逻辑（寄存器等） |
|--|----------|----------------------|
| 行为 | 电平稳定后输出有效 | **时钟上升沿** 采样写入 |
| 记忆 | 无 | 有 |

**处理器 = 组合逻辑块 + 寄存器（+ 时钟）**。真芯片还有 setup/hold；Y86 模拟器多半抽象掉。

### HCL 是干嘛的？

简化伪代码，描述电路信号；**不是 Verilog/VHDL 替代品**，不上生产线。

```hcl
// 组合：集合成员 → 多路选择（opcode 译码表的味道）
word srcA = [
  icode in { IRRMOVQ, RMMOVQ, ... } : rA;
  icode in { IRMMOVQ, ... } : RNONE;
  1 : RNONE;
];

// 时序：reg 表示时钟更新的存储
```

作用：避开 HDL 复杂度，快速看 **控制信号怎么从 icode 流到 mux/ALU**。

---

### HCL 是 DSL，但和 TOML 不是一类

**DSL** = Domain-Specific Language：只解决狭小领域问题，不是通用编程语言。  
CSAPP 的 **HCL 确实是 DSL** — 领域是 **数字电路 / 硬件信号选择**。

但和 Cargo 的 **TOML** 同属「DSL」时，**用途与执行逻辑天差地别**：

| | **HCL（CSAPP）** | **TOML（如 Cargo.toml）** |
|--|------------------|---------------------------|
| **领域** | 数字电路：mux、ALU、信号分支、寄存器更新 | 静态配置：键值、数组、表 |
| **类型** | **逻辑执行型** — 表达「输入如何算出输出」 | **数据声明型** — 只存数据，无运算 |
| **有无计算** | 有判断、选择、位级信号运算 | **无** 分支/运算；只有 `name = "xxx"` 一类 |
| **更贴切类比** | 极简阉割版 **Verilog / VHDL** | **JSON / YAML**（配置存储） |

```hcl
// HCL：能表达动态选择（电路味道）
out = [
  ctrl : val1;
  1    : val2;
];
```

```toml
# TOML：只能声明静态数据
name = "my-crate"
version = "0.1.0"
```

> **易混同名：** HashiCorp 也有叫 **HCL** 的配置语言（Terraform 等）— 更接近 TOML/声明式配置，**不是** CSAPP 这套硬件控制语言。本书 HCL = 教学用电路描述。

**学习要求：** 工业界写硬件用 **Verilog / SystemVerilog** 等；本书 HCL **只读懂 mux 写法即可**，不必手写背诵 — 它是画图讲 CPU 的教具。

→ 数字电路补课书单 / 视频 / Logisim：[补充 · 数字电路学习资料](./section-补充-数字电路学习资料.md)

### 和前后章、HFT 的关联

1. **← §4.1：** 指令定义「要做什么」；本节讲「电路如何实现」  
2. **→ §4.3 SEQ：** 整颗 SEQ 就是本节 **组合 + 时序** 拼起来的  
3. **HFT 直觉（不必写 HCL）：**  
   - 运算延迟 ≈ 组合逻辑 **传播延迟**；路径越长，单周期越难做短  
   - 后面 **结构冒险** = 硬件资源（端口/ALU）冲突 → 电路侧争用  
   - 读住一句即可：**控制信号在时钟边沿锁存** → 流水线为何插寄存器、stall 为何浪费周期

### 最简总结

§4.2 不讲业务软件，只拆 CPU 数字电路：

1. **组合** — mux、ALU、拆字段：算、选数据  
2. **时序** — 寄存器堆、PC、CC、内存：存状态  
3. **HCL** — 教学描述语言 → 为 SEQ 打基础  

---

### 口述巩固 · 自测

1. **组合 vs 时序差在哪？** — 有无记忆；是否必须等时钟更新  
2. **mux / ALU / 寄存器谁「最重要」？** — **平级核心**；MUX 数量最多、先吃透；无 ALU 不能算，无寄存器不能存状态  
3. **2 选 1 的 `sel=0/1` 各出哪路？** `sel` 从哪来？ — 译码控制信号  
4. **ALU 只有加法器吗？** — **否**；加法器 + 逻辑 + **结果 MUX**  
5. **HCL 工业界用吗？** — **不用**；教具。逻辑型 DSL（≈迷你 Verilog），不是 TOML 配置型。原理真、语法假  
6. **加法器+MUX = SEQ 吗？** — **否**；缺 PC/寄存器堆/内存/CC、缺拆指令与全链路译码  
7. **和 SEQ 的关系？** — SEQ = 本节 **全部** 积木接线，不是只焊 ALU
