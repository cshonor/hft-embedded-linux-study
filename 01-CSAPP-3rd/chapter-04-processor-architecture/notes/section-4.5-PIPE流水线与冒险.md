## 4.5 Y86-64 流水线 PIPE（4.5.1–4.5.10）

> [章导读](../README.md) · 上节 [§4.4](./section-4.4-流水线原理与局限.md)  
> **Ch4 / 全书 CPU 硬件最核心一节（HFT）：** 搭完整 PIPE + 拆解三类冒险 + 转发 / stall / flush。  
> 无分支、`inline`、拆依赖 — 本质都是 **少 stall / 少冲刷**。

---

### 本节定位

| 节 | 角色 |
|----|------|
| **§4.4** | 流水线为何快；顺带点出冒险会 stall |
| **§4.5** | 完整 **PIPE**；三类冒险是什么、怎么拖慢、硬件怎么补、软件怎么躲 |

**流水线核心痛点：** 多指令并行 → 数据 / 硬件 / 分支互相打架 → 只能 **stall** 或 **flush**。每白停一拍，就多一份延迟；HFT 纳秒级损耗大量来自这里。

> **一句话：** 搭五级 PIPE，讲清数据/控制/结构冒险与旁路、停顿、冲刷 — **底层性能优化的硬件理论根基**。

---

### 4.5.1–4.5.3 PIPE 硬件结构（相对 SEQ / §4.4）

| 部件 | 作用 |
|------|------|
| **流水线锁存器** `F/D` `D/E` `E/M` `M/W` | 隔开各级，保存数据与控制信号 |
| **转发旁路 + MUX** | 解决多数 **数据冒险** 的核心电路 |
| **stall / bubble（flush）控制** | 检测冒险；暂停或插入气泡清空错误路径 |
| 底层积木 | 仍是 [§4.2](./section-4.2-HCL逻辑与组合电路.md) 的 MUX、ALU、寄存器堆、内存 |

- **SEQ+：** 把 PC 计算挪到更合适的位置，为预测留结构  
- 每段读上一段锁存值（信号在级间「接力」）

---

### 三大冒险完整拆解（本节核心）

#### 1. 数据冒险（最常见：指令间依赖）

后指令要用前指令结果，但前指令还没 **写回**，寄存器尚未更新：

```asm
addq %rax, %rbx   # 结果将来写入 %rbx
subq %rbx, %rcx   # 立刻读 %rbx — 若还没就绪 → 冒险
```

**硬件对策：**

| 手段 | 何时 |
|------|------|
| **转发 (forwarding / bypass)** | 从 E/M 等阶段把结果 **旁路** 给后指令，多数依赖 **不必等 W** |
| **stall** | **无法转发** 时暂停 1～若干拍 |

**经典考点 — load-use：** `mrmovq` 的数据要到 **M** 才出来，紧接着用 → **通常无法转发，必须 stall**：

```asm
mrmovq 8(%rax), %rbx
addq %rbx, %rcx    # 需停顿
```

**HFT：** 热循环避免 **load 后立刻依赖**；拆开依赖链 / 让编译器重排（`-O` 常做）。

#### 2. 控制冒险（分支冒险 · HFT 重点规避）

`jmp` / `jXX` / `call` / `ret` 改 PC；取指端往往已按 **顺序** 预取。  
要等执行阶段才知是否跳转：

- **跳了 / 预测错** → 已进管的错误路径指令作废 → **flush / bubble**，白烧多拍  
- 每次乱 `if`、深 `call` 都可能冲刷 → 延迟暴涨  

**缓解：**

| 侧 | 手段 |
|----|------|
| **硬件** | 分支预测（真 x86：BTB、历史表…）；Y86 书中常用简单策略（如预测 taken → `valC`） |
| **软件** | 无分支 / 查表 / `cmov`；热路径 **`inline` 消 `call`/`ret`**（减控制冒险） |

**HFT ↔ 真 CPU：** `perf` 的 `branch-misses` = 预测失败代价；热路径分支尽量 **可预测**。

#### 3. 结构冒险（硬件资源争抢）

同拍多条指令抢同一部件（如都要走数据内存口）→ 端口不够 → **stall** 其中一条。

Y86 教学模型里常弱化；真机常用 **I-cache / D-cache 分离**、多端口等缓解。

| 类型 | 原因 | 对策（概） |
|------|------|------------|
| **结构** | 资源端口冲突 | 停顿、双端口、分离缓存 |
| **数据** | RAW 真相关 | **转发** + 必要时 **stall** |
| **控制** | 分支/`ret` 改 PC | **预测** + 错则 **bubble** |

---

### 4.5.6–4.5.8 异常与控制逻辑

- 异常指令标记 **Stat**，后续 **bubble**，避免错误写回  
- **stall 条件：** load-use、`ret` 等（书中控制表）  
- **bubble：** 预测失败、异常 — 插入等价 `nop` 的气泡  

### 4.5.9 性能

- **CPI** ≈ 1 + 停顿占比；分支误预测、load-use 频率直接拉低 **IPC**  
- **HFT：** `perf stat` 看 `instructions` / `cycles` / `branches` / `branch-misses` → 结合 [Ch5](../../chapter-05-optimizing-performance/) 改热路径

### 冒险对 HFT 的现实意义（必记）

1. **`-O3` 有天花板** — 编译器做不到替你消掉所有冒险（对照 [§5.1](../../chapter-05-optimizing-performance/notes/section-5.1-优化编译器的能力和局限性.md)）  
2. **学硬件是为了改代码少停顿：**  
   - 拆连续依赖 → 少数据冒险 stall  
   - 查表 / 少乱分支 → 少冲刷  
   - **`inline`** → 少 `call`/`ret` 控制冒险  
   - 结构体 / 缓存行友好 → 少畸形访存引发的等待  
3. 市面低延迟手册里的套路，底层理论大量落在 **本节三类冒险**
