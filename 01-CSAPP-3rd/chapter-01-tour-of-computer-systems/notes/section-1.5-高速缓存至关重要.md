## 1.5 高速缓存至关重要

### 为何 cache 是性能第一公民

- CPU 与主存速度差距 **持续拉大**（「内存墙」）
- **缓存 (cache)** — 由 SRAM 构成的小而快的存储，自动保存主存中 **最近用过** 的数据副本
- 典型三级：**L1（指令/数据）→ L2 → L3（LLC）→ 主存**

| 命中 | 含义 | 延迟量级（直觉） |
|------|------|------------------|
| L1 hit | 数据已在 L1 | ~1 ns 级 |
| L2/L3 hit | 逐级更慢 | 数 ns ~ 十数 ns |
| **cache miss** | 需访问 DRAM | ~50–100+ ns |
| 跨 NUMA | 远程节点 DRAM | 更慢 + 不一致风险 |

---

### LLC 是什么（纠正「L3C」误区）

**LLC = Last Level Cache**，直译：**最后一级缓存**。

| | |
|--|--|
| **含义** | CPU 片上所有 cache 里，**离内存最近、最底层那一层**；再 miss → 访问 **DRAM** |
| **主流桌面/服务器** | 三级缓存（L1+L2+L3）→ **L3 = LLC**（同一块硬件、两种叫法） |
| **为何不叫 L3C** | **L3** 只是层级编号（Level-3）；**LLC** 是 **功能定位名** — 强调「进内存前最后一道关」。高性能 / 服务器 / **HFT** 文档统一写 **LLC** |

**一句话分清 L3 vs LLC：**

| 名称 | 命名角度 |
|------|----------|
| **L3** | 按层级编号（1 → 2 → 3） |
| **LLC** | 按功能（最后一级） |

- 普通三级 CPU：**L3 = LLC**  
- 极少数只有两级（L1+L2）时：**L2 = LLC**

---

### L1 · L2 · LLC(L3) 存什么、归谁

#### 1. L1（单核私有，拆两块）

| | 存什么 |
|--|--------|
| **L1I** | **只存指令**（机器码：`mov`/`add`/`call`…） |
| **L1D** | **只存数据**（变量、数组、临时值） |

**为何拆分：** 取指通路与数据通路 **并行**，互不堵；延迟最低。

#### 2. L2（单核私有，统一混存）

- L1 淘汰下来的 **指令 + 数据** 混放  
- **本核私有** — 定位：L1 的后备；其他核 **不直接用你的 L2**

```
执行单元
├─取指令 → L1I ──┐
└─读写数据 → L1D ─┘
                  ↓
                L2（指令+数据混放 · 本核私有）
                  ↓
              LLC = L3
```

#### 3. LLC = L3（全核共享，混存）

| 典型内容 | |
|----------|--|
| 各核 L2 淘汰后仍可能复用的冷/热混合行 | |
| **多线程共享** 全局数据（跨核交互、共享变量） | |

**特点：** 全核并联共用；片上终点 — **miss 必下 DRAM**。

```
Core0 [L1I+L1D] → L2 ──┐
Core1 [L1I+L1D] → L2 ──┤
Core2 [L1I+L1D] → L2 ──┼── LLC(L3) 全核共享
Core3 [L1I+L1D] → L2 ──┘
                         ↓ miss
                       DRAM
```

#### 热度 · 容量 · 归属对照

| 层级 | 别名 | 内容 | 归属 | 热度 | 容量 |
|------|------|------|------|------|------|
| **L1I / L1D** | 一级 | 指令 / 数据 **分开** | 单核私有 | 最热 | 最小（数十 KB） |
| **L2** | 二级 | 指令+数据混 | 单核私有 | 次热 | 中等（数百 KB～数 MB） |
| **L3 = LLC** | 最后一级 | 指令+数据混 | **全核共享** | 相对更冷、共用 | 最大（数十 MB） |

### 局部性（locality）— 程序行为决定 miss 率

1. **时间局部性** — 刚访问过的，很可能很快再访问（循环变量、热结构体）
2. **空间局部性** — 刚访问地址附近，很可能接着访问（数组顺序扫描）

**编译器/程序员能做的：** 数据结构布局、SoA vs AoS、对齐、避免冷路径污染 cache line。

**HFT 高频场景：**

- **Order book / ring buffer** 顺序访问 → 空间局部性好
- **指针 chasing、链表跳来跳去** → miss 多，P99 抖
- **伪共享 (false sharing)** — 两线程改同一 cache line 不同字段 → 行乒乓（→ [Ch 6](../../chapter-06-memory-hierarchy/)、[Ch 12](../../chapter-12-concurrent-programming/)）
- **perf `cache-misses` / `perf c2c`** — 生产验证（→ [14-Systems-Performance Ch 13](../../../15-Systems-Performance-2nd/chapter-13-perf/)）
- **LLC miss** — 热路径上常比 L1/L2 miss 更贵；关注共享数据布局、避免无意义的跨核踩 LLC

- 常见 **64 字节一行** — 读 1 字节可能整行载入
- **预取 (prefetch)** 硬件/软件可隐藏部分延迟
