## 5.1 优化编译器的能力和局限性

> **Ch5 §5.1** · [章导读](../README.md) · 上节 — · 下节 [§5.2 →](./section-5.2-表示程序性能.md)

---

**编译器常能做：**

- 常量折叠、死代码消除、内联、强度削减（`*8` → `<<3`）
- 寄存器分配、循环不变量外提、公共子表达式消除
- 自动向量化、循环展开（`-O3` / `-funroll-loops`）

**编译器不敢/不能做（除非你帮它）：**

| 障碍 | 例子 |
|------|------|
| **指针别名** | `*x` 与 `*y` 是否同一地址未知 → 不敢乱重排 store |
| **全局可见副作用** | 函数可能改内存 → 不敢删「看似多余」调用 |
| **跨编译单元** | 无 LTO 时不知道 `inline` 候选 |
| **违反严格别名** | `int*` 与 `float*` 互转 — UB |
| **算法/结构级** | 不会替你把深 if-else 改成无分支查表；不会重排结构体消 cache miss |
| **ABI / 屏障** | 无锁队列、内存屏障、参数个数压栈 — 要人按 [Ch3 调用约定](../../chapter-03-machine-level-programs/notes/section-3.7-过程与栈帧.md) 写 |

**帮编译器：**

```c
restrict  // C99：承诺指针不重叠
__restrict__
#pragma GCC ivdep   // 谨慎：声称无依赖
static inline + 同文件
__attribute__((noinline))  // 约束过度内联伤 icache
-fno-strict-aliasing  // 妥协，非首选
```

#### 灵魂问题：既有 `-O3`，为何还要学 Ch3 机器码 / Ch4 流水线？

**`-O3` = 通用规则优化，有天花板。** 全员 HFT 都会开 O3 — 那只是门票。拉开差距的是 O3 **覆盖不到** 的手动优化与排障能力。

| O3 做不到 / 做不好 | 你要会什么 | 对应 |
|--------------------|------------|------|
| 分支最多重排，**不会**改成无分支查表 | 看汇编 + 控制冒险 → 手写消分支 | Ch3 + [Ch4](../../chapter-04-processor-architecture/README.md) |
| **不懂** 你的 cache 布局 | 结构体对齐、数组访问顺序、缓存行 | Ch3 数据布局 + Ch6 |
| 多线程屏障 / 寄存器传参习惯 | ABI、少压栈、正确屏障 | Ch3 §3.7、并发章 |
| 过度内联/展开 → **icache 变慢** | 读 `objdump`，加 `noinline` / 限展开 | 本节 + 工具链 |
| 「无副作用」计算被删光 | 懂 volatile/屏障/可观测副作用 | Ch5 后续节 |
| 尾部延迟毛刺 | `perf` stall + 对汇编定位 | Ch3 读码能力 |

**一句话：** 普通业务不用全仓 O3 — CPU 不是瓶颈，编译慢、调试难、收益体感近零（→ [Ch3 §3.2.1 对照表](../../chapter-03-machine-level-programs/notes/section-3.2.1-机器级代码与编译链路.md)）。  
HFT **必须**热路径 O3，但 **只靠 O3 远远不够** — 纳秒级优化与毛刺排查靠机器码 / 流水线 / 缓存知识。

**HFT 实践：** 热路径 `.cpp` **单文件或 LTO**；`restrict` 明确 buffer；避免热循环调 **未知副作用** 函数。  
等级：`-O0`/`-Og` 学调，**`-O3` 热路径上线**；浮点放宽用 **`-Ofast`/`-ffast-math`** 才需单独评估（默认 O3 ≠ 放宽 IEEE）。

---

### 口述巩固 · 自测

1. **为何后台多用 O2 而非全仓 O3？** — 编译/调试/体积；瓶颈在 IO；CPU 收益体感小  
2. **为何学汇编仍必要？** — O3 天花板；负优化可控；perf/objdump 排毛刺；全员都会开 O3  
3. **O3 默认会破坏 IEEE754 吗？** — **一般不会**；放宽看 **`-Ofast`/`-ffast-math`**
