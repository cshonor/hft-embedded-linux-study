## 3.2.2 栈帧（内核 / 函数调用底层核心）

> [章导读](../README.md) · 上节 [§3.2.1](./section-3.2.1-机器级代码与编译链路.md) · 下节 [§3.2.3 AT&T](./section-3.2.3-AT&T汇编语法.md)

### 先正名：是「栈帧」，不是「战争」

听写/口误常混成别的词。这里的正式名是 **栈帧 (stack frame)**：

> 函数被调用时，CPU 在 **内存栈** 上划出的一块 **专属空间**，用来存放该次调用相关的状态。

典型会放（随 ABI / 优化程度不同，未必全部上栈）：

| 内容 | 作用 |
|------|------|
| **返回地址** | `call` 压入；`ret` 弹出后继续执行调用点下一条 |
| **保存的 `%rbp` 等** | 链到上一帧（经典帧指针约定） |
| **局部变量 / spill** | 寄存器装不下或需要取址时落到栈 |
| **溢出参数** | 超过寄存器传参个数后的参数（System V 整型：第 7 个起） |

**为何要紧：** 每次真实 `call` 都可能建立/拆除栈帧 → 内存读写 + 指令开销。HFT 热路径要理解 **帧怎么建、编译器怎么省掉帧指针、何时用 `inline` 消灭调用**，才能砍调用延迟。

→ 建帧细节见下；完整调用约定、递归、`inline` → [§3.7](./section-3.7-过程与栈帧.md)  
→ 栈在整进程地址空间里的位置（堆 / `.data` / `.text`…）→ [补充 · 五大内存分区](./section-补充-C程序五大内存分区.md)

### 易混：栈帧 vs 以太网帧（同词「帧」，本质两码事）

表层类比 **成立**：都叫「帧」，都是 **一段有独立完整意义的数据单元**。

| | **栈帧 (stack frame)** | **以太网帧 (Ethernet frame)** |
|--|------------------------|-------------------------------|
| **载体** | 进程 **虚拟内存栈区**；`%rsp`/`%rbp` 本地操作；**纳秒级**，无网卡 IO | **网卡缓冲 / 线缆**；跨机器；**微秒～毫秒** 量级 IO |
| **作用** | 保存一次函数现场（返回地址、局部变量、溢出参数等），服务 **CPU 执行** | 跨设备传数：源/目的 MAC、类型、载荷、CRC |
| **生命周期** | `call` 建、`ret` 毁；**仅本进程内**，不跨机 | 一端发出、另一端收；与函数调用 **无关** |
| **结构** | 帧指针链、局部区、返回地址…（本章 / §3.7） | 前导码、MAC、EtherType、payload、FCS…（网络栈） |

**一句话：** 用词都像「打包成一块」；**硬件、用途、原理完全无关** — 只是同名。CSAPP Ch3 考的是前者；网卡收发优化是另一条线（DPDK / 内核网络）。

**HFT 两条线都要，但别串台：**

1. 交易逻辑：`call` → 栈帧；过多调用拉延迟 → **`inline` / 少深调用**  
2. 行情 / 下单：底层是 **以太网帧**（及更上层协议）→ 另一套延迟优化

---

**经典函数序言（建立栈帧）：**

```asm
push %rbp          ; 保存调用者帧指针
mov  %rsp, %rbp    ; 当前 rsp 作为本帧基址
sub  $N, %rsp      ; 在栈上开辟 N 字节（局部变量）
```

| 步骤 | 作用 |
|------|------|
| **`push %rbp`** | 链到上一帧，便于回溯栈 |
| **`mov %rsp,%rbp`** | 固定本函数 **帧基址** |
| **`sub $N,%rsp`** | 局部变量、 spill 槽在栈上 |

**现代编译器 & 高性能代码（HFT、UEFI 驱动）：** 常 **省略帧指针**（`-fomit-frame-pointer` / `-O2` 默认），只用 **`%rsp` 相对寻址** — 少几条指令、多一个可用寄存器，**profile 里常见无 `%rbp` 的序言**。

→ 完整调用约定、返回地址、`call`/`ret` 见 [§3.7 过程与栈帧](./section-3.7-过程与栈帧.md)

### 嵌套调用：`A() → B() → C()` 会建 **3 个独立栈帧**

栈是 **层层堆叠、后进先出 (LIFO)**。x86-64 栈向 **低地址** 生长；画图时习惯把 **栈顶画在上面**（= 当前 `%rsp`，地址更低）。

**执行：**

1. 进入 `A` → 建 **A 栈帧**  
2. `A` 里 `call B` → 压返回地址，建 **B 栈帧**（叠在 A「之上」= 更靠近栈顶）  
3. `B` 里 `call C` → 再建 **C 栈帧**；此时 **C 在栈顶**，CPU 只直接操作栈顶一带

```
低地址（栈顶，%rsp）
┌─────────┐
│  C 栈帧  │  ← 当前
├─────────┤
│  B 栈帧  │
├─────────┤
│  A 栈帧  │
└─────────┘
高地址（栈底方向）
```

**返回（只能从顶往下拆，像叠书）：**

1. `C` 的 `ret` → 拆掉 C，`%rsp` 回到 B 顶，继续跑 B  
2. `B` 的 `ret` → 拆掉 B，回到 A  
3. `A` 的 `ret` → 拆掉 A  

不能「中间抽走」A/B 的帧而不动上面的 — 那是栈的定义。

**HFT：**

| 点 | 含义 |
|----|------|
| 每次真实 `call` | 可能新建帧：压返回地址、保存寄存器、开局部区 → **嵌套越深，累积开销越大** |
| **`inline`** | 抹掉 `call`/`ret`，**往往不再为该次调用单独建帧** → [§3.7.7](./section-3.7-过程与栈帧.md) |
| **递归** | 同一函数多层帧无限叠；深度大 → **栈溢出**（默认栈约数 MB） |

→ 单帧内部布局、`call`/`ret` 细节 → [§3.7](./section-3.7-过程与栈帧.md)

### 口述巩固 · 自测

1. **栈帧是什么？** — 一次函数调用在栈上的专属块；常含返回地址、局部变量等（不是「战争」）  
2. **和以太网帧一样吗？** — 仅「独立数据单元」类比成立；载体/用途/结构 **完全无关**  
3. **`A→B→C` 几个帧？谁在栈顶？返回顺序？** — **3 个**；**C** 在顶；**C→B→A**（LIFO）  
4. **为何 Release 里常看不到 `%rbp`？** — **省略帧指针** 换性能；用 `%rsp` 相对寻址  
5. **和 HFT 的关系？** — 嵌套调用叠帧拉延迟；`inline` 少帧；递归慎用
