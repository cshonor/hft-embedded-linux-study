# 5.4 警惕 Interpositioning

## 本节讲什么

**Interpositioning（插桩 / 符号介入）** 指在 **正常目标函数之前** 插入自己的实现，仍 **调用原函数** 或直接 **替换** 行为。Linux 上常用 **`LD_PRELOAD`**、链接器 **`--wrap`**、以及 **malloc 调试钩子** 实现。能力强大：性能分析、内存泄漏检测、安全审计；也 **可被滥用** 或 **误用** 导致诡异 bug。本节建立机制心智模型，并强调 **生产环境警惕**。

## 要点

### 什么是 Interpositioning

```text
  应用调用 malloc()
        ↓
  [ 你的 my_malloc ]  — 记录、统计、加 fence
        ↓
  [ 真正的 malloc ]   — libc 实现
```

要求：**你的符号在链接 / 加载顺序上「赢过」 libc 中的同名符号**，且通常需要 **保存原函数指针** 以便转调。

### 手段 1：`LD_PRELOAD`

动态链接器加载共享库顺序中，**预加载库** 优先：

```bash
export LD_PRELOAD=/path/to/myhook.so
./my_program
```

`myhook.so` 中若 **导出** `malloc`、`free` 等 **全局函数**，会 **覆盖** `libc.so` 中的默认解析（同一进程内）。

**典型合法用途**：

| 用途 | 说明 |
|------|------|
| **内存调试** | 统计分配、检测 double-free、写越界 fence |
| **I/O 追踪** | 记录 `open`/`read`/`write` |
| **延迟加载故障注入** | 测试错误路径 |

**风险**：

- 与 **静态链接** 程序 **无效**（无动态符号表可替换）；
- 与 **直接链接 `-lmalloc`** 或 **LTO 内联** 组合时行为复杂；
- **安全**：不可信环境勿运行带陌生 `LD_PRELOAD` 的程序；
- **性能**：每次调用多一层间接。

**最小示例思路**（教学伪代码）：

```c
#define _GNU_SOURCE
#include <dlfcn.h>
#include <stdlib.h>

static void *(*real_malloc)(size_t);

void *malloc(size_t n)
{
    if (!real_malloc)
        real_malloc = dlsym(RTLD_NEXT, "malloc");
    /* log: size n */
    return real_malloc(n);
}
```

编译：`gcc -shared -fPIC myhook.c -o myhook.so -ldl`

### 手段 2：`--wrap` 符号（链接期）

GCC / ld 支持 **包装** 指定符号：

```bash
gcc main.c wrap_malloc.o -Wl,--wrap=malloc -o app
```

- 调用 `malloc` 的代码被改写成调用 **`__wrap_malloc`**；
- 你在 `wrap_malloc.o` 里实现 **`__wrap_malloc`**，并用 **`__real_malloc`** 调回原函数。

| 符号 | 角色 |
|------|------|
| `__wrap_malloc` | 你的插桩实现 |
| `__real_malloc` | 链接器提供的 **原 malloc** 别名 |

适合 **静态链接** 或 **不想依赖 LD_PRELOAD** 的场景；需在 **链接命令** 中显式启用。

### 手段 3：malloc 调试钩子（历史 / 特定 libc）

部分环境提供 **`malloc_hook`**、**`mtrace`/`muntrace`**（glibc 文档中有说明，新版本可能变更或受限）：

```c
#include <mcheck.h>
int main(void) {
    mtrace();   /* 写 MALLOC_TRACE 指定文件 */
    /* ... */
    muntrace();
}
```

与 **5.2 延迟绑定** 结合：钩子必须在 **第一次 malloc 前** 安装才有效。

### 与 5.1 / 5.2 符号规则的关系

| 机制 | 生效阶段 | 覆盖对象 |
|------|----------|----------|
| **强符号重复定义** | 静态链接 | 报错（**5.1**） |
| **LD_PRELOAD** | 动态加载 | 同名 **全局函数** 优先 |
| **`--wrap`** | 链接重写 | 调用点改为 `__wrap_*` |
| **`dlsym(RTLD_NEXT, ...)`** | 运行时 | 在预加载链中找到 **下一个** 定义 |

**弱符号**（**5.1 demo03**）与插桩不同：弱符号是 **链接器合并规则**，不是「拦截调用」。

### 开发中的正当 vs 危险用法

**正当**：

- 本地调试内存；CI 里对测试二进制注入 trace `.so`；
- 用 **`--wrap`** 在单元测试里替换 `time()` 返回固定值。

**危险 / 警惕**：

- 生产服务器全局 `LD_PRELOAD` 未知 `.so`；
- 插桩 **非线程安全** 的 libc 函数却用于多线程服务；
- 假设插桩了 `malloc` 就能捕获 **mmap/brk** 等底层分配路径（不完整）；
- 与 **Sanitizer**（ASan/LSan）同时用可能 **冲突**。

### 与 ch04 对比

**4.2** 的 bug 是 **错误声明** 导致 **错误地址** 被使用；Interpositioning 是 **故意** 替换符号实现。二者都说明：**最终跑的是链接 / 加载后 **谁** 提供了该名字的定义**。
