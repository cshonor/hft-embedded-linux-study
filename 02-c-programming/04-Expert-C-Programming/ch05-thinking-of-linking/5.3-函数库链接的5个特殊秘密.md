# 5.3 函数库链接的5个特殊秘密

## 本节讲什么

静态库链接有许多 **反直觉** 规则，导致「明明有 `libfoo.a` 却 undefined reference」或「加上 `-lm` 就好了」类问题。本节归纳书中 **五个秘密**：归档 **从左到右扫描**、**未定义符号** 才从 `.a` 拉 `.o`、**循环依赖** 的处理、**`-lm` 必须放在对象后面**、静态链接导致 **重复代码**。结合 **demo02_static_lib** 与 **ch04 链接不做类型检查** 一并理解。

## 要点

### 秘密 1：链接器按命令行从左到右扫描

```bash
gcc main.o -L. -ladd -o demo    # 常见正确顺序
gcc -ladd main.o -o wrong       # 错误：扫描 -ladd 时尚无 undefined，可能拉不到 add.o
```

链接器维护 **可执行映像 + 当前未定义符号集合**。处理每个输入时：

- **`.o` 文件**：**全部** 链入（不管是否被引用——除非 `--gc-sections` 等优化）；
- **`.a` 归档**：**按需** 提取成员（见秘密 2）。

**规则**：把 **还没被满足的库** 放在 **引用它们的 `.o` 之后**；若库 A 依赖库 B，常写：**`-lA -lB`** 且整体在 **`main.o` 后**。

### 秘密 2：只有未定义符号才会从 `.a` 里拉 `.o`

**静态库** 不是「一整块链接进去」，而是 **符号索引的 `.o` 集合**：

```text
libadd.a = [ add.o ]     # ar rcs libadd.a add.o
```

链接 `main.o` 时若 `main.o` 有 **`U add`**（undefined `add`），扫描 `libadd.a` 时发现 **`add.o` 定义了 `add`** → **提取** `add.o` 并入链接。

若 `main.c` **从不调用** `add()`，且没有其它 undefined 指向 `add.o` 内符号 → **`add.o` 不会进最终可执行文件**（减小体积）。

```bash
cd demo/demo02_static_lib
make clean && make
nm main.o | grep add          # U add
nm demo_static | grep add     # T add（已定义）
ar t libadd.a                 # 列出归档成员
```

### 秘密 3：循环依赖要重复或分组

典型模式：

```text
libA.a  需要  libB.a  中的 foo()
libB.a  需要  libA.a  中的 bar()
```

单次 **`gcc ... -lA -lB`** 可能失败：扫完 A 拉入部分成员，B 又需要 A 中新符号，但 A **不再扫描**。

**解法**：

```bash
gcc ... -lA -lB -lA          # 重复扫描
# 或
gcc ... -Wl,--start-group -lA -lB -Wl,--end-group
```

内核 / 大型 C 项目 Makefile 里常见 **group** 或 **重复 `-l`**。

### 秘密 4：`-lm` 为什么常放最后

数学库 **`libm.so` / libm.a`** 中的 `sin`、`cos`、`pow` 等 **不会** 在链接开始时自动参与扫描。

```bash
gcc main.c -lm -o ok           # 错：main 的 undefined 尚未出现时就扫了 -lm
gcc main.c -o ok -lm           # 对：main.o 先入，undefined sin 再由 -lm 满足
gcc main.o -o ok -lm           # 更明确
```

**gcc** 驱动有时会把 `-l` 重排，但 **手写 ld 命令** 或 **静态链接** 时必须自己保证顺序。与 **秘密 1** 同一原理。

### 秘密 5：静态链接造成重复代码

每个用 **静态 `.a`** 的可执行文件 **各自拷贝** 一份库 `.o` 代码：

- 10 个程序都静态链 `libcrypto.a` → 10 份相同机器码占 **磁盘与内存**；
- **动态 `.so`** 则 **一份物理页多进程共享**（**5.2**）。

权衡：

| 选择 | 优点 | 缺点 |
|------|------|------|
| 静态 `.a` | 部署简单、无 `.so` 依赖 | 体积大、更新需重链 |
| 动态 `.so` | 共享、易更新 | 依赖 ABI、启动需 ld.so |

### 与 ch04：链接器不检查 C 类型

**秘密 2** 只保证 **符号名** 一致，例如 `arr` 在 A 文件是数组、B 文件声明成指针 —— 仍可能 **链接成功**（**4.2 demo02_wrong**），运行才崩溃。

> **类型匹配** 靠头文件与程序员；**链接器** 只拼 **名字 + 地址**。

### 实用命令速查

```bash
ar rcs libfoo.a foo.o bar.o    # 创建 / 更新静态库
ar t libfoo.a                  # 列出成员
nm -A libfoo.a                 # 看归档里各 .o 的符号
gcc main.o -L. -lfoo -o app    # 库放 main.o 后
```

### 常见链接错误解读

| 错误信息 | 常见原因 |
|----------|----------|
| `undefined reference to 'foo'` | 缺 `.o`、库顺序错、库未加入 |
| `multiple definition of 'foo'` | 两个 **强符号** 同名（**5.1**） |
| `cannot find -lfoo` | `-L` 路径不对或库不存在 |
