# 5.1 函数库、链接和载入

## 本节讲什么

从 **「编译完 .c 之后发生了什么」** 入手：目标文件如何 **链接（link）** 成可执行文件或共享库，**载入器（loader）** 如何把程序放进内存并启动。建立 **四阶段编译流水线**（预处理 → 编译 → 汇编 → 链接）、**ELF 段布局**（`.text` / `.data` / `.bss`）、**符号解析与重定位** 三件套，并区分 **强符号 / 弱符号 / 静态符号**。最后用 **`nm`、`readelf`** 读懂 `.o` 里有什么——**demo01_four_stage** 逐步演示。

## 要点

### 四阶段编译流水线

一条典型的 `gcc main.c -o main` 背后，工具链大致是：

```text
main.c  →  cpp  →  cc1  →  as  →  ld  →  main (可执行文件)
          预处理   编译    汇编   链接
          .i       .s      .o
```

| 阶段 | 工具 | 输入 → 输出 | 做什么 |
|------|------|-------------|--------|
| **预处理** | `cpp`（`gcc -E`） | `.c` → `.i` | 展开 `#include`、宏替换、条件编译 |
| **编译** | `cc1`（`gcc -S`） | `.i` → `.s` | 生成 **汇编**；语法/类型检查在此完成 |
| **汇编** | `as`（`gcc -c`） | `.s` → `.o` | 生成 **可重定位目标文件**（ELF `.o`） |
| **链接** | `ld`（`gcc` 调用） | `.o` + 库 → 可执行文件 | **符号解析** + **重定位** + 段合并 |

**demo01_four_stage** 的 Makefile 把四步拆开：`main.i` → `main.s` → `main.o` → `main`。

```bash
cd demo/demo01_four_stage
make all
make inspect    # nm main.o; readelf -S main.o; readelf -r main.o
```

### 目标文件里的段（Section）

每个 `.o` / 最终可执行文件按 **段** 组织，常见三类 **可加载段**：

| 段 | 内容 | 典型来源 | 特点 |
|----|------|----------|------|
| **`.text`** | 机器码（函数体） | 函数定义 | 只读、可执行 |
| **`.data`** | **已初始化** 全局/静态变量 | `int g = 1;` | 映像里占空间，有初值 |
| **`.bss`** | **未初始化** 全局/静态变量 | `int g;` | **NOBITS**：文件里不占字节，载入时 **零初始化** |

**demo01** 中 `g_init = 1` 进 `.data`，`g_bss` 进 `.bss`；运行输出 `g_bss=0` 即 BSS 零初始化。

```bash
readelf -S main.o          # 看段表：.text .data .bss .rodata ...
readelf -S main.o | grep bss   # .bss 常带 NOBITS 标志
```

**`.rodata`**：只读常量（字符串字面量、`const` 全局等），与 **5.2** 动态链接的 **PIC** 相关。

### 符号解析（Symbol Resolution）

链接器维护 **全局符号表**，对每个 **全局符号**（函数名、全局变量名）决定：

1. **定义在哪**：哪个 `.o` / 库成员提供了 **唯一** 定义？
2. **引用在哪**：哪些 `.o` 有 **未定义引用（undefined reference）**？

| 符号类别 | 含义 | 示例 |
|----------|------|------|
| **强符号（strong）** | 有初值的全局，或函数 | `int x = 1;`、`void foo(void){}` |
| **弱符号（weak）** | 未初始化 tentative，或显式 `__attribute__((weak))` | `int x;`（GCC 10+ 默认 `-fno-common` 时进 `.bss` 当强定义）；教学 demo 用 `weak` 属性 |
| **外部引用** | 本单元用、别处定义 | `extern int x;` 后使用 `x` |

**强 vs 弱规则（GNU ld，面试高频）**：

- 不允许两个 **强符号** 同名（链接错误 `multiple definition`）。
- 一个 **强** + 一个 **弱** → 选 **强** 的定义。
- 多个 **弱** 同名 → 选 **其中一个**（通常第一个遇到的）。

**demo03_weak_strong**：`weak.c` 中 `__attribute__((weak)) int g_val;`，`strong.c` 中 `int g_val = 20;`（强），链接后 `g_val == 20`。

```bash
cd demo/demo03_weak_strong
make && ./demo_weak
nm demo_weak | grep g_val
```

**静态符号（`static`）**：

```c
static int counter;      /* 文件内可见，不出现在全局链接 */
static void helper(void) { }
```

- 链接器 **不** 把 `static` 全局/函数当作跨文件符号解析。
- 不同 `.c` 里各有一个 `static int x`，**不冲突**——各自是不同符号（通常带 `.o` 名修饰）。

### 重定位（Relocation）

编译单个 `.c` 时，编译器 **不知道** 最终地址，故在 `.o` 里留下 **重定位条目**：「此处 4 字节需填上符号 `foo` 的地址 + 偏移」。

链接阶段：

1. 合并各 `.o` 的同名段（所有 `.text` 拼在一起……）。
2. 分配 **虚拟地址**（或相对地址）。
3. **回填** 所有重定位槽位。

```bash
readelf -r main.o    # Relocation section：哪些偏移引用哪些符号
```

**5.5** 会用 **map 文件**、**objdump -R** 进一步查看。

### 函数库：静态 `.a` 与链接

**静态库** = 若干 `.o` 的归档（archive），用 **`ar rcs`** 打包：

```bash
ar rcs libadd.a add.o    # demo02_static_lib
```

链接时 **只拉入** 解决 **当前未定义符号** 所需的 `.o` 成员（**5.3** 详述扫描顺序）。

```bash
cd demo/demo02_static_lib
make
./demo_static
```

### 用 `nm` 读符号

```bash
nm main.o
```

常见列含义：

| 列 | 含义 |
|----|------|
| **T / t** | 代码段符号（`T` 全局，`t` static） |
| **D / d** | 已初始化数据 |
| **B / b** | BSS |
| **U** | **Undefined** — 本文件引用、别处定义 |
| **W / w** | 弱符号 |

### 与 ch04 的衔接：链接不做类型检查

**4.2 / 4.4** 的经典 bug：`char arr[]` 定义 + `extern char *arr` 声明 —— **符号名** `arr` 匹配，**类型** 不匹配，链接器 **不报错**，运行崩溃。

> 链接器只认 **符号名字节串**，不做 C 类型等价检查。声明/定义匹配是 **程序员** 的责任。

### 载入（Loading）

可执行文件生成后，**载入器**（Linux 上常由内核 `execve` + 动态链接器配合）：

1. 映射 **PT_LOAD** 段到进程地址空间；
2. 为 **BSS** 分配并清零；
3. 设置 **入口点**（`ENTRY`，见 **demo04** 的 `kernel.lds`）；
4. 动态链接时还会加载 **共享库**（**5.2**）。

### 链接脚本预览（kernel.lds）

嵌入式 / 内核常用手写 **`.lds`** 规定段地址与边界符号：

```text
ENTRY(_start)
_stext / _etext   — .text 起止
_sbss  / _ebss    — .bss  起止
```

**demo04_linker_script** 用 `ld -T kernel.lds` 生成 **kernel.elf**（无 libc），C 代码通过 `extern char _stext, _ebss;` 引用边界。
