# 5.2 动态链接的优点

## 本节讲什么

**5.1** 侧重静态链接与目标文件结构；本节说明 **动态链接（dynamic linking）** 为何成为现代 Unix/Linux 默认方案：**共享库（shared library）** 在磁盘与内存中 **只存一份**，多进程共享；**位置无关代码（PIC）** 使 `.so` 可加载到任意地址；**延迟绑定（lazy binding）** 推迟符号解析到 **首次调用**。介绍 **`ldd`** 查看依赖，并与 **5.3 静态库**、**5.4 插桩** 对比。

## 要点

### 静态链接 vs 动态链接

| 维度 | 静态链接（`.a`） | 动态链接（`.so`） |
|------|------------------|-------------------|
| **链接时机** | 编译链接时把库代码 **拷进** 可执行文件 | 链接时只记录 **依赖**；运行时再解析 |
| **体积** | 每个程序一份库代码副本 | 磁盘上 **一份** `.so`，多程序共享 |
| **更新** | 改库需 **重新链接** 所有程序 | 替换 `.so`（兼容 ABI）即可生效 |
| **内存** | 每进程独立代码副本 | 物理页可 **共享**（同版本 `.so`） |
| **启动** | 无运行时链接开销 | 需 **动态链接器**（如 `ld-linux-x86-64.so.2`） |

```bash
gcc -static main.c -o main_static    # 尽量静态链接（体积大）
gcc main.c -o main_dyn               # 默认动态链接 libc 等
file main_dyn                        # "dynamically linked"
```

### 共享库是什么

- Linux 上通常为 **ELF 共享目标** `libfoo.so`（Windows 对应 `DLL`）。
- 编译：`gcc -shared -fPIC -o libfoo.so foo.c`
- 链接程序：`gcc main.c -L. -lfoo` → 可执行文件的 **`.dynamic` / `.got.plt`** 段记录依赖。

**命名惯例**：`libm.so` → 链接选项 **`-lm`**（**5.3** 讲顺序）。

### 位置无关代码（PIC）

共享库被 **不同进程** 映射到 **不同虚拟地址**，因此：

- 库内 **不能** 写死绝对地址访问全局数据/函数；
- 编译加 **`-fPIC`**，通过 **全局偏移表（GOT）** 与 **过程链接表（PLT）** 间接寻址。

```text
  调用 printf@plt  →  第一次：动态链接器解析真实地址，写入 GOT
                     之后：  经 GOT 跳转到 libc 中的 printf
```

**`.text`** 中 PIC 代码可 **只读映射多进程共享**；可写数据在 **`.data` / `.bss`** 或 per-process 副本（取决于 `RTLD` 与链接方式）。

### 延迟绑定（Lazy Binding）

默认策略（如 GNU ld 的 **PLT + lazy**）：

1. 程序启动时 **不** 解析每个 `printf`、`malloc` 的真实地址；
2. 第一次 **调用** 某 PLT 槽位时，**动态链接器** 才解析并 **补丁** GOT；
3. 后续调用走 **已解析** 的快速路径。

优点：启动快；未用到的库函数 **零解析成本**。

```bash
LD_BIND_NOW=1 ./a.out    # 强制启动时全部解析（调试 / 安全场景）
```

### 动态链接器与 `ldd`

进程启动链（简化）：

```text
kernel execve → 程序 PT_INTERP 指向 ld.so → ld.so 加载 .so、重定位、跳 _start
```

**`ldd`** 列出可执行文件需要的共享库及路径（**注意**：对不可信二进制勿盲目运行 `ldd`，有历史安全坑；教学环境可用）：

```bash
ldd ./main_dyn
# linux-vdso.so.1
# libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6
# /lib64/ld-linux-x86-64.so.2
```

### 与 5.1 段布局的关系

动态可执行文件除 **5.1** 的 `.text/.data/.bss` 外，常见：

| 段 / 节 | 作用 |
|---------|------|
| **`.dynamic`** | 依赖库名、重定位类型等 **元数据** |
| **`.got` / `.got.plt`** | 全局偏移表（存解析后的地址） |
| **`.plt`** | 过程链接表（调用共享函数的跳板） |
| **`.rel.dyn` / `.rela.dyn`** | 动态重定位项 |

静态 `.o` 用 **`readelf -r`**（**5.1**）；已链接的动态可执行文件还可 **`objdump -R`**（**5.5**）。

### 何时仍用静态库

| 场景 | 原因 |
|------|------|
| 嵌入式 / 无 OS | 无 `ld.so`，常 **纯静态** 或裸机 **demo04** |
| 单文件分发 | 避免目标机缺 `.so` |
| 性能敏感且库很小 | 避免 PLT 间接跳转（现代差距常很小） |
| 内核模块 | 内核自有链接模型，非用户态 `.so` |

**demo02_static_lib** 演示 **`ar rcs`** + **`-ladd`**；与 **5.3** 的扫描规则直接相关。

### 安全与插桩（预告 5.4）

动态链接使 **运行时替换** 共享库函数成为可能：

- **`LD_PRELOAD`** 优先加载指定 `.so`，**覆盖** `malloc` 等符号；
- 调试、性能分析、（恶意）拦截都利用同一机制。

**5.4** 专讲 **Interpositioning**。
