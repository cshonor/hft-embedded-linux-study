# 2.2 多做之过

## 本节讲什么

**Sins of Commission（多做之过）**：语言 **主动做了额外的事**——有时是便利，有时是陷阱。本节覆盖 `switch` 贯穿、相邻字符串字面量拼接、文件作用域标识符默认 **extern 链接**、K&R 时代的 **隐式 int** 与 **无原型函数声明**，以及预处理器宏 **多做** 的文本替换（`MULT`、`MAX`）。这些行为在标准中合法，却常年制造生产事故。

## 要点

### 1. `switch` 贯穿（fall-through）

C 规定：`case` 标签 **不会自动 break**；匹配后代码 **一直执行到 `switch` 结束或遇到 `break`**。

```c
switch (x) {
case 1: puts("one");
case 2: puts("two");   /* 无 break → 继续 */
case 3: puts("three"); break;
}
/* x==2 时输出 two 然后 three */
```

| 设计动机 | 风险 |
|----------|------|
| 早期编译器简单；多 case 共享一段代码 | 漏写 `break` → 逻辑错误 |
| C 刻意保留（非疏忽） | 现代可用 `-Wimplicit-fallthrough`、注释 `/* fallthrough */` |

**Demo**：`demo/demo01_switch.c` — `x=2` 时先打印 `case2` 再 `case3`。

```bash
cd demo && make all && ./demo01_switch
```

**工程习惯**：除非故意贯穿，每个 `case` 末尾写 `break`；或改用 `if-else` / 查表。

### 2. 相邻字符串字面量自动拼接

```c
char *msg = "Hello, " "world!";   /* 等价于 "Hello, world!" */
printf("part1" "part2" "part3");   /* 编译期合并为一个字符串 */
```

- **只做一件事**：在 **编译前** 把相邻字面量合成一个
- **用途**：长字符串分行写、宏展开后拼接
- **陷阱**：与 **宽字符**、**字符串化 `#`** 混用时需小心；不是运行时拼接

### 3. 文件作用域默认 extern 链接

```c
/* file1.c */
int counter = 0;        /* 定义：extern 链接，其他 .c 可见 */

/* file2.c */
extern int counter;     /* 声明，链接到 file1 的定义 */
```

| 关键字 | 链接属性 | 典型用途 |
|--------|----------|----------|
| （默认）全局变量/函数 | **external linkage** | 跨翻译单元共享 |
| `static`（文件作用域） | **internal linkage** | 仅本 `.c` 可见，防符号冲突 |
| `extern`（声明） | 引用外部定义 | 头文件中声明 API |

**多做之过**：不写 `static` 就 **默认对外暴露符号** → 链接时命名冲突、意外耦合。头文件应配合 **include guard** 与 **单一职责**（**ch05** 链接专题）。

### 4. 隐式 int 与 K&R 函数声明

**隐式 int（C89 前 / 宽松模式）**：

```c
f(x)   /* 未声明返回类型 → 默认 int */
{
    auto y;            /* 未声明类型 → 默认 int */
}
```

**无原型声明（K&R）**：

```c
int f();             /* 不是 void f()！表示「返回 int，参数未指定」 */
f(3.14, "hello");    /* 旧式：不检查参数个数与类型 → 栈/寄存器错乱 */
```

ANSI C 要求 **原型**：

```c
int f(int a, double b);   /* 类型检查 */
int g(void);              /* 无参数必须用 void */
```

| 对比 | K&R `int f()` | ANSI `int f(void)` / `int f(int)` |
|------|---------------|-------------------------------------|
| 参数检查 | 无 | 有 |
| 与 `f(...)` 可变参 | 易混淆 | 用 `<stdarg.h>` 规范处理 |

见 **ch01 1.4**、`demo01_kr_vs_ansi.c`。

### 5. 预处理器：宏的「多做」

宏是 **纯文本替换**，编译器不做语义分析——比 Fortran 漏逗号更隐蔽。

#### MULT / MUL：缺少括号 → 优先级陷阱

```c
#define MULT(a, b) a * b
#define MULT_SAFE(a, b) ((a) * (b))

MULT(1 + 2, 3 + 4);
/* 展开: 1 + 2 * 3 + 4  → 按 * 优先 → 1 + 6 + 4 = 11，而非 21 */
MULT_SAFE(1 + 2, 3 + 4);  /* ((1+2)*(3+4)) = 21 */
```

**Demo**：`demo/demo03_macro_mul.c`。

#### MAX：括号仍挡不住副作用重复求值

```c
#define MAX(a, b) ((a) > (b) ? (a) : (b))

int i = 0, j = 5;
MAX(i++, j++);   /* i++, j++ 可能各执行两次 → 未定义行为 */
```

| 问题 | 原因 | 对策 |
|------|------|------|
| 优先级 | 宏体无括号 | 整体与参数 **双层括号** |
| 副作用 | `a`/`b` 在 `?:` 两侧各出现一次 | **改用 inline 函数**（C99）或 GCC `statement expression` |
| 无类型 | 文本替换 | `static inline` + `<stdbool.h>` / 泛型（C11 `_Generic`） |

```c
static inline int max_int(int a, int b)
{
    return a > b ? a : b;   /* 实参只求值一次 */
}
```

**与 ch01 1.3 的关系**：1.3 讲宏机制；本节强调 **「多做」的替换次数** 与 **链接/声明** 类陷阱。

### 小结：多做之过清单

| 特性 | 「多做了什么」 | 防御 |
|------|----------------|------|
| `switch` fall-through | 匹配后继续执行 | 显式 `break` / 编译器警告 |
| 字符串拼接 | 合并相邻字面量 |  intentional 使用，别与指针运算混淆 |
| 默认 extern | 全局符号导出 | 文件内 helper 用 `static` |
| 隐式 int / 无原型 | 缺省类型与参数 | `-std=c11 -Wall`、头文件原型 |
| 宏替换 | 展开整段文本 | 括号、避免副作用、优先 inline |
