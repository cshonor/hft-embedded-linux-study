# 2.1 这关语言特性何事，在Fortran里这就是Bug呀

## 本节讲什么

本章开篇用一个 **Fortran DO 循环** 的经典事故说明：程序员眼中的「明显 Bug」，在语言规则下可能是 **完全合法的行为**。C 同样充满这类「语言特性」——它们往往来自 **历史妥协**（为兼容旧代码、简化编译器、贴近硬件），而非现代语言设计中的「最佳实践」。本节建立全书基调：**先读懂规则，再谈对错**。

## 要点

### Fortran DO 循环事故

程序员本意（循环 `I` 从 1 到 10）：

```fortran
      DO 10 I = 1, 10
         ... 循环体 ...
   10  CONTINUE
```

若 **漏写逗号**，写成：

```fortran
      DO 10 I = 1.10
```

在 Fortran 规则下可能被解析为：

1. **`DO 10 I = 1`** — 把常数 `1` 赋给 `I`，并 **只执行一次** 标号为 `10` 的语句
2. **`.10`** — 被当作浮点常数或标号相关 token，而非「从 1 到 10 的步进」

结果：本应是 10 次迭代的循环变成 **单次赋值 + 一次跳转**，航天器制导软件因此出错——在 Fortran 里这是 **按语法合法** 的，在工程上却是灾难。

| 视角 | 结论 |
|------|------|
| 程序员 | 「漏了一个逗号，显然是 Bug」 |
| 语言规范 | 「空格/标点解析结果符合文法 → 合法程序」 |
| 工程教训 | **编译器不会替你想意图**；静态分析、代码审查、测试必不可少 |

### 「Bug」vs「语言特性」

| 类型 | 含义 | C 中的例子（后文详述） |
|------|------|------------------------|
| **语言特性** | 标准明确规定的行为，可能反直觉但可预测 | `switch` 贯穿、`unsigned` 比较提升 |
| **未定义行为（UB）** | 标准不保证后果，编译器可任意处理 | 有符号溢出、越界访问 |
| **实现定义** | 标准允许多种行为，文档需说明 | `char` 有符号与否 |
| **真·历史 Bug** | 设计失误但无法移除 | 三字符组（trigraphs），见 **2.5** |

核心区分：**特性 = 可读标准能推导出行为；UB = 标准放手，优化器可「删掉你的代码」**。

### 历史妥协：C 为何继承怪癖

C 不是从零设计的「干净语言」，而是 **B → C → Unix 工具链** 演进产物（见 **ch01 1.1–1.2**）：

```text
硬件/编译器约束（PDP-11、单遍编译）
    ↓
语法极简、少运行时检查
    ↓
「多做」：默认 extern 链接、字符串自动拼接、宏文本替换
「少做」：无数组边界、无内建 bool（C89）、字符串无长度
「误做」：运算符优先级、* & [] 多重含义
    ↓
ANSI C（1989）标准化 → 保留绝大多数旧行为（quiet change，见 ch01 1.10）
```

**妥协逻辑**：已有海量 C 代码与 Unix 生态；改语义 = 静默破坏旧程序。标准选择 **文档化现有行为**，而非全面修正。

### 读本章的方法

1. **遇到怪行为**：先查是 **特性、UB 还是实现定义**（ch01 **1.6**）
2. **写可移植代码**：假设最严格语义；开启 `-Wall -Wextra -Wconversion`
3. **写系统/内核代码**：更要懂 **实现定义** 与 **GCC/clang 扩展**（Embedded C **ch06**）
