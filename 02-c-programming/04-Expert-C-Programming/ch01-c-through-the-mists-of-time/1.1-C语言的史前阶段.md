# 1.1 C语言的史前阶段

## 本节讲什么

C 不是凭空出现的语言，而是从 **Multics → BCPL → B → C** 这条「为写编译器/操作系统而造工具链」的脉络里长出来的。1972 年 Dennis Ritchie 在 B 的基础上加入类型系统，形成 C；1973 年 **Unix 内核用 C 重写**，把「高级语言写系统软件」从实验变成工业实践。理解这段历史，才能理解 C **编译器优先、贴近机器、极简内核** 的设计哲学。

## 要点

### 时间线

| 年代 | 事件 | 意义 |
|------|------|------|
| 1960s | **Multics** 大型分时系统 | 复杂、昂贵；Bell Labs 退出后催生 Unix |
| 1967 | Martin Richards 的 **BCPL** | 无类型、编译器友好；Ken Thompson 参与 |
| 1969 | Thompson 的 **B** | BCPL 子集，为 PDP-7 上的 Unix 服务 |
| **1972** | Ritchie 的 **C** | 在 B 上加入 `char/int/float` 等类型 |
| **1973** | **Unix 内核用 C 重写** | 证明 C 可胜任系统编程；传播随 Unix 扩散 |

### 编译器优先（Compiler-first）

Ritchie 的设计目标不是「语法优雅」，而是 **让编译器在有限内存的 PDP-11 上能跑起来**：

- **类型直接映射硬件**：`char` 字节、`int` 字长、`float` 浮点单元——便于生成高效机器码
- **语法极简**：少关键字、少特殊规则，降低 parser 复杂度
- **可移植编译器**：C 编译器本身用 C 写（自举），换平台主要改后端
- **内核 + 库分离**：语言核心小，I/O 等放进 **stdio 库**（见 **1.3**），而非语言内置

```text
Multics (复杂 OS)
    ↓ 简化
Unix (Thompson/Ritchie, 1969)
    ↓ 需要系统编程语言
BCPL → B (无/弱类型) → C (有类型, 1972)
    ↓
Unix 内核 C 重写 (1973) → C 随 Unix 传播到全世界
```

### 为什么是 C 而不是别的

- **汇编**：可移植性差，维护 Unix 成本过高
- **Fortran**：科学计算导向，不适合系统编程
- **B**：无类型，指针运算易错，难以表达结构体等抽象
- **C**：在 B 的简洁与 BCPL 的类型之间取平衡——**足够低层写内核，足够高层写应用**
