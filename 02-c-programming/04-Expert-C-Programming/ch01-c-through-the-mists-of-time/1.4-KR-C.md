# 1.4 K&R C

## 本节讲什么

**K&R C**（《The C Programming Language》1978 第一版所描述的语言）是 C 工业化的里程碑，但按今天标准看 **极不安全**：**无函数原型**、**省略返回类型则默认为 `int`**、实参类型在调用处 **不做检查**。本节对比 K&R 与 ANSI 声明写法，理解为何 C89 引入原型是 C 历史上最重要的安全改进之一。

## 要点

### K&R 时代特征（1978 前主流）

| 特性 | K&R C | 后果 |
|------|-------|------|
| 函数声明 | 仅 **标识符列表**，无参数类型 | 实参类型错误 **编译通过、运行错乱** |
| 返回类型省略 | 默认为 **`int`** | `void` 函数不存在；忘记写返回类型静默变 int |
| 实参传递 | **default argument promotion** | `float→double`，`char/short→int` |
| 原型 | 无 | 链接阶段才解析符号，类型信息丢失 |

### 声明写法对比

**K&R 旧式定义（历史演示，禁止新代码使用）：**

```c
/* 声明：只告诉编译器「有个叫 add 的函数」*/
int add();

/* 定义：类型单独列在参数名之后 */
int add(a, b)
    int a;
    double b;
{
    return a + b;
}
```

**ANSI C 原型（现代写法）：**

```c
/* 声明与定义合一，带完整类型 */
int add(int a, double b);

int add(int a, double b)
{
    return (int)(a + b);
}

/* 真正无参必须用 void */
int getchar(void);
```

### 类型检查缺失的灾难

```c
/* K&R 风格 */
int mystery();          /* 编译器不知道参数是什么 */

main()                  /* 返回类型省略 → int */
{
    mystery("oops", 3.14);   /* 编译通过！ */
    return 0;
}
```

- 调用 `mystery(1, 2.5)` 与 `mystery("x", 1)` 在 K&R 下 **都能编译**
- 栈上按错误类型解释比特 → 崩溃或静默数据损坏
- **demo01** 演示 ANSI 原型如何拒绝错误实参：

```bash
cd demo && make all && ./demo01_kr_vs_ansi
# 取消 demo01_kr_vs_ansi.c 中 add_ansi("x", 1) 注释后重编译 → 报错
```

### `int` 的隐式默认

```c
foo() { return; }       /* K&R：返回类型是 int，return; 值未定义 */
unsigned bar;             /* K&R 某些编译器：省略 int → unsigned int */
```

- C89 起：**无返回类型非法**（旧代码需 `-implicit-int` 等扩展才能编过）
- `main` 在 C99 起必须 `int main(void)` 或 `int main(int argc, char *argv[])`

### K&R 第二版（1988）的过渡

- 已引入 **ANSI 原型**，与第一版并存说明
- 标志 K&R C → ANSI C 的正式过渡；考试常问「K&R 指哪一版」——通常指 **无原型的第一版时代**

### 与编译器优先设计的关系

- 早期编译器 **单遍扫描**：先见到调用、后见到定义，难以做类型检查
- 原型需要 **两遍或符号表**；硬件进步后才成为标准
- Unix 源码 1970s 大量 K&R 风格；移植到 ANSI 是巨大工程（Linux 早期亦如此）
