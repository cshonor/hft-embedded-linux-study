# 1.6 它很棒，但它符合标准吗

## 本节讲什么

「能编译、能跑」≠「符合 C 标准」。本节区分 **严格符合（strictly conforming）**、**符合（conforming）**、**Hosted vs Freestanding** 实现，以及 **实现定义（implementation-defined）**、**未指定（unspecified）**、**未定义（undefined）** 行为——写可移植 C、读内核/DPDK 代码、答面试题都绕不开这套分类。

## 要点

### 符合性层次

| 术语 | 含义 |
|------|------|
| **严格符合程序** | 不依赖扩展、不产生未定义行为；仅使用标准语法与库 |
| **符合程序** | 可能用实现扩展，但行为在标准允许范围内 |
| **Hosted 实现** | 完整 OS 环境：有 `main`、标准库、文件 I/O |
| **Freestanding 实现** | 引导/内核/裸机：无完整库；启动方式由实现定义 |

```text
你的 .c 源文件
    ↓
编译器 + 标准库（hosted）  或  交叉工具链 + 最小运行时（freestanding）
    ↓
严格符合？ → 可移植   |   用扩展（GNU/Microsoft）→ 换编译器可能挂
```

### 行为四分类（必考）

| 分类 | 标准是否规定结果 | 典型例子 |
|------|------------------|----------|
| **Defined** | 完全规定 | `1 + 2 == 3` |
| **Implementation-defined** | 标准列出备选，**文档必须说明** | `sizeof(int)`、`char` 有符号与否、右移负数 |
| **Unspecified** | 合法但未规定选哪种 | `i++ + i++` 求值顺序（C11 前更糟）、同一表达式内多个函数调用的顺序 |
| **Undefined** | **无保证**，编译器可任意处理 | 越界访问、有符号溢出、`NULL` 解引用、违反有效类型规则 |

### 实现定义 vs 未定义 — 对比例子

**Implementation-defined（换平台可能变，但合法）：**

```c
/* sizeof(int) 可能是 2/4/8，查 limits.h / 文档 */
printf("%zu\n", sizeof(int));

/* char 默认 signed 还是 unsigned — 编译器 -fsigned-char 决定 */
char c = 255;

/* 右移负数 — 算术右移还是逻辑右移，实现定义 */
int x = -1 >> 1;
```

**Undefined（编译通过也可能随时炸）：**

```c
int a[10];
a[10] = 0;              /* 越界：UB */

int i = INT_MAX;
i + 1;                  /* 有符号溢出：UB */

int *p = NULL;
*p = 42;                  /* 空指针解引用：UB */

/* 未序列化的副作用 */
i = i++;                  /* UB */
```

### 编译器扩展：far / near（Microsoft）

```c
/* 16 位 DOS / 早期 Windows 分段内存模型 — 非 ISO C */
char far  *fp;   /* 远指针：跨段 */
char near *np;   /* 近指针：段内 */

/* 现代 32/64 位 flat 模型下已废弃；面试问「见过 far/near 吗」→ DOS 时代 MS 扩展 */
```

- **GCC/Clang**：`__attribute__`、`__builtin_*`
- **MSVC**：`__declspec`、`#pragma` 段布局
- 使用扩展 → 程序 **不是严格符合**，但可能是 **符合的**（若扩展不破坏标准语义）

### 如何判断「你的代码是否符合标准」

```bash
# 尽量严格：ISO C + 警告
gcc -std=c11 -pedantic -Wall -Wextra -Werror prog.c

# 查看实现定义文档
man gcc | less          # GCC 文档
info gcc "C Dialect Options"
```

| 场景 | 建议 |
|------|------|
| 可移植库 | `-std=c11 -pedantic`，避免扩展 |
| Linux 用户态 | `-std=gnu11` 常用，知悉扩展边界 |
| Linux 内核 | **GNU C**，另有一套规范 |
| 嵌入式/DPDK | 工具链文档 + `-std=c11` + 禁用 UB |

### 「很棒但不标准」的例子

- **Linux 内核**：语句表达式、`typeof`、属性——强大但锁在 GCC/Clang
- **DPDK**：大量宏与编译器内建假设
- **旧 Unix 源码**：K&R 风格 + 实现细节
