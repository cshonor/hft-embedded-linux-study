# 1.10 「安静的改变」究竟有多少安静

## 本节讲什么

标准升级时，并非所有变化都会 **编译报错**。许多是 **「安静的改变（quiet change）」**：旧代码在新标准下 **仍能编译**，但 **语义已变**——可能从「碰巧能跑」变成 **UB**，或从 UB 变成 defined。本节列举 C89→C99→C11 典型 quiet change，建立升级编译选项时的警觉。

## 要点

### 什么是 quiet change

```text
旧标准下编译 + 运行「看起来 OK」
    ↓ 换 -std=c99 / 换编译器
新标准下仍编译通过
    ↓
行为变了（更严 UB / 不同结果）— 无警告 = 「安静」
```

- 与 ** loud change ** 相对：`void func()` 从「参数未声明」变为语法/约束问题（C++ 更严格）

### 典型 quiet change 清单

| 变化 | 旧行为/假设 | 新标准/现代编译器 | 风险 |
|------|-------------|-------------------|------|
| **`//` 注释** | C89 不支持 | C99+ 支持 | 旧代码 `a//b` 可能被误解析 |
| **`inline`** | 非标准 / GNU | C99 关键字 | 与 static 链接规则变化 |
| **变长数组 VLA** | 扩展 | C99 标准；C11 可选 | 栈溢出、可移植性 |
| **`long long`** | 扩展 | C99 标准 | 格式串 `%lld` |
| **`restrict`** | C99 | 别名优化更激进 | 违反 restrict → UB |
| **复合字面量** | — | C99 | 生命周期易错 |
| **for 循环声明** | — | C99 `for (int i=0; …)` | 作用域变化 |
| **混合声明与代码** | 声明须在块首 | C99 任意 | 旧 C 风格差异 |
| **printf 格式** | 较松 | 现代 `-Wformat` | 类型不匹配变警告/UB |
| **有符号整数表示** | 常假设二进制补码 | C99 仍允许 one's complement 等 | 极少平台 |
| **指针 alias** | 较松 | strict aliasing（C99 语义） | `-fno-strict-aliasing` 掩盖问题 |
| **`//` 在宏里** | — | 宏中 `//` 注释掉行尾 | 宏维护陷阱 |

### 实例 1：strict aliasing

```c
float f = 1.0f;
int *ip = (int *)&f;
*ip = 0;                    /* 违反 effective type → UB（C99 严格别名）*/
/* 「安静」：GCC -O2 可能优化掉看似无关的读写 */
```

- 旧代码「类型双关」看内存；优化后出错 **无编译错误**
- 合法做法：`memcpy`、`union`（C99 对 union 仍有争议，常用 `memcpy`）

### 实例 2：函数声明空括号

```c
void foo();     /* C：旧式，参数未检查；C++：等价 void foo(void) */
void bar(void);

foo(1, 2, 3);   /* C 中可能仍编译；C++ 中 foo() 不接受参数 */
```

- 从 K&R 迁移到 ANSI 时，**最大 loud change 是原型**；遗留 `()` 仍是 quiet 陷阱

### 实例 3：`//` 注释与 URL

```c
#define URL http://example.com   /* C89：可能 token 化错误 */
/* C99 中 // 才是注释；老宏要小心 */
```

### 实例 4：链接与 inline

```c
/* C99：inline 函数多个 TU 需规则一致，否则 ODR 式问题 */
inline int helper(void) { return 42; }
```

### 如何防御 quiet change

```bash
# 升级标准时开全警告
gcc -std=c11 -Wall -Wextra -Wpedantic old_code.c

# 对比旧/新标准行为
gcc -std=c89 -o old old_code.c
gcc -std=c11 -o new old_code.c
./old; ./new

# 动态检测 UB
gcc -fsanitize=undefined -g old_code.c
```

| 策略 | 说明 |
|------|------|
| 固定 `-std=` | 团队统一 `gnu11` 或 `c11` |
| CI 开 `-Werror` | 把 loud 问题提前 |
| UBSan/ASan | 抓 quiet 语义变化 |
| 读 **Annex J** / 编译器 **Porting** 笔记 | |

### 「究竟有多少安静」

- **很多**：尤其涉及 **别名、序列点、整数 promotion、格式串**
- **标准刻意保留**：避免 break 海量旧代码
- **Expert C 立场**：了解 quiet change 才能说「懂 C 历史包袱」

## 与后续衔接

- **1.4–1.5**：K&R → ANSI 是最大一轮；C99/C11 是后续 quiet 波
- **1.6**：UB 分类解释为何「编译过」仍可能变
- **Expert C ch02**：语言特性很多是 quiet change 的个案
- **Expert C ch08**：C vs C++ 同一写法不同语义（更「吵」）
- **内核**：`-std=gnu11` + 大量 `#ifdef __GNUC__` 隔离变化
- **面试**：「升级 C 标准有哪些 silent breakage 例子？」
