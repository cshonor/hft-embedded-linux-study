# 7.4 Cache存储器

## 本节讲什么

**Cache** 是 CPU 与主存之间的 **高速缓冲**，利用 **时间局部性** 与 **空间局部性** 隐藏内存延迟。本节介绍 **L1/L2/L3** 层次、典型 **64 字节 Cache 行**、以及多核下的 **伪共享（false sharing）**——同一 Cache 行被不同核心频繁修改导致性能暴跌，可用 **padding** 隔离（**demo04_false_sharing**）。

## 要点

### Cache 层次（典型 x86 桌面/服务器）

```text
  CPU Core
    ├─ L1I (指令)  ~32 KB   ~4 cycles
    ├─ L1D (数据)  ~32 KB   ~4 cycles
    ├─ L2 (统一)   ~256 KB  ~12 cycles
    └─ L3 (共享)   ~8–32 MB ~40+ cycles
           │
           ▼
        主存 DRAM        ~100–300 cycles
```

| 级别 | 范围 | 特点 |
|------|------|------|
| **L1** | **每核私有** | 分 **I-Cache / D-Cache**；命中最快 |
| **L2** | **每核私有**（常见） | 统一指令+数据 |
| **L3** | **多核共享** | 容量大；跨核 **一致性协议（MESI）** |
| **主存** | 全系统 | **page fault** 未命中时更慢（**7.3**） |

### Cache 行（Cache Line）

- **典型大小**：**64 B**（x86-64、ARM 常见；以 **`getconf LEVEL1_DCACHE_LINESIZE`** 为准）
- **行为**：读 **1 字节** 也会把 **整行** 载入 Cache
- **写策略**：常见 **write-back** + **MESI** 状态（Modified/Exclusive/Shared/Invalid）

```text
  地址 0x1000–0x103F  →  同一 Cache Line
  访问 0x1000 的字节  →  整行 64B 进 L1D
```

### 局部性原理

| 类型 | 含义 | 代码示例 |
|------|------|----------|
| **时间局部性** | 刚访问的数据 **很快再访问** | 循环内 **累加变量 `sum`** |
| **空间局部性** | 相邻地址 **即将访问** | **顺序扫描数组** `a[i]` |

**反模式**：

- 大 **结构体数组** 只用一个字段 → 其余字节浪费带宽（可考虑 **AoS → SoA**）
- **随机指针追逐** → Cache 命中率低
- **过大工作集** → L3 装不下，**thrashing**

### 伪共享（False Sharing）

**定义**：两个线程修改 **不同变量**，但它们落在 **同一 Cache 行** → 多核间 Cache 行 **反复失效/同步**（**cache line bounce**），**像共享锁一样慢**，但逻辑上 **无数据竞争**。

**demo04_false_sharing** 结构：

```c
/* 坏：c1 与 c2 同一 64B 行 */
typedef struct { volatile int64_t c1; volatile int64_t c2; } shared_t;

/* 好：c2 与 c1 间隔 ≥ 64B */
typedef struct {
    volatile int64_t c1;
    char pad[64 - sizeof(int64_t)];
    volatile int64_t c2;
} padded_t;
```

```bash
cd demo/demo04_false_sharing && make && ./main
# shared 通常显著慢于 padded
```

| 修复手段 | 说明 |
|----------|------|
| **padding 到 Cache 行** | 简单；浪费空间 |
| **`alignas(64)`** | C11 显式对齐 |
| **每线程局部累加 + 最后合并** | 避免跨核写同一行 |
| **SoA 布局** | 热字段分离 |

**注意**：**真共享**（同一变量多线程写）需要 **原子 / 锁**；**伪共享** 是 **性能** 问题，不是 **数据竞争**（但仍可能用 `-fsanitize=thread` 查真竞争）。

### 与对齐、结构体 padding 的关系

- **7.2 padding** 主要为 **ABI 对齐**；过大 padding 也可能 **意外** 把热字段拆到不同行（有时有益）
- **7.2 packed** 使字段挤在同一行 → **更易** 伪共享 + 未对齐访问
- **多线程 + 全局计数器数组** `counters[N]`：相邻 **`int64_t`** 易伪共享 → **`counters[i]` 按 64B 对齐**

### 测量与工具

| 工具 | 用途 |
|------|------|
| **`perf stat -e cache-misses,cache-references`** | 粗略 Cache 行为 |
| **`perf c2c`**（部分平台） | 伪共享检测 |
| **Intel VTune** | **False sharing** 分析 |
