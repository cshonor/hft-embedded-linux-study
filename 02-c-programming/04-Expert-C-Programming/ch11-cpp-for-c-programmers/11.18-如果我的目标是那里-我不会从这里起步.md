# 11.18 如果我的目标是那里，我不会从这里起步

## 本节讲什么

本书 **哲学小节之一**：作者从 **C 程序员视角** 审视 C++ — **若最终目标是简洁、可预测、贴近机器的代码（如内核、协议栈）**，**从 C++ OOP 这条路「起步」未必最优**。不是说 C++ **无用**，而是 **目标函数不同**：应用层要 **抽象与复用**；系统底层要 **透明与可控**。本节与 **11.19** 构成 **Tradeoff 对话**。

## 要点

### 「那里」指什么？

| 目标 | 更贴近的语言/风格 |
|------|-------------------|
| **Linux 内核 / 驱动** | **C + ops 表**（**11.15**） |
| **DPDK / 线速转发** | **C、无 virtual、无堆** |
| **硬实时 / 固件** | **C / 受限 C++ 子集** |
| **业务应用 / 工具链** | **C++ / 现代 STL** |

**Expert C 读者** 多在做 **「那里」** 的工作 — **ch1–10** 整本书为此服务；**ch11** 是 **桥梁**，不是 **换宗教**。

### C++ 从 OOP 起步的「绕路感」

```text
C 程序员已有:
  struct + 函数指针 + 宏 + malloc
      ↓ 直接可进内核源码

若从 C++ 课堂起步:
  class → 继承 → 多继承 → 虚函数 → 模板元编程 → ...
      ↓ 很多层才回到「内存里到底发生什么」
```

**本书立场**：**先懂 C 底层（ch1–10）再读 C++（ch11）** — 顺序 **正确**；反过来 **易浮在语法**。

### C++ 中「从这里起步」的具体痛点

1. **隐藏成本**：**`virtual`、异常、RTTI、全局 ctor** — 不在源码一眼可见
2. **编译模型复杂**：模板 **编译慢**、错误信息 **难读**
3. **ABI 脆弱**：**跨编译器/版本** 布局与 mangling
4. **过度抽象**：**设计模式堆叠** 掩盖 **简单数据流**

### 与 ch08「软件规则之难」呼应

- **ch08**：语言 **规则与实现** 有缝隙
- **C++**：规则 **更多层** — **学习曲线** 与 **踩坑面** 更大
- **C**：规则 **少而硬** — **内核选型** 原因 partly 在此

### 何时仍值得用 C++（即使目标是系统）

| 场景 | 理由 |
|------|------|
| **用户态驱动框架** | RAII、类型安全 |
| **仿真 / 测试 harness** | 快速搭建 |
| **非热路径配置** | **`std::map`** 可读性 |
| **与现有 C++ 生态集成** | 无法只用 C |

**c-to-cpp-migration-checklist.md §7**：**底层保留 C 风格** 的 **白名单**。

### 实践建议

```text
若目标是内核/DPDK:
  主路径: C + ch10 函数表 + ch11.15 ops 思维
  选修:   ch11.17 严格性 + extern "C" 混编

若目标是应用:
  主路径: ch11 OOP + STL + 现代 C++
  基础:   ch1–10 防 UB 与链接坑
```
