# 11.19 它或许过于复杂，但却是唯一可行的方案

## 本节讲什么

与 **11.18** 相对，本节承认：**大型软件** 在 **人力、时间、变更** 约束下，**纯 C + 手动纪律** 往往 **撑不住** — **C++ 的复杂度** 是 **为管理更大复杂度** 而付的 **保险费**。**模板、类层次、STL** 让 **百万行代码** 可维护；**内核** 选择 C 是 **问题规模与发布节奏** 不同，**非 C++ 本质失败**。

## 要点

### 复杂度的两种

| 类型 | 例子 | 应对 |
|------|------|------|
| **本质复杂** | 分布式协议、编译器 | **抽象** 分割问题 |
| **偶然复杂** | 手动内存、宏、void* | **语言/库** 消除 |

C++ **同时增加** 语言偶然复杂（语法）并 **减少** 工程偶然复杂（泄漏、类型错）。

### 「唯一可行」在何时成立

```text
小模块 / 热路径 / 内核:
  → C + 函数表 足够（11.15）
  → 「唯一可行」常指 C 或 受限 C++

大应用 / GUI / 游戏 / 中间件:
  → 无 STL/OOP 则 **重复发明** vector/map/字符串
  → C++ 抽象 **成为唯一可行** 的工程路径
```

**规模阈值**：当 **「手动 C 纪律」** 成本 **超过** **「C++ 学习+编译」** 成本。

### C++ 换来的东西（11.1–11.17 汇总）

| 机制 | 管理的复杂度 |
|------|--------------|
| **class + RAII** | 资源泄漏（**11.4**） |
| **virtual** | 扩展点爆炸（**11.13**） |
| **模板/STL** | 容器/算法重复（**11.17**） |
| **namespace** | 全局名冲突 |
| **类型系统** | void* 标签错误（**ch10**） |

### 与 C 哲学共存

**不是二选一**：

```cpp
/* 热路径 — C 风格 */
void fast_path(struct pkt *p) { /* 无 virtual */ }

/* 控制面 — C++ */
class Config {
    std::map<std::string, std::string> kv_;
};
```

**Linux**：内核 **C**；**用户态工具** 大量 **C++**。**DPDK**：**C API**；周边 **C++**。

### 设计权衡表

| 维度 | **偏 C** | **偏 C++** |
|------|----------|------------|
| **团队熟悉度** | 内核背景 | 应用背景 |
| **性能可预测** | 高 | 需约束子集 |
| **交付速度** | 低（手写） | 高（STL） |
| **长期维护** | 依赖规范 | 依赖类型+RAII |
| **调试透明** | 高 | 需懂 vtable/模板实例 |

### Expert C 的平衡结论

1. **懂 C 底层** 是 **用对 C++** 的前提（**vtable = ops 表**）
2. **不懂底层** 的 C++ 是 **危险浮层**
3. **系统编程** 默认 **C 主 C++ 辅**；**应用** 相反
4. 用 **[c-to-cpp-migration-checklist.md](./c-to-cpp-migration-checklist.md)** **按模块迁移**，非 **全盘重写**
