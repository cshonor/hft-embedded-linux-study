# 11.2 抽象——取事物的本质特性

## 本节讲什么

**抽象（abstraction）** 是 OOP 的第一块基石：**只保留与当前问题相关的属性与行为，忽略无关细节**。对 C 程序员而言，这并不全新—— **`FILE*`** 就是典型抽象：你只调用 **`fopen` / `fread` / `fclose`**，不必关心缓冲区、系统调用或内核页缓存。C++ 的 **`class`** 让你 **自定义** 这种「黑盒接口」，并在 **编译期** 用类型系统强制边界（比裸 **`struct` + 约定** 更硬）。

## 要点

### 抽象在 C 里已存在

| C 抽象 | 隐藏的细节 | 暴露的接口 |
|--------|------------|------------|
| **`FILE*`** | 缓冲区、fd、锁 | **`stdio`** |
| **`void*`** 通用容器 | 具体类型 | 仅「一块内存地址」 |
| **设备驱动 `ops` 表** | 硬件寄存器 | **`open/read/write/ioctl`** |

**ch07–ch10** 讲的是 **看清底层**；**抽象** 讲的是 **在合适层次故意不看底层**。

### C++ 抽象：类型即契约

```cpp
class Stack {
public:
    void push(int x);
    int  pop();
    bool empty() const;
private:
    /* 实现可换：数组 / 链表 / vector — 用户不可见 */
    int  *data_;
    size_t top_;
};
```

**用户代码** 只依赖 **`push/pop/empty`**，不依赖 **`data_` 布局** — 这就是 **11.3 封装** 与 **11.5 访问控制** 要保护的东西。

### 抽象 vs 具体

```text
具体（implementation）     抽象（interface）
─────────────────────     ─────────────────
数组下标、指针算术    →    「栈」：后进先出
寄存器位操作          →    「串口」：send/recv
红黑树节点旋转        →    「map」：insert/find
```

**设计原则**：接口 **稳定**，实现 **可替换**。C 用 **不透明指针**（**`typedef struct X X`** 不暴露定义）达到类似效果；C++ 用 **`private` + 头文件最小暴露**。

### 抽象层次与 ch04/ch08 的呼应

- **ch04**：「数组 ≠ 指针」— 抽象层选错类型会穿透到底层
- **ch08**：软件规则之难 — 抽象 **不能** 违背语言/ABI 真实规则
- **ch11**：在 **理解底层** 的前提下，用 **class 接口** 把复杂度 **关进盒子**

### 抽象过头的风险（预告 11.18–11.19）

抽象不是免费午餐：**虚函数、RTTI、模板** 都有成本。内核/DPDK 等场景常 **刻意不用** 高级抽象，保留 C 风格 **函数表 + 静态分配**（见 **11.15**, **c-to-cpp-migration-checklist.md §7**）。
