# 3.1 只有编译器才会喜欢的语法

## 本节讲什么

本章开篇点题：**C 语言的声明语法是为编译器设计的，不是为人设计的**。同一行里 `*`、`[]`、`()` 反复出现，读起来像「绕口令」，却是面试、读头文件、写函数指针时的基本功。本节说明 **为何声明难读**、**为何必须学**，并预告全书唯一的通用解法——**螺旋规则（Spiral Rule）**。

## 要点

### 声明为何让人头疼

C 声明把 **类型信息** 和 **名字** 揉在一起，且 **从右向左、由内向外** 绑定，与日常从左到右阅读习惯相反：

```c
int *p;              /* p 是指向 int 的指针 */
int *p, q;           /* 常见误解：p、q 都是指针？错！只有 p 是指针，q 是 int */
char *str[10];       /* str 是数组，元素是指向 char 的指针 */
char (*p)[10];       /* p 是指针，指向含 10 个 char 的数组 */
int (*func)(int, char); /* func 是指针，指向接受 (int,char) 返回 int 的函数 */
```

| 现象 | 原因 |
|------|------|
| `*` 贴类型还是贴名字？ | 语法上属于 **declarator**；`int* p, q` 不能表示两个指针 |
| `[]` 与 `*` 谁优先？ | **`[]` 和 `()` 优先于 `*`**（见 **3.3**） |
| 多层嵌套 | 函数返回指针、指针指向函数、数组元素是指针……组合爆炸 |
| typedef 再叠一层 | 宏替换与 typedef 作用域不同（**3.5–3.6**） |

### 「只有编译器才会喜欢」

- **历史**：B/C 单遍编译、声明与使用分离；语法极简，不引入类似 Pascal `var p: ^integer` 的显式类型后置写法。
- **符号过载**：`*`、`&`、`[]` 在表达式与声明中含义不同（**ch02 2.3** 已述）。
- **可读性牺牲**：一行声明可编码任意复杂的类型；编译器有固定文法，人脑没有。

```text
程序员期望：名字 → 类型，线性阅读
C 文法实际：declarator 递归嵌套，* [] () 按优先级绑定到 identifier
```

### 不学声明的代价

| 场景 | 后果 |
|------|------|
| 读 `stdio.h`、`signal.h` | 看不懂回调、函数指针原型 |
| 内核/DPDK 回调 | `void (*handler)(...)` 写错类型 → 编译通过、运行崩溃 |
| 与 ch04 数组 | 分不清 `char *p` 与 `char p[]` 在声明中的差异 |
| 与 ch08 类型转换 | 隐式转换建立在「先搞清真实类型」之上 |
