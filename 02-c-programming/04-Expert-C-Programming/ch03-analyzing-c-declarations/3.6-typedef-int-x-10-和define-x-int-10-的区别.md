# 3.6 typedef int x[10] 和 #define x int[10] 的区别

## 本节讲什么

`typedef` 与 `#define` 都能造「别名」，但 **机制完全不同**：typedef 在 **编译器符号表** 中定义 **完整类型**；`#define` 在 **预处理阶段** 做 **文本替换**。本节用 `typedef int x[10]` 与 `#define x int[10]` 对比 **声明展开、多变量、const、可读性**，说明为何 **类型别名用 typedef，常量/条件编译用宏**。

## 要点

### 核心对比

```c
typedef int x[10];
#define x int[10]
```

| 维度 | `typedef int x[10];` | `#define x int[10]` |
|------|----------------------|---------------------|
| **处理阶段** | 编译（符号表类型） | 预处理（文本替换） |
| **`x` 是什么** | 类型名：「含 10 个 int 的数组」 | 宏：替换成 token 序列 `int[10]` |
| **`x y, z` 含义** | y、z **都是** `int[10]` | 展开 `int[10] y, z` → **语法错误** |
| **`unsigned x y`** | 非法（`unsigned` 不能修饰数组类型名） | 展开 `unsigned int[10] y` → 可能合法但难读 |
| **作用域** | 遵循 C 作用域与 typedef 规则 | 从 `#define` 到 `#undef` 或文件尾 |
| **调试器** | 可显示类型名 `x` | 无类型信息，只见 `int[10]` |

### 多变量声明：决定性差异

```c
typedef int x[10];
x y, z;           /* OK：y 和 z 都是 int[10] */

#define x int[10]
x y, z;           /* 预处理 → int[10] y, z;  ← 非法 */
x y; x z;         /* 勉强：两行各自 OK，但宏污染 */
```

**面试一句话**：`typedef` 定义 **类型**；`#define x int[10]` 只是把 `x` 换成 `[10]` 前面的片段拼进声明符，**无法** 像类型一样与多个 declarator 组合。

### 更隐蔽的宏陷阱

```c
#define int_ptr int *
int_ptr a, b;
/* 展开：int * a, b;  → 只有 a 是指针！与 PSTR 陷阱同源（3.5） */

typedef int *int_ptr;
int_ptr a, b;
/* a、b 都是 int* */
```

| 需求 | 正确工具 |
|------|----------|
| 指针/数组/函数类型别名 | **typedef** |
| 字面常量、`MIN`/`MAX`（谨慎） | **宏** 或 **static inline**（ch02 2.2） |
| 条件编译 | **`#ifdef`**，不是 typedef |

### const 绑定差异

```c
typedef char *string_t;
const string_t s;     /* const 指针，不是 const char* */

#define string_t char *
const string_t s;     /* 预处理 → const char * s;  ← const 贴 char*，语义不同！ */
```

宏在 **const 之前** 展开，qualifier 绑定顺序可能 **与 typedef 路径不一致**——又一个 **别用宏 typedef 指针** 的理由。

### 何时仍用 `#define`

- 包含守卫、版本号、功能开关  
- 需要 **字符串化** `#`、**连接** `##`  
- 标准库常：`NULL`、`BUFSIZ`（部分环境）  

**类型层面**：C11 起可考虑 `_Generic` 做类型选择；C++ 侧 `using`；纯 C 系统头文件仍以 **typedef** 为主（`size_t`、`uint32_t`）。

### 记忆口诀

```text
typedef → 「这是一个类型，名字叫 x」
#define  → 「见到 x 就替换成一串字符」
```

复杂声明 **先 typedef 拆层**（3.5），**不要用宏伪造 typedef**。
