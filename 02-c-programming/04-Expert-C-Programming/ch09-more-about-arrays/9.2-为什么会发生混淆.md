# 9.2 为什么会发生混淆

## 本节讲什么

既然 **9.1** 说多数场合数组「像指针」，为何又说 **数组 ≠ 指针**？本节集中解释 **三类混淆源**：**`sizeof` 的双面性**、**`&arr` 与 `arr` 的不同步长**、**跨文件 `extern` 把数组声明成指针**。这些正是 **ch04 4.2 段错误** 与 **链接期类型不匹配** 的根源；搞清它们，才能避免「代码能编译、运行却崩溃」。

## 要点

### 混淆根源一览

| 来源 | 直觉（错） | 实际规则 |
|------|------------|----------|
| **`sizeof`** | 数组名永远算数组大小 | **形参、指针变量** 里 `sizeof` 是指针宽 |
| **`&arr` vs `arr`** | 都是「数组地址」 | **`arr`** decay 为 `&arr[0]`；**`&arr`** 类型 `T(*)[N]`，步长不同 |
| **`extern` 声明** | `extern char *arr` 与定义 `char arr[]` 一样 | **类型不同** → 链接通过、运行崩溃（**ch04 4.2**） |
| **二维数组** | `int **` 可接收 `int[][4]` | **必须** 保留内维，用 `int (*)[4]`（**9.3, 9.6**） |

### `sizeof` 例外（最易考）

```c
int local[10];

void foo(int arr[]) {
    printf("%zu\n", sizeof arr);  /* 指针宽度！4 或 8，不是 40 */
}

int main(void) {
    printf("%zu\n", sizeof local); /* 40 — 定义处看见真数组 */
    foo(local);
}
```

| 语境 | `sizeof` 对象 | `int a[10]` 结果（int=4B） |
|------|---------------|---------------------------|
| **定义数组的作用域** | 整个数组 | `40` |
| **函数形参 `int a[]`** | 指针 `int*` | `4` 或 `8` |
| **指针变量 `int *p`** | 指针本身 | `4` 或 `8` |
| **二维 `int m[3][4]`** | 整个二维块 | `3*4*4 = 48` |
| **二维行 `m[0]`** | 一行 `int[4]` | `16` |

**面试一句话**：**只有编译器「知道数组边界」的语境，`sizeof(数组名)` 才返回总字节数**；形参处边界信息已丢失。

### `&arr` 与 `arr`：地址相同，类型不同

对 `int arr[10]`：

```c
printf("%p %p\n", (void *)arr, (void *)&arr);  /* 数值常相同 */
printf("%td %td\n",
       (char *)(arr + 1) - (char *)arr,      /* +1 跳 4 字节：一个 int */
       (char *)(&arr + 1) - (char *)&arr);   /* +1 跳 40 字节：整数组 */
```

| 表达式 | 类型 | `+1` 步长 |
|--------|------|-----------|
| **`arr`**（退化后） | `int*` | `sizeof(int)` |
| **`&arr`** | `int (*)[10]` | `10 * sizeof(int)` |
| **`&arr[0]`** | `int*` | 与 `arr` 退化后相同 |

**混淆点**：打印地址时 **`arr` 与 `&arr` 数值相等**，但 **指针算术步长完全不同**——类型决定步长（**ch03 声明螺旋**、**ch10**）。

### 二维的 `sizeof` 与 `&arr`（**9.6** 预习）

```c
int m[3][4];
sizeof m;      /* 48 — 整块 */
sizeof m[0];   /* 16 — 一行 */
sizeof &m;     /* 指针宽度 — &m 是指向 int[3][4] 的指针 */
```

**`&m + 1`** 跳过 **整个 `int[3][4]`**；**`m + 1`**（`m` 先 decay 为 `int (*)[4]`）只跳过 **一行**。见 **demo04_ptr_arith_2d**。

### `extern` 陷阱（ch04 4.2–4.4 核心）

**定义**（`arr_def.c`）：

```c
char arr[] = "hello";   /* 类型 char[6]，含 '\0' */
```

**错误声明**（`use_wrong.c`）：

```c
extern char *arr;       /* 类型 char* — 与定义不匹配！ */
```

**正确声明**：

```c
extern char arr[];      /* 或 extern char arr[6]; */
```

| | **`extern char arr[]`** | **`extern char *arr`** |
|--|-------------------------|------------------------|
| **链接** | 符号类型为数组 | 符号类型为指针 |
| **内存布局** | 期望连续 6 字节 | 期望一个指针变量 |
| **典型后果** | ✅ 正常 | ❌ 把 `'h'` 的 ASCII 当地址解引用 → **SIGSEGV** |

**关键**：形参等价 **`int a[]` ≡ `int *a`** **绝不能** 推广到 **`extern`**（**9.3** 反复强调）。

### 其他常见混淆

| 写法 | 问题 |
|------|------|
| **`sizeof(char *str[10])`** | 只算 10 个指针，**不含** 字符串本体 |
| **`sizeof(char str[10][20])`** | `10*20` 字节，字符串内嵌数组 |
| **把 `argv` 当成 `char argv[][]`** | 实为 **`char *argv[]`**（指针数组，**9.5**） |
| **`malloc(N*sizeof(int*))` 当二维数组** | 行不连续，与 `int[N][M]` 布局不同 |
