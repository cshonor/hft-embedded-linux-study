# 6.8 setjmp和longjmp

## 本节讲什么

**`setjmp` / `longjmp`**（`<setjmp.h>`）实现 **非本地跳转**：从深层调用 **直接跳回** 较早的 **`setjmp` 标记点**，绕过正常 **`return` 链**。本节说明其 **栈帧失效** 语义、与 **6.5 活动记录** 的冲突，以及 **不可返回后仍使用 auto 局部** 等陷阱。

## 要点

### 基本用法

```c
#include <setjmp.h>
jmp_buf env;

void deep(void) {
    if (some_error)
        longjmp(env, 1);   /* 跳回 setjmp，返回值非 0 */
}

void caller(void) {
    if (setjmp(env) == 0) {
        deep();            /* 正常路径 */
    } else {
        /* longjmp 着陆：错误处理 */
    }
}
```

| 函数 | 行为 |
|------|------|
| **`setjmp(env)`** | 保存 **寄存器 + 栈上下文** 到 **`jmp_buf`**；首次返回 **0** |
| **`longjmp(env, val)`** | 恢复上下文；**`setjmp` 仿佛再次返回 `val`（非 0）** |

### 与栈帧的关系

```text
  caller: setjmp ──► A() ──► B() ──► longjmp
                              │
                              └── 跳过 B、A 的正常 ret
                                  栈指针「弹回」setjmp 时快照
```

- **6.5** 中 **B、A 的栈帧** 在 **`longjmp` 后不再按 C 规则销毁**
- **RA / 局部变量** 占用的栈空间可能被 **后续调用复写**
- 因此：**从 `setjmp` 到 `longjmp` 之间、在 **`longjmp` 之后仍「存活」的 auto 局部变量，若未被 volatile，其值不确定**（C 标准）

### 必须遵守的规则（C / POSIX）

| 规则 | 原因 |
|------|------|
| **`longjmp` 不得跳转到其他线程的 `setjmp`** | 栈不属于同一线程（**6.7**） |
| **`jmp_buf` 仅在 setjmp 作用域有效**，直到 matching longjmp | 栈上移后 **缓冲内容无效** |
| **勿 longjmp 穿过 C++ 析构 / VLA 生命周期** | C++ 中更危险；C 中 **VLA** 同理（**6.13**） |
| **`longjmp` 后不可依赖未 volatile 的 auto 局部** | 编译器优化 + 栈复用 |

```c
void bad(void) {
    int x = 42;
    if (setjmp(env) == 0) {
        foo();              /* 可能 longjmp 跳出 */
    }
    printf("%d\n", x);      /* x 可能不是 42 — 未定义或不可靠 */
}
```

### 典型合法用途

| 场景 | 说明 |
|------|------|
| **统一错误处理** | 深层解析失败 **longjmp** 到顶层 cleanup（类似 **goto done**） |
| **解释器 / 协程雏形** | 旧式；现代多用 **异常 / 显式状态机** |
| **信号处理**（慎用） | **`sigsetjmp` / `siglongjmp`** 保存信号掩码（**6.13**） |

### 与 return 的区别

| | **`return`** | **`longjmp`** |
|---|--------------|---------------|
| 销毁栈帧 | 按序 **pop** | **一次性** 恢复旧 SP |
| 调用析构 / cleanup | C 无析构；**goto 出块** 也跳过 |
| 可跨多少层 | 一层 | **多层** |

### 现代替代

- **`goto err`** + **单函数内 cleanup**（Linux 内核风格）
- **C++ exceptions**（若可用）
- **coroutine / async** 库

**面试**：**`setjmp` 保存了什么？** → 至少 **SP、PC、 callee-saved 寄存器** 等足以恢复执行点的机器相关状态（**`jmp_buf` 不透明**）。
