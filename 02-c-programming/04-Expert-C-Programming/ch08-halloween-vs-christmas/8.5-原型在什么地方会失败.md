# 8.5 原型在什么地方会失败

## 本节讲什么

原型不是银弹。本节列举 **编译器仍无法（或尚未）拦住** 的错误：**声明与定义不一致**、**结构体/指针混淆但 ABI 碰巧相同**、**可变参数 `...`**、**跨翻译单元旧式声明**、**链接时符号涂改（linkage mismatch）**。理解这些边界，才能明白为何 **8.8 的 UB** 与 **8.9 的 cast** 仍会在「有原型」的代码里出现——**类型系统覆盖不到求值顺序**。

## 要点

### 声明与定义不匹配

```c
/* file1.c */
int foo(int x) { return x; }

/* file2.c — 错误声明 */
double foo(double x);   /* 链接可能仍成功（名字相同），调用约定/解释错误 → 灾难 */
```

| 失败模式 | 原因 | 后果 |
|----------|------|------|
| **返回类型不同** | C **不要求** 跨 TU 返回类型诊断（旧链接） | 返回值 **按错误类型解释** |
| **指针 vs 整数** | 宽度相同（如 64 位） | **静默错误** |
| **`char*` vs `const char*`** | 形参兼容规则 | 可能 **const 丢失** |

现代编译器 **`-Wlto` / 全程序优化** 可抓部分跨 TU 错误；**传统 `.o` 链接** 仍可能 **只报 warning 或不报**。

### 可变参数：`printf` 的原型局限

```c
int printf(const char *fmt, ...);
```

- 编译器 **仅检查** 固定参数（ **`fmt`** ）
- **`...` 部分无类型** → **`%d` 配 `long long`**、 **`%s` 配非指针** → **UB**
- **默认提升**： **`float` 必须写 `%f` 但传 **`double`**； **`char` 传 **`int`**

**工程规则**： **`printf` 格式串与实参一一核对**；生产代码优先 **`snprintf`** + 静态检查（ **`__attribute__(format)`** ）。

### 旧式声明与混合风格

```c
/* header 里旧式 */
double atof();

/* 调用 */
atof("3.14");   /* 现代：仍可能链接 libc 的 double atof(const char*) */
```

**一个 TU 用原型、另一个无原型** → **C 标准未定义**（C99 约束）。维护 **单一头文件** 是唯一稳妥做法。

### Linkage 与 `static` / `inline`

| 问题 | 说明 |
|------|------|
| **`static` 函数重复名** | 各 TU 独立，**无链接冲突** |
| **`inline` 无外部定义** | C99 **`extern inline`** 规则易踩坑 |
| **C 与 C++ 链接** | **`extern "C"`** 与 **重载** 不匹配 |

### 原型救不了「表达式 UB」（8.8）

```c
void f(int i);
f(i++ + ++i);   /* 原型正确；实参求值顺序仍 UB */
```

**类型检查 ≠ 副作用顺序检查**——这是 **8.8 Christmas 问题** 仍存在的根本原因。

### 与 ch05 链接错误对照

- **ch05**：**符号解析**、**强/弱符号**、**COMMON**
- **8.5**：**符号同名但类型语义不同** — **链接「成功」但运行错误**，比 **undefined reference** 更阴险

## 与后续衔接

- **8.8**：**求值顺序**、**序列点** — 原型完全不管
- **8.9**：程序员用 **cast「骗过」原型** → **`void*` 丢失类型**
- **8.6**：**`getchar` 返回 `int`** — 用 **`char` 接** 是 **原型正确用法** 仍可能 **逻辑错**
- **ch03**：复杂函数指针声明（**3.8**）；**ch05**：**one definition rule**
- **ch09/ch10**：数组形参 **退化指针** 与原型 **表面一致、语义不同**
