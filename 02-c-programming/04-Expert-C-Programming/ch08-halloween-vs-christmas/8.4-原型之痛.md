# 8.4 原型之痛

## 本节讲什么

**函数原型（function prototype）** 是 ANSI C 最重要的改进之一：在 **调用前** 声明 **参数个数与类型**，使编译器能 **检查实参**、自动插入 **默认参数提升（default argument promotions）**。本节回顾 **K&R C** 时代 **无原型** 的写法及其痛苦——与 **8.1** 的 Portzebie 类似，**靠约定而非编译器** 维系类型正确；并梳理 **隐式 `int`** 历史、**旧式声明 vs 原型声明** 的语法差异。

## 要点

### K&R 风格 vs ANSI 原型

**K&R 定义**（参数类型写在函数体 **内**）：

```c
/* 旧式声明：编译器不检查实参类型 */
int foo();
foo(x, y);   /* 可能 silently wrong */

/* K&R 定义 */
int foo(a, b)
    int a;
    char *b;
{
    ...
}
```

**ANSI 原型**：

```c
int foo(int a, char *b);   /* 原型：完整类型信息 */
int foo(int, char *);      /* 重复声明可省略参数名 */
```

| 特性 | **无原型 `()`** | **原型 `(int, char*)`** |
|------|-----------------|-------------------------|
| 实参类型检查 | ❌ | ✅ |
| **`()` 含义（C++ 不同）** | **旧式：不检查** | N/A |
| 空参 **`void foo(void)`** | **`foo()` ≠ 无参保证** | **`void` 显式无参** |
| 链接 | 仍参与 **外部链接** | 同左 |

**C 规则**：**定义若用原型，声明也必须用原型**；混用是 **约束违规（constraint violation）** 或 **未定义行为**（视标准版本与诊断）。

### 隐式 `int` 的历史

K&R 时代 **未写类型** 的函数返回值 **默认为 `int`**：

```c
/*  ancient */
foo() { return 1; }   /* 返回 int — 现代编译器报错 */
```

C99 起 **删除隐式 int**；遗留代码、老教材是 **面试 trivia**。与 **8.1 `031`** 同属 **「老 C 习惯在现代是坑」**。

### 默认参数提升（Default Argument Promotions）

**无原型调用** 或 **可变参数 `...`** 时， **`float` → `double`**， **`char/short` → `int/unsigned int`**：

```c
void old();
old('a');        /* char 提升为 int 传递 */
printf("%c", c); /* 必须传 int（提升后） */
```

**有原型** 的普通调用 **不做** 这种提升到 **`int`** 的转换（类型必须 **完全匹配或可赋值转换**）—— **`short` 实参给 `short` 形参** 不会先变 **`int`** 再截断。

### 原型如何减轻「等待中类型变化」（8.3）

原型让 **隐式转换发生在编译期可见处**：

```c
void bar(unsigned u);
bar(-1);   /* -1 转为 unsigned → 巨大值；有原型时编译器可能 warn */
```

**`-Wall`** 对 **sign-conversion** 的警告应打开（**ch02** 工具链习惯）。

### 工程规则

- **头文件** 放 **完整原型**；**`.c`** 中 **定义与声明一致**
- **禁止** 新代码使用 **无原型 `()`**；用 **`(void)`** 表示无参
- **`static` 函数** 也写原型（文件内 **第一个定义前** 或头文件）
