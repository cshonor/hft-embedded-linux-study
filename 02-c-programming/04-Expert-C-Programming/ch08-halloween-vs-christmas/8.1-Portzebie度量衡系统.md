# 8.1 Portzebie 度量衡系统

## 本节讲什么

本章标题 **「Why Programmers Can't Tell Halloween from Christmas」** 是一句程序员圈内著名的 **双关笑话**：**八进制 `031` 看起来像 10 月 31 日（万圣节）**，而 **`031` 的十进制值恰好是 `25`——12 月 25 日（圣诞节）**。程序员「分不清」的，不是日历，而是 **同一串字面量在 C 里按八进制还是十进制解释**。

本节以 **Mad 杂志** 虚构的 **Portzebie 度量衡** 开篇：荒诞的单位换算（如 1 Portzebie = 1.33 个标准单位）类比 C 语言里 **看似合理、实则陷阱重重的「换算规则」**——**字面量进制、隐式类型转换、运算符优先级**。全书第 8 章的核心命题：**软件（语言规则）比硬件（位模式）更难推理**；从 Portzebie 的胡闹到 `031` vs `25`，都是在提醒：**不查规则就写代码，等于在用 Portzebie 做工程**。

## 要点

### 万圣节 vs 圣诞节：八进制字面量陷阱

C 以 **`0` 开头** 的整数字面量是 **八进制**，不是十进制：

```c
int halloween = 031;   /* 八进制：3×8 + 1 = 25 */
int christmas = 25;    /* 十进制：25 */

/* halloween == christmas  →  true */
printf("%d %d\n", 031, 25);  /* 输出：25 25 */
```

| 写法 | 程序员直觉 | 实际值 |
|------|------------|--------|
| **`031`** | 像「10 月 31 日」 | **25**（八进制） |
| **`25`** | 「12 月 25 日」 | **25**（十进制） |
| **`0`** 前缀 | 有时当「填充零」 | **八进制标志** |

**工程规则**：

- 除 **位掩码、权限位** 等明确场景外，**禁止** 用前导 `0` 写十进制数；`008` 甚至 **编译错误**（八进制无数字 8）。
- 位常量用 **`0x` 十六进制** 或 **`(1u << n)`**，可读且无二义性。
- Code review 看到 **`0\d+`** 应警觉：是八进制还是手滑？

### Portzebie 度量衡的寓意

书中 **Portzebie** 来自 Mad 杂志，是一套用 **完全不合理换算** 堆出来的单位体系。作者用它映射 C 的 **「隐式规则」**：

| Portzebie 式胡闹 | C 语言对应 |
|------------------|------------|
| 单位名称听起来正经 | **`unsigned` 静默提升**、**usual arithmetic conversions**（**8.3**） |
| 换算表要背才能用 | **运算符优先级表**（**8.8**、**[operator-precedence-cheatsheet.md](./operator-precedence-cheatsheet.md)**） |
| 混用单位出灾难 | **`=` vs `==`**、**`&` vs `&&`**（**8.2**、**8.8**） |
