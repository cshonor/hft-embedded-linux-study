# 4.4 使声明与定义相匹配

## 本节讲什么

**4.3** 区分了声明与定义；本节给出 **可操作的匹配规则**：当数组在一个编译单元 **定义**、在其它单元通过 `extern` **引用** 时，声明符必须 **逐字对应类型结构**——`[]` 与 `*` **不可互换**。同时说明 **数组长度** 在声明/定义两侧如何省略或写全，以及头文件、多文件工程中的 **最佳实践**。

## 要点

### 黄金法则

> **定义用什么类型，extern 声明就必须用同一类型。**

| 定义（某个 .c） | 正确声明 | 错误声明 |
|-----------------|----------|----------|
| `char arr[] = "hello";` | `extern char arr[];` | `extern char *arr;` |
| `int table[256];` | `extern int table[];` 或 `extern int table[256];` | `extern int *table;` |
| `char *msg = "err";` | `extern char *msg;` | `extern char msg[];` |

**4.2** 崩溃 = 违反本条 + C 链接 **无类型检查**。

### 为何 `extern char arr[]` 而非 `extern char *arr`

```c
/* defs.c — 定义 */
char arr[] = "hello";
```

```c
/* use.c — 声明 */
extern char arr[];
```

编译器在 `use.c` 中知道：

- `arr` 的类型是 **数组**（长度不完整但 **元素连续**）；
- `arr` **就是** 首元素地址，**不是** 「存地址的指针变量」；
- `arr[i]` 等价于从 `arr` 首地址偏移 `i` 个 `char`。

若写成 `extern char *arr`：

- 编译器认为存在 **指针对象** `arr`；
- 访问路径变成 **load 指针值 → 再解引用**；
- 与定义侧内存布局 **不一致**（**4.2** 逐步推演）。

### 数组长度的匹配规则

```c
/* 定义 */
char buf[100] = {0};

/* 声明 — 以下均可（若兼容） */
extern char buf[];      /* 省略长度：最常见 */
extern char buf[100];   /* 完全匹配 */
extern char buf[80];    /* C：长度可更小？不 — 声明长度须 ≥ 定义使用范围，且兼容规则以标准为准 */
```

**实践建议**：

| 做法 | 说明 |
|------|------|
| `extern char buf[];` | 推荐：长度只在 **定义处** 维护一处 |
| `extern char buf[N];` | 头文件用 `#define N` 统一，定义也用 `N` |
| 不要在声明侧随意改 `[]` 为 `*` | 即使「用起来像字符串」 |

**注意**：`extern char arr[6]` 与 `char arr[] = "hello"`（6 含 `\0`）兼容；声明长度 **不能小于** 实际元素个数（否则越界风险）。

### 头文件模板

```c
/* shared.h */
#ifndef SHARED_H
#define SHARED_H

extern char arr[];           /* 数组：与定义 char arr[] = ... 配对 */
extern int *index_table;     /* 指针：与定义 int *index_table = ... 配对 */

void print_first(void);

#endif
```

```c
/* arr_def.c */
#include "shared.h"

char arr[] = "hello";

void print_first(void) {
    printf("arr[0]=%c\n", arr[0]);
}
```

```c
/* main.c */
#include "shared.h"

int main(void) {
    print_first();
    printf("%s\n", arr);
    return 0;
}
```

**demo02_extern** 即此模式的极简版：`arr_def.c` + `use_correct.c` / `use_wrong.c`。

### 函数声明与数组声明的对比

| 对象 | 声明可省略的信息 | 仍须匹配的核心 |
|------|------------------|----------------|
| 函数 | 参数名可省略 | 返回类型、参数 **类型列表** |
| 数组 | `extern` 可省略长度 | **必须是数组** 不能变指针 |
| 指针 | — | 指向类型必须一致 |

函数形参 `void f(int a[])` 与 `void f(int *a)` **等价**——那是 **函数形参的特殊规则**（**4.5**），**不适用于** 文件作用域的 `extern`。

### 链接器视角（为何必须人眼保证）

```text
编译器（各 .c）:  类型检查仅在单个 TU 内
       ↓
链接器:            只合并同名符号地址，不验证 char* vs char[]
       ↓
运行:              错误声明 → 错误访存指令
```

**ch05 5.3** 会讨论链接的更多「秘密」；**C++** 通过 mangling 在链接期做类型检查，**C 没有**。

### 自检清单

1. 定义是 `T name[N]` 还是 `T *name`？
2. 头文件 `extern` 是否与之一致（`[]` vs `*`）？
3. 是否在 **仅一处** `.c` 定义，其余只声明？
4. 字符串全局：定义常用 `char s[] = "..."` 还是 `char *s = "..."`？对应 extern 不同。
5. 用 **demo02** 验证：正确版输出 `hello`，错误版崩溃。

### 与 const 的组合

```c
const char msg[] = "read-only array";
extern const char msg[];   /* 正确 */

const char *p = "literal";
extern const char *p;      /* 正确，与指针定义配对 */
```

**不要** `const char msg[]` 定义却 `extern const char *msg` 声明。
