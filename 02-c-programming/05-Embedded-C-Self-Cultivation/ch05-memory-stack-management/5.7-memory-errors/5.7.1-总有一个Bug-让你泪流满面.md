# 5.7.1 总有一个 Bug，让你泪流满面

## 本节讲什么

内存 Bug **复现难、定位慢**：时好时坏、Release 才崩、多线程竞态。本节建立 **系统化排查流程**：稳定复现 → 最小用例 → 工具（GDB/Valgrind/ASAN/core）→ 对照源码假设。

## 要点

### 排查顺序

1. **可复现吗？** 固定输入、`-O0 -g`、单线程先查
2. **哪类错误？** SIGSEGV/SIGABRT/glibc corruption message
3. **工具**：Valgrind > ASAN > mtrace > 肉眼
4. **ARM 板子**：无 Valgrind 时用 **JTAG/GDB**、`printk`、栈回溯

```bash
gcc -g -O0 -fno-omit-frame-pointer -o bug bug.c
ulimit -c unlimited
./bug
gdb ./bug core
(gdb) bt full
```

### 内核/DPDK

- **Kernel BUG/oops**：`call trace` + `addr2line` / `decode_stacktrace`
- **DPDK**：开启 `RTE_LIBRTE_MALLOC_DEBUG`、mbuf 断言

### 心态

- 假设 **自己的越界** 先于怀疑编译器
- Git bisect 找引入提交的 commit
