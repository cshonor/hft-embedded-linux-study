# 7.2.1 为什么要数据对齐

## 本节讲什么

对齐不是编译器「浪费内存」，而是**硬件访问约束与性能**的折中：总线一次读取宽度固定，跨边界访问需多次事务或触发异常。理解动机后，才能合理解释结构体 padding、为何 DMA 缓冲区要 cache line 对齐。

## 要点

**硬件原因**：

1. **原子宽度**：32 位寄存器读 4 字节，地址末两位为 0 时一次完成
2. **跨行访问**：未对齐可能拆成两次总线周期
3. **MMU / 外设**：部分 SoC 外设寄存器必须字对齐

```
地址 0x1001 读 uint32_t（需 0x1000 对齐）:
  未对齐 → 两次 16/32 位访问或 fault
```

**性能**：

| 对齐 | 典型代价 |
|------|----------|
| 自然对齐 | 1 次内存访问 |
| 跨 cache line | 2 次 line fill |
| 严格未对齐（ARM） | 内核 alignment trap 或 SIGBUS |

```c
/* 缓存行对齐，避免 false sharing（多核） */
#define CACHE_LINE 64
struct alignas(CACHE_LINE) per_cpu_counter {
    long count;
    char pad[CACHE_LINE - sizeof(long)];
};
```

```bash
# ARM 未对齐访问示例（用户态可能 SIGBUS）
# 内核: /proc/cpu/alignment 控制对齐检查级别
```

- **协议结构体**：为紧凑用 `packed`，牺牲 CPU 访问速度换布局确定
- **原子变量**：`stdatomic` / 内核 `atomic_t` 常要求自然对齐
