# 7.5.5 如何避免typedef被大量滥用

## 本节讲什么

typedef 是工具而非目的。过度别名使代码像「另一种语言」，IDE 跳转困难，新人无法判断是否为指针。本节给出内核/嵌入式社区普遍接受的**节制清单**，与 Linux 风格文档精神一致。

## 要点

**宜 typedef**：

- 长度随平台变化的类型（`size_t` 已有标准）
- 函数指针、复杂函数签名
- 对外 API 的 opaque 句柄
- 协议/硬件相关的语义单位（`baud_t`、`vlan_id_t`）

**不宜 typedef**：

- `typedef int bool_t`（用 `<stdbool.h>`）
- `typedef struct { ... } foo_t` 仅为了省写 `struct`
- 把 `char **` 包成 `strings_t` 却无额外语义
- 每一层调用都再造别名

```c
/* 推荐：语义清晰 */
typedef void (*irq_handler_t)(int irq, void *dev_id);

/* 不推荐 */
typedef int INT;
typedef INT *PINT;
```

| 自检问题 | 若答「否」则慎用 |
|----------|------------------|
| 别名是否表达新语义？ | |
| 换平台是否只需改一处？ | |
| 去掉 typedef 是否明显更难读？ | |

- **与宏对比**：需要条件编译用 `#ifdef ARCH_xxx`，别用 typedef 假装多态
- **审查**：代码评审质疑无意义 `_t` 后缀
