# 7.5 为什么很多人编程时喜欢用typedef

## 本节讲什么

`typedef` 为已有类型创建**别名**，不改变底层表示。内核与驱动大量用 `u32`、`pid_t`、`spinlock_t` 提升可读性并隔离平台差异。滥用 typedef 会隐藏指针语义、阻碍工具理解代码；本节建立「何时 typedef、何时避免」的整体观。

## 要点

```c
typedef unsigned long uintptr_kernel;

typedef struct list_head {
    struct list_head *next, *prev;
} list_head_t;   /* 结构体 typedef 常见 */
```

| 合理用途 | 示例 |
|----------|------|
| 平台宽度抽象 | `phys_addr_t`、`dma_addr_t` |
| 函数指针别名 | `typedef int (*calc_fn)(int,int);` → **7.12** |
| 简化复杂声明 | `typedef char *string_t;`（争议） |
| opaque 类型 | `typedef struct device device_t;` 头文件隐藏定义 |

| 滥用信号 | 问题 |
|----------|------|
| `typedef char *PSTR` | 指针语义被「类型名」掩盖 |
| 每层都 typedef | 跳转定义成本高 |
| 与 `#define` 混用 | 预处理与类型系统脱节 |

- **不改变类型属性**：`typedef const int CI` 与 `const int` 在 const 组合时易混淆 → **7.5.3**
- **内核风格**：小写 + `_t` 后缀（用户态保留 `_t` 给 POSIX）
- **C++ 互操作**：typedef 与 `using` 类似，头文件共享时注意
