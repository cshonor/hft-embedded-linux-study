# 10.5.2 文件系统的挂载

## 本节讲什么

**挂载（mount）** 把某设备上的文件系统 **附加** 到统一目录树的某个 **挂载点**。本节说明 `mount` 命令、 **`/etc/fstab`**、嵌入式 **自动挂载脚本** 与 **namespace**，衔接 **10.5.3 根 FS**。

## 要点

**挂载语义**：

```bash
mount -t ext4 /dev/mmcblk0p2 /mnt/data
# 访问 /mnt/data/foo 即访问该分区上的 foo
```

- 挂载点必须是 **空目录**（或已挂载可被覆盖）
- 一设备一 FS 类型需匹配（ext4 分区不能 mount vfat）

**内核数据结构**：

```
super_block（每 FS 实例）
    └── mount 结构 ──► 挂载点 dentry
            └── 根 inode of mounted FS
```

**fstab 开机自动挂载**：

```
# /etc/fstab
/dev/mmcblk0p2  /userdata  ext4  defaults,noatime  0  2
/dev/mmcblk0p1  /boot      vfat  defaults          0  2
```

**嵌入式常见流程**：

1. Bootloader 加载 kernel + **initramfs**（**10.5.3**）
2. 内核 mount root
3. `init` 脚本 mount `/proc` `/sys` `/dev`
4. 再 mount **data 分区** 到 `/var` 或 `/userdata`

**伪文件系统**：

```bash
mount -t proc proc /proc
mount -t sysfs sysfs /sys
mount -t devtmpfs devtmpfs /dev
```

**容器与 namespace**：

- **mount namespace**：进程 A 看到 `/` 是 overlay，进程 B 不同
- Docker 镜像层即多次 mount 组合

**RTOS**：

- FatFs `f_mount(&fs, "0:", 1)` 类似逻辑，无内核 VFS

**排错**：

| 现象 | 可能原因 |
|------|----------|
| `wrong fs type` | 类型错或未 mkfs |
| `can't read superblock` | 分区偏移错、介质损坏 |
| `device busy` | 有进程 cwd 在该 FS |
