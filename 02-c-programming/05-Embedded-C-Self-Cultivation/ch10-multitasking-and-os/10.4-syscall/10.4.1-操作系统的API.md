# 10.4.1 操作系统的API

## 本节讲什么

应用程序看到的 **API** 往往是多层封装：POSIX、C 标准库、OS 专有扩展。本节理清 **库函数 vs 系统调用 vs 内核 ioctl**，避免误以为 `printf` 一定是 syscall，并为 **10.4.4 Linux 接口** 打基础。

## 要点

**三层模型**：

```
应用代码
   │
   ▼
┌─────────────────────────────────┐
│  libc / libpthread / libm       │  ← 用户态库
└──────────────┬──────────────────┘
               │ 部分函数内部 syscall
               ▼
┌─────────────────────────────────┐
│  内核：VFS、调度、网络栈、驱动    │
└─────────────────────────────────┘
```

**是否触发 syscall 示例**（Linux）：

| 函数 | 通常行为 |
|------|----------|
| `strlen` | 纯用户态 |
| `malloc` | 多数在用户态；brk/mmap 不足时才 syscall |
| `read`/`write` | **syscall** |
| `printf` | 缓冲后 `write(1, ...)` → syscall |
| `pthread_create` | **clone** 等 syscall + 库调度 |

**RTOS API 分类**（FreeRTOS）：

| 类别 | API 例 |
|------|--------|
| 任务 | `xTaskCreate`, `vTaskDelete` |
| 同步 | `xSemaphoreCreateMutex`, `xQueueSend` |
| 时间 | `vTaskDelay`, `xTimerCreate` |
| 内存 | `pvPortMalloc`（可配置） |
| Hook | `vApplicationIdleHook`, `vApplicationStackOverflowHook` |

- 均为 **链接进镜像的函数调用**，非 Linux 式 syscall 号

**嵌入式 HAL vs OS API**：

```
app → RTOS API → 驱动/HAL → 寄存器（10.8）
app → Linux read() → 内核驱动 → 寄存器
```

**错误处理风格**：

- RTOS：`pdPASS` / `pdFAIL`，或 `configASSERT`
- Linux：`-1` + `errno`；库函数返回 `NULL`/`EOF`
