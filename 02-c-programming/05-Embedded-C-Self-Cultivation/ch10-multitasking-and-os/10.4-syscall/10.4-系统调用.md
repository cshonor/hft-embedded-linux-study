# 10.4 系统调用

## 本节讲什么

**系统调用（System Call）** 是用户程序请求 **操作系统内核服务** 的受控入口：打开文件、创建进程、映射内存等。本节对比 **RTOS 无 syscall 直调内核 API** 与 **Linux 用户态/内核态边界**，为 **10.4.1–10.4.4** 展开，并与 **10.9 MMU**、**10.10 进程线程** 呼应。

## 要点

**为什么需要 syscall**：

- 硬件、内存、全局资源不能交给任意用户代码直接操作
- 内核在 **特权级** 统一校验参数、拷贝缓冲区、维护内核数据结构

**调用链对比**：

```
RTOS 应用:
  xQueueSend() ──► 内核函数（同地址空间，可能仍用户/特权取决于 MPU）

Linux 应用:
  write() ──► libc ──► syscall 指令 ──► 内核 sys_write()
```

**syscall 典型编号**（Linux x86/ARM）：

- 用户把 **编号 + 参数** 放寄存器 → `svc`/`syscall` → 内核查 **syscall 表**

**与 API 的关系**（**10.4.1**）：

| 层次 | 示例 |
|------|------|
| **系统调用** | `read`, `fork`, `mmap` |
| **C 库 API** | `fopen`, `pthread_create`（内部可能调 syscall） |
| **POSIX** | 可移植接口规范，多种 OS 实现 |

**RTOS 侧**：

- FreeRTOS API **不是** Linux syscall；任务与内核 **共享地址空间**
- 有 MPU 时可通过 **SVC** 做特权切换（较少见）

**性能**：

- syscall 涉及 **模式切换 + 参数拷贝** → DPDK 等绕过内核 **poll 模式**
- 嵌入式 RTOS API 调用 ≈ 普通函数 + 可能调度
