# 10.2.3 可重入函数

## 本节讲什么

多任务与中断并发下，同一函数可能被 **重入**（在未执行完时再次被调用）。本节定义 **可重入（Reentrant）**、**线程安全** 与 **可重入** 的区别，给出嵌入式 C 的编写准则，衔接 **10.2.4 临界区** 与 **demo04/05** 中的共享资源保护。

## 要点

**术语对比**：

| 术语 | 含义 |
|------|------|
| **可重入** | 多个执行流 **同时** 进入同一函数，各自使用 **独立数据**，结果正确 |
| **线程安全** | 多线程调用不出错（常靠锁，允许串行化） |
| **可重入 ⟹ 线程安全** | 是；反之不一定 |

**不可重入的典型原因**：

```c
/* 坏：静态局部 → 所有调用者共享 */
void bad(void) {
    static int count;
    count++;
}

/* 坏：返回静态缓冲区指针 */
char *get_ip_str(void) {
    static char buf[16];
    sprintf(buf, "%d.%d.%d.%d", ...);
    return buf;
}
```

**中断 + 任务重入场景**：

```
任务调用 uart_send() ──► 执行到一半
        ▲
        │  UART 中断再次调用 uart_send()
        └─ 若共享静态状态 →  corruption
```

**编写可重入代码的原则**：

1. **不使用** 非 const 全局/静态变量（或进入前加锁）
2. **只操作** 参数与栈上局部变量
3. **不调用** 已知不可重入的库函数（如部分 `strtok`、`rand` 无锁版）
4. 标准 C 库：用 **`strtok_r`**、**`reentrant` 版 syscalls**（Newlib `_r` 后缀）

**RTOS 中的实践**：

| 场景 | 做法 |
|------|------|
| `printf` | 用 mutex 包裹或专用日志任务 + queue（**demo05_queue**） |
| 硬件寄存器 | 短 **临界区** 关中断或 mutex |
| 纯计算 | 保持无状态即可重入 |

**与 **10.2.4** 的关系**：

- 不可重入函数 + 共享资源 → 必须 **临界区保护** 或改为 **每任务一份上下文**
- **malloc** 在多任务下需 **重入锁**（Newlib `--enable-newlib-reent`）

**Linux 对比**（**10.10**）：

- 内核函数区分 `_irqsave` 版本；用户态 pthread 函数要求线程安全
- **`errno`**：线程局部存储 `__errno_location()`
