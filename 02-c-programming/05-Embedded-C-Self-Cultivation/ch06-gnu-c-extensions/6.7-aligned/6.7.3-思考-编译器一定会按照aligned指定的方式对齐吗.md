# 6.7.3 思考：编译器一定会按照aligned指定的方式对齐吗

## 本节讲什么

**`aligned(n)`** 表示 **至少** n 字节对齐，编译器可给出更大对齐；但不能低于 ABI 对类型的 **minimum alignment**。动态分配、栈对象、嵌套 `packed` 结构体中有 **例外与陷阱**。

## 要点

| 场景 | 行为 |
|------|------|
| 全局/静态 `aligned(64)` | 通常严格满足 |
| 栈上局部 | GCC 会插入 padding；极端大对齐可能失败 |
| `malloc` 返回 | **仅保证 max_align_t**；大对齐用 `aligned_alloc` |
| 结构体 `aligned(16)` | 整个 struct 按 16 对齐 |
| 成员 `aligned(16)` | 该成员 offset 约束；可能增大 sizeof |
| **`packed` 结构体内** | 成员 align 被压到 1，`aligned` 可能被 **忽略或降级** |

```c
struct trouble {
    char c;
    int x __attribute__((aligned(8)));
} __attribute__((packed));
/* x 可能无法在 offset 4 处 8 对齐 — 编译器警告或行为实现定义 */
```

```c
/* 动态：aligned_alloc 对齐必须是 2 的幂且 ≥ sizeof(void*) */
void *p = aligned_alloc(64, 256);
```

- **链接脚本 `ALIGN(64)`** 也可提升段对齐
- **False sense**：源码写了 `aligned` 仍要用 `_Static_assert` + 运行时 assert 校验 DMA 缓冲区
- 内核 **`__packed`** 与 **`__aligned(n)`** 同存时以文档/实测为准
