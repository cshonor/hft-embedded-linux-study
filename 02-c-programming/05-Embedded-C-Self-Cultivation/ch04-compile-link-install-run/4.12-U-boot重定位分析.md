# 4.12 U-boot重定位分析

## 本节讲什么

**U-Boot** 常先由 ROM 加载到 **低地址 SRAM/DRAM 临时区**，再 **自拷贝** 到链接脚本指定的高地址并 **修正重定位项**——因为初始 PC 与链接 VMA 不一致。这是 **4.4.3 重定位** 在嵌入式 Bootloader 中的手工版，与内核解压后重定位同类。

## 要点

### 为何重定位

- 链接时假设运行在 `TEXT_BASE`（如 `0x40200000`）
- 上电时可能在 `0x00000000` 或另一 RAM 区执行
- 必须拷贝代码/数据并修补 **相对/绝对地址引用**

### 典型步骤（概念）

1. `relocate_code`：拷贝 `.text/.data` 到目标区
2. 遍历 `__rel_dyn_start`…`__rel_dyn_end` 修补
3. 跳转至重定位后地址继续执行

```bash
# 分析 U-Boot ELF（交叉工具链）
arm-linux-gnueabihf-readelf -S u-boot
arm-linux-gnueabihf-nm u-boot | grep relocate
arm-linux-gnueabihf-objdump -d u-boot | grep relocate
```

### 与链接脚本

- `u-boot.lds` 定义 `TEXT_BASE`、`__image_copy_start`
- **4.14** `MEMORY` 区分 Flash 与 RAM

### 对照 Linux 启动

| 阶段 | 重定位者 |
|------|----------|
| U-Boot | 自身 `board_init_f` / `relocate` |
| Linux 内核 | `head.S` + 解压桩 |
| 用户程序 | `ld.so`（**4.8**） |
