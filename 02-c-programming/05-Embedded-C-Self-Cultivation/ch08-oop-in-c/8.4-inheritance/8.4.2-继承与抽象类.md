# 8.4.2 继承与抽象类

## 本节讲什么

**抽象类**定义协议但不完整实现，不能直接实例化；子类必须实现纯虚方法。内核中 **`struct device`** 常作为抽象基类——你不会 `kzalloc(sizeof(struct device))` 单独使用，而总是通过 **`platform_device_register`**、**`pci_register_device`** 等子类路径注册。框架定义生命周期，驱动填 `probe`/`remove`。

## 要点

**抽象 device 约束**：

```c
/* 错误：缺少 bus、parent、release 等 */
struct device *dev = kzalloc(sizeof(*dev), GFP_KERNEL);

/* 正确：通过子类 API */
struct platform_device *pdev = platform_device_alloc("myuart", -1);
platform_device_add(pdev);
```

| 抽象基类 | 子类 / 注册 API | 驱动实现 |
|----------|-----------------|----------|
| `struct device` | `platform_device` | `platform_driver.probe` |
| `struct net_device` | `alloc_netdev` | `netdev_ops` |
| `struct file` | `open` 时分配 | `file_operations` |

**纯虚 probe 模式**：

```c
struct platform_driver {
    int (*probe)(struct platform_device *);
    int (*remove)(struct platform_device *);
    struct device_driver driver;   /* 内嵌「基类」 */
};

/* 驱动作者只实现 probe/remove，bus 核心负责 match */
static struct platform_driver my_drv = {
    .probe = my_probe,
    .remove = my_remove,
    .driver = { .name = "myuart" },
};
```

- 抽象类 API **强制**正确初始化顺序（`device_initialize` → `device_add`）
- **`devm_*`** 把资源生命周期绑到 `device`，析构由框架在 `remove` 时触发
- 用户态 **demo02**：`Device` 只通过 `UartDev`/`SpiDev` 具体实例化
