# 8.1 代码复用与分层思想

## 本节讲什么

嵌入式与内核代码规模庞大，**重复逻辑**与**硬件耦合**是维护噩梦。本节建立**分层架构**与**代码复用**的第一性原理：上层只依赖抽象接口，下层提供具体实现；同一套框架可覆盖 UART、SPI、以太网等不同硬件。这是后续 **C 模拟 OOP**（struct + 函数指针 + ops 表）的动机——用 C 写出可扩展、可测试的驱动骨架。

## 要点

**三层驱动模型**（**demo06_layered**）：

```
┌─────────────────────────────────┐
│  app/     应用层：open/read/write │
├─────────────────────────────────┤
│  abstract/ 抽象层：dev_ops 接口   │
├─────────────────────────────────┤
│  hw/      硬件层：寄存器/MMIO     │
└─────────────────────────────────┘
```

| 层次 | 职责 | 复用方式 |
|------|------|----------|
| **抽象** | 定义 `DevOps`、生命周期 API | 所有驱动共享同一套头文件 |
| **硬件** | 寄存器读写、中断 | 每颗芯片一份 `.c` |
| **应用** | 业务逻辑、协议栈 | 通过 ops 调用，不直接碰寄存器 |

**弱符号默认实现**（衔接 **ch06 6.9**）：

```c
/* abstract/dev_default.c */
void __attribute__((weak)) hw_uart_init(void) { /* stub */ }
void __attribute__((weak)) hw_uart_tx(char c) { (void)c; }

/* hw/uart_pl011.c — 强符号覆盖 */
void hw_uart_init(void) { /* 写 PL011 寄存器 */ }
```

- 未链接 `hw/` 时仍可编译运行 stub；链接真实 BSP 后自动选用强定义
- 避免 `#ifdef BOARD_XXX` 散落各处

**复用手段对照**：

| 手段 | C 实现 | 典型场景 |
|------|--------|----------|
| **函数指针 / ops** | `struct file_operations` | 运行时多态 |
| **宏 / inline** | `readl`/`writel` | 零开销抽象 |
| **不透明类型** | `typedef struct dev dev_t` | 隐藏内部字段 |
| **container_of** | 从 `list_head` 得宿主 struct | 内核链表遍历 |

- **单一职责**：每层只暴露必要 API；应用层不应 `#include` 硬件寄存器头
- **依赖方向**：app → abstract ← hw，hw 不反向依赖 app
- **测试友好**：mock 层提供 weak/stub 实现，单测无需真实硬件
