# 9.5 模块设计原则

## 本节讲什么

在 **9.2 划分** 与 **9.3 封装** 之后，需要可执行的**设计原则**指导日常决策：何时拆模块、如何控制依赖、怎样处理错误与日志。本节汇总 **高内聚、低耦合、单一职责**，并给出嵌入式项目常见的 ** pitfalls 清单** 与标准组件约定（**demo05**）。

## 要点

**SOLID 的 C 化解读**：

| 原则 | C 落地 |
|------|--------|
| **S 单一职责** | 一个模块目录 = 一类资源（UART、日志、环形缓冲） |
| **O 开闭** | 新 BSP 加 `platform/` 文件 + weak/ops，不改 app |
| **L 里氏替换** | 所有 `uart_ops` 实现满足同一契约（**ch08**） |
| **I 接口隔离** | 头文件只暴露必要 API；`internal.h` 隔离 |
| **D 依赖倒置** | app 依赖 `uart_handle_t` 抽象，不依赖 PL011 寄存器 |

**解耦手段对照**（全章主线）：

| 手段 | 机制 | 章节/Demo |
|------|------|-----------|
| **接口抽象** | ops / vtable | **ch08**、**demo01** |
| **weak 符号** | 链接期替换默认实现 | **ch06 6.9**、**demo04_weak** |
| **回调** | 函数指针注册 | **9.7.2**、**demo06_callback** |
| **统一错误码** | `err_t` + `err_str()` | **demo05** |
| **不透明类型** | 隐藏 struct | **9.3**、**8.3.1** |

**标准横切组件**（每个项目应有）：

```c
/* common/log.h — demo05 */
LOG_INFO("uart open baud=%u", baud);
LOG_ERR("write failed: %s", err_str(e));

/* common/err.h */
typedef enum { ERR_OK=0, ERR_INVAL=-1, ERR_NOMEM=-2, ... } err_t;
```

| 组件 | 职责 |
|------|------|
| `log.h` | 分级日志，编译期 `LOG_ENABLE` |
| `err.h` / `err.c` | 全项目错误码与字符串 |
| `config.h` | 功能开关、板型、缓冲区大小 |
| `.gitignore` | 忽略 `build/`、`*.o`、`out/` |

**常见 pitfalls**：

| 问题 | 症状 | 对策 |
|------|------|------|
| **循环依赖** | `a.h` ↔ `b.h` | 前向声明、抽第三模块（**9.4.5**） |
| **全局变量泛滥** | 难测、竞态 | `static` + getter/API（**9.7.1**） |
| **头文件膨胀** | 改一行全编 | 减 include、不透明类型（**9.4**） |
| **增量构建失效** | 链接总是全量 | 子 Makefile 产出 `.a`（**9.1**） |
| **misc 模块** | 死代码堆 | 及时拆分或删除 |
| **#ifdef 地狱** | 可读性崩溃 | weak / CMake option / Kconfig 式开关 |

**模块评审四问**：

1. 能否用一句话描述模块职责？
2. 去掉它，哪几个模块会坏？
3. 头文件 include 树是否单向无环？
4. 能否在 PC 上 stub 测试，无需硬件？
