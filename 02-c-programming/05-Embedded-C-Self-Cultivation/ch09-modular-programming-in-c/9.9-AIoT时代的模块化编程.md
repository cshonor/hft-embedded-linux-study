# 9.9 AIoT时代的模块化编程

## 本节讲什么

AIoT 设备 = **传感 + 连接 + 云 + OTA + 安全**，软件栈比传统 MCU 项目更厚，模块化不是可选项而是交付底线。本节从云边端协同视角，说明如何在本章所学之上组织 **SDK 接入、增量 OTA、安全边界与 CI**，并为 **ch10 终章综合项目** 奠定工程骨架。

## 要点

**AIoT 典型模块地图**：

```
┌─────────────────────────────────────────┐
│  app/        业务：采集策略、边缘推理      │
├─────────────────────────────────────────┤
│  cloud/      云 SDK：MQTT/CoAP/LwM2M     │
├─────────────────────────────────────────┤
│  middleware/ OTA、TLS、JSON、DB          │
├─────────────────────────────────────────┤
│  driver/     传感器、模组、以太网/WiFi    │
├─────────────────────────────────────────┤
│  platform/   RTOS、BSP、secure element   │
├─────────────────────────────────────────┤
│  utils/      log、err、ringbuf、crc      │
└─────────────────────────────────────────┘
```

| 模块 | 模块化要点 |
|------|------------|
| **云 SDK** | vendor `.a` 当黑盒；adapter 层转 `err_t` 与 `LOG_*` |
| **OTA** | 独立 `middleware/ota`；经 driver flash 抽象写分区 |
| **TLS** | 不 spread `mbedtls` 头；wrapper 模块 |
| **AI 推理** | `middleware/inference`；tensor 内存经 platform alloc |
| **配网** | BLE/WiFi provisioning 独立 app 或子模块 |

**解耦在 AIoT 中的落地**（全章回顾）：

| 手段 | AIoT 场景 |
|------|-----------|
| ops / 回调 | 模组 `at_ops`、传感器 `sample_cb` |
| weak platform | 同一固件树支持 WiFi/BLE/4G 模组变体 |
| 统一 `err_t` | 云断线、传感器失败、OTA 校验同一语义 |
| 异步队列 | 模组 URC 中断 → 解析任务 → app |
| `.a` + install | 芯片厂交付 BSP 包，app 团队独立开发 |

**安全与模块化**：

- **密钥不进 app 源码**：secure element / NVS 模块
- **最小权限**：cloud 模块不 link driver 寄存器层
- **OTA 签名验证**在 `ota` 模块完成，app 只调 `ota_apply()`

**工程化清单**：

| 项 | 实践 |
|----|------|
| CI | PC 上 `make TARGET=host test` + 交叉编译 job |
| 配置 | `config.h.in` + 产品矩阵 CMake preset |
| 日志 | 分级 + 模块 tag（**demo05**） |
| 版本 | 固件 semver + 各 `lib*.a` API version |
| `.gitignore` | 忽略密钥、build、SDK 解压目录 |

**与 ch10 终章项目**：

- **ch09** 提供：目录树、Makefile/CMake、common 组件、解耦模式
- **ch10** 叠加：任务/锁/中断、完整设备场景（如多传感器 + 上报 + OTA）
- 建议终章直接复用 **demo02_make** 分层 + **demo05** log/err + **demo06** 回调

**演进路径**：

```
Demo 级 (demo01) → 单产品 → 产品族 (config/weak) → 平台化 (install SDK)
```
