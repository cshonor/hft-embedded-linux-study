# 9.6 被误解的关键字：goto

## 本节讲什么

`goto` 在教科书与编码规范中声名狼藉，但 **Linux 内核**与大量成熟 C 代码仍大量使用——主要用于**单函数内的集中错误清理**，替代深层嵌套的 `if/else`。本节澄清何时该用、何时该禁，以及与模块化中**资源配对释放**模式的关系。

## 要点

**内核风格：err 标签集中清理**

```c
int mod_uart_open(uart_handle_t **out, const char *port)
{
    uart_handle_t *h = NULL;
    void *dma_buf = NULL;
    int fd = -1;
    int ret;

    h = calloc(1, sizeof(*h));
    if (!h)
        return ERR_NOMEM;

    fd = platform_uart_open(port);
    if (fd < 0) {
        ret = ERR_IO;
        goto err;
    }

    dma_buf = platform_dma_alloc(BUF_SIZE);
    if (!dma_buf) {
        ret = ERR_NOMEM;
        goto err;
    }

    h->fd = fd;
    h->buf = dma_buf;
    *out = h;
    return ERR_OK;

err:
    if (fd >= 0)
        platform_uart_close(fd);
    free(dma_buf);
    free(h);
    return ret;
}
```

| 对比 | 嵌套 if | goto err |
|------|---------|----------|
| 可读性 | 5 层缩进 | 线性主路径 + 底部清理 |
| 漏释放风险 | 每层都要写 | 统一 err 块 |
| 适用 | 1–2 个资源 | 多资源、多失败点 |

**适用场景（推荐）**：

- 同一函数内**Acquire 多个资源**，任一步失败需**逆序释放**
- 驱动 `probe`、文件 `open`、协议握手等多步骤初始化
- **仅向前跳转**到函数末尾的 err/cleanup 标签

**禁用场景**：

| 场景 | 原因 |
|------|------|
| 跨函数跳转 | 不可能；也勿用 setjmp 代替设计 |
| 绕开模块边界 | 破坏结构化 |
| 实现循环 | 用 `for/while` |
| 掩盖逻辑混乱 | goto 不能救烂设计 |

**与模块化的关系**：

- **模块 API 边界**仍应返回 `err_t`，不在 API 层暴露 goto
- goto 是 **`.c` 内部实现细节**，不出现在 `.h`
- 配对 `create/destroy`（**9.3**）+ err 路径 `free`，避免泄漏

**替代方案**：

```c
/* 小规模：do { ... } while(0) + break */
do {
    if (!alloc_a()) break;
    if (!alloc_b()) break;
    return OK;
} while (0);
cleanup();
```

- C++ RAII 在 C 中无直接对应；`goto err` 是惯用解法
- 宏包装：`DEFINE_GUARD` 等（复杂项目），小项目 goto 足够

**编码规范参考**：

- Linux kernel **Documentation/process/coding-style.rst**：`goto` 用于 error path OK
- MISRA C：限制 goto（嵌入式安全关键系统需评估）
