# 9.4.9 头文件中的内联函数

## 本节讲什么

`inline` / `static inline` 函数放在头文件是 C 项目的常见做法——在调用点展开，消除函数调用开销，同时保持 API 在头文件中可见。本节说明 inline 的链接规则、与宏的取舍、以及过度 inline 导致代码体积膨胀的问题。

## 要点

**三种 inline 写法**：

```c
/* 1. static inline — 最常用，每翻译单元独立副本，无链接冲突 */
static inline uint32_t readl(const volatile void *addr)
{
    return *(const volatile uint32_t *)addr;
}

/* 2. inline + 一处 extern 定义（C99）— 较少用 */
inline int fast_min(int a, int b) { return a < b ? a : b; }
/* 某个 .c */ extern inline int fast_min(int a, int b);

/* 3. 宏 — 无类型检查，内核 readl/writel 早期风格 */
#define MIN(a, b) ((a) < (b) ? (a) : (b))
```

| 方式 | 优点 | 缺点 |
|------|------|------|
| `static inline` | 简单、无 ODR 问题 | 多份副本可能增大 `.text` |
| 宏 | 最大灵活性 | 无类型安全、难调试 |
| 普通函数在 `.c` | 单份代码 | 有调用开销 |

**适用场景**：

- **位操作、小包装**：`min/max`、 endian 转换、寄存器读写
- **热路径**：DPDK `rte_*` 大量 `static inline`
- **不适合**：大函数、含循环/调用的复杂逻辑 → 放 `.c`

**与模块化**：

```c
/* log.h — demo05 风格：宏 + 可选 inline */
#define LOG_INFO(fmt, ...) \
    do { if (LOG_ENABLE) printf("[INFO] " fmt "\n", ##__VA_ARGS__); } while (0)
```

- 日志宏放头文件 OK；避免在头文件 inline 复杂格式化
- 模块 **公开 API** 的大函数应在 `.c` 实现，保持 ABI 稳定

**体积权衡**（嵌入式 Flash 受限）：

- 10 个文件 include 同一 `static inline`，链接器可能保留 10 份
- GCC `-flto` 可跨单元去重；或改普通函数 + LTO
- 测量：对比 `.map` 文件中符号大小

**内核惯例**：

- `include/linux/` 中大量 `static inline` helper（`list_entry`、`container_of` 宏）
- 架构相关 hot path 在 `arch/xxx/include/asm/` inline
