# 9.4.6 定义与声明的一致性

## 本节讲什么

C 的链接器**只匹配符号名**，不校验函数签名。声明与定义不一致时，编译可能通过、链接也可能通过，但运行时会栈错乱或数据损坏。本节强调 **ODR 在 C 中的弱保障**及工程化防护：单一头文件来源、`-Wstrict-prototypes`、静态分析。

## 要点

**必须完全一致的项目**：

| 项 | 声明 | 定义 |
|----|------|------|
| 返回类型 | `err_t` | 不能是 `int` |
| 参数类型 | `const void *` | 不能是 `void *`（严格别名下） |
| 参数个数 | 3 | 不能 2 |
| 调用约定 | 默认 | 必须相同 |
| `const`/volatile | 保留 | 不能省略 |

```c
/* mod_uart.h */
int uart_write(uart_handle_t *h, const void *buf, size_t len);

/* mod_uart.c — 必须匹配 */
int uart_write(uart_handle_t *h, const void *buf, size_t len)
{
    /* ... */
}
```

**变量声明一致性**：

```c
/* header */
extern int g_count;

/* wrong.c */
long g_count = 0;   /* ❌ 类型不一致 → 未定义行为 */
```

**头文件与实现分离的最佳实践**：

1. **API 只写一次**——在 `.h`
2. `.c` 第一行 `#include "mod_uart.h"`，让编译器校验定义与声明
3. 不在 `.c` 手写第二份原型

**编译器辅助**：

```makefile
CFLAGS += -Wall -Wextra -Wstrict-prototypes -Werror
```

| 警告 | 捕获 |
|------|------|
| `-Wstrict-prototypes` | `foo()` 空参数列表（非原型） |
| `-Wmissing-prototypes` | 全局函数无头文件声明 |
| `-Werror` | 警告即失败 |

**跨模块一致性**：

- 改 API 时**先改 `.h`**，再改所有 `.c` 调用点
- 版本化：Breaking change 递增 major，文档标注 deprecated
- **demo05** `err_t` 枚举：全项目统一错误码，避免各模块自定义 `-1/-2`
