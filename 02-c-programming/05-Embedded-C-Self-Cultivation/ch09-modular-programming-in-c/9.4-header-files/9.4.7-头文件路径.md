# 9.4.7 头文件路径

## 本节讲什么

`#include` 用双引号还是尖括号、`-I` 怎么设，决定了模块能否**干净地隔离公开头与内部头**。本节给出用户态工程与交叉编译场景下的路径规范，避免 `-I.` 导致的命名冲突与意外 include 内部实现。

## 要点

**两种 include 语法**：

```c
#include "mod_uart.h"    /* 1. 当前 .c 目录  2. -I 路径 */
#include <mod_uart.h>    /* 仅 -I 路径（及系统目录） */
```

| 写法 | 语义 | 推荐场景 |
|------|------|----------|
| `"..."` | 项目/模块本地头 | 同目录 `internal.h` |
| `<...>` | 安装后的公开 API | `out/include` 已 install 的头 |

**推荐 `-I` 布局**（**9.2.3**）：

```makefile
# 顶层
CFLAGS += -Iinclude -Idriver/mod_uart -Iutils -Icommon

# 或 install 后统一
CFLAGS += -Iout/include
LDFLAGS += -Lout/lib
```

**模块内部 vs 对外**：

```
driver/mod_uart/mod_uart.c:
    #include "mod_uart_internal.h"   /* "" 仅本模块 */
    #include <mod_uart.h>            /* 公开 API，也可 "" */

app/main.c:
    #include <mod_uart.h>            /* 只走 include 路径 */
    /* 禁止 #include "mod_uart_internal.h" */
```

**反模式**：

| 做法 | 问题 |
|------|------|
| `CFLAGS += -I.` | 任意 `.c` 可 include 任意内部头 |
| 递归 `-I$(shell find . -name '*.h')` | 同名头冲突、`stdint.h` 被 shadow |
| app 用相对路径 `#include "../driver/secret.h"` | 路径耦合，重构即碎 |

**交叉编译**：

```cmake
target_include_directories(mod_uart PUBLIC
    $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}>
    $<INSTALL_INTERFACE:include>
)
```

- `BUILD_INTERFACE`：构建树内路径
- `INSTALL_INTERFACE`：安装后 `#include <mod_uart.h>`
- 与 **demo03_cmake** 的 `target_link_libraries` 传播一致

**与 Linux 内核**（**9.4.8**）：

- 内核用 `#include <linux/xxx.h>`，路径由 kbuild `-I` 固定
- 驱动 `.c` 不应 `#include` 其他驱动私有头
