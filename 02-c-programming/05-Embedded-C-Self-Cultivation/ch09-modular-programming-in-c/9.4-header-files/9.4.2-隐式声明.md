# 9.4.2 隐式声明

## 本节讲什么

C89 允许在**未见到函数原型**时调用函数，编译器会**隐式假设**返回 `int`、参数类型可隐式转换——这在现代 C（C99 起）中是**硬错误**，且即使能编译也极易产生未定义行为。本节说明隐式声明的成因、危害与 `-Wall`/`-Werror` 防护，强调**每个对外函数必须在头文件声明**。

## 要点

**隐式声明规则（历史遗留）**：

```c
/* 无 #include，无原型 */
int main(void) {
    foo(3.14);  /* C89：编译器假设 int foo() */
    return 0;
}
```

编译器假设：

- 返回类型 `int`
- 参数类型未知，调用处做默认提升
- **不会**做 arity 检查（参数个数可错）

**现代编译器行为**：

| 标准 | 隐式声明 |
|------|----------|
| C89 | 允许，警告 |
| C99+ | **错误**：`call to undeclared function` |
| GCC `-std=gnu11 -Wall` | 默认拒绝 |

**典型事故**：

```c
/* 头文件声明 */
int uart_write(uart_handle_t *h, const void *buf, size_t len);

/* 某 .c 手写错原型 */
int uart_write(uart_handle_t *h, const char *buf, int len); /* 错！ */

/* 链接可能通过（名字相同），运行时栈/layout 错乱 */
```

| 症状 | 根因 |
|------|------|
| 返回值乱码 | 实际返回 `void*` 却假设 `int` |
| 栈损坏 | 参数类型/个数不匹配 |
| 链接通过行为错 | 原型不一致但 mangling 相同（C 无 overload） |

**防护措施**：

1. **所有 API 进 `.h`**，调用方只 `#include` 模块头
2. 编译选项：`-std=gnu11 -Wall -Wextra -Werror=implicit-function-declaration`
3. 禁止在 `.c` 里复制粘贴函数原型——单一来源是 `.h`
4. CI 对警告零容忍

**与模块化关系**：

- 模块边界 = 头文件边界；绕过 header 直接「知道」函数存在 = 破坏封装
- `static` 函数无需头文件声明，但只能在定义之后或同文件前置声明
