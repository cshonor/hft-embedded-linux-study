# 9.4 头文件深度剖析

## 本节讲什么

头文件是 C 模块化编程的**枢纽**：编译器靠它做类型检查，链接器靠 `.c` 里的定义合并符号，增量构建速度也几乎完全由头文件依赖图决定。本节作为 **9.4.1–9.4.9** 的总览，建立「头文件写什么、不写什么、如何组织」的全局地图，并对照 Linux 内核头文件体系（**9.4.8**）。

## 要点

**头文件的三种角色**：

| 角色 | 内容 | 示例 |
|------|------|------|
| **API 契约** | 函数声明、公开类型 | `mod_uart.h` |
| **配置/常量** | 宏、`enum`、内联小函数 | `config.h`、`log.h` |
| **内部共享** | 模块内多 `.c` 共用 | `mod_uart_internal.h` |

**黄金法则**：

1. **头文件只放声明，不放定义**（`inline`/`static inline` 例外，**9.4.9**）
2. **能前向声明就不 include**（**9.4.5**）
3. **include 顺序**：本模块头 → 系统头 → 第三方
4. **依赖最小化**：头文件不应拖入半个 stdlib

**典型问题链**：

```
头文件放定义 → 多 .c include → multiple definition
隐式声明     → C99 报错 / 原型不匹配 → 未定义行为
头文件膨胀   → 改一行重编全工程 → 迭代变慢
循环 include → 编译错误或诡异缺类型
```

**模块头文件 vs 聚合头**：

```
include/mod_uart.h     # 单模块 API
include/project_api.h  # 可选：给外部 SDK 的一次性入口
```

- 聚合头方便用户，但会拖慢编译；内核倾向按需 include
- app 只 include 直接依赖的模块头，不 include 「万能 headers.h」

**与 Linux 内核的对照**（**9.4.8** 展开）：

| 用户态模块 | 内核 |
|------------|------|
| `mod_uart.h` | `include/linux/serial_core.h` |
| `mod_uart_internal.h` | `drivers/tty/serial/imx.c` 内 static |
| `include/` install | `headers_install` |
| Kconfig 开关 | `config.h` / CMake option |

**自检清单**（发布模块前）：

- [ ] 每个 `.h` 有 include guard
- [ ] 无全局变量/非 inline 函数定义
- [ ] 公开 struct 是否必须全暴露？能否不透明？
- [ ] `#include` 能否改为前向声明？
- [ ] 路径用 `<>` 还是 `""` 符合 **9.4.7** 约定？
