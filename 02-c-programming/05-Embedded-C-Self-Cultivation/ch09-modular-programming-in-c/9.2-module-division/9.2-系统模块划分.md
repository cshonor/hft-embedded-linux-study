# 9.2 系统模块划分

## 本节讲什么

模块化编程的第一步不是写 Makefile，而是**回答「系统由哪些职责块组成、块与块如何依赖」**。本节给出嵌入式/AIoT 项目常见的**分层模块模型**，建立 **高内聚、低耦合、单一职责** 三条总纲，为 **9.2.1–9.2.3** 的具体划分方法与目录规划定调。

## 要点

**经典五层架构**（可裁剪为三层或扩展为六层）：

```
┌──────────────────────────────────────┐
│  app/          应用：业务逻辑、状态机   │
├──────────────────────────────────────┤
│  middleware/   中间件：协议栈、OTA、DB  │
├──────────────────────────────────────┤
│  driver/       驱动：UART/SPI/传感器   │
├──────────────────────────────────────┤
│  platform/     平台：时钟、GPIO、BSP   │
├──────────────────────────────────────┤
│  utils/        工具：日志、环形缓冲、CRC │
└──────────────────────────────────────┘
```

| 层次 | 职责 | 不应做的事 |
|------|------|------------|
| **app** | 业务流程、配置解析 | 直接读写寄存器 |
| **middleware** | 可复用服务（MQTT、Modbus） | 绑定具体芯片型号 |
| **driver** | 设备抽象 + 硬件操作 | 包含 UI 或网络协议细节 |
| **platform** | 板级支持、启动、中断向量 | 业务逻辑 |
| **utils** | 无状态纯函数、通用容器 | 依赖 driver 头文件 |

**三大设计原则**：

| 原则 | 含义 | 违反时的症状 |
|------|------|--------------|
| **高内聚** | 模块内元素围绕单一主题 | 改 UART 要动 5 个目录 |
| **低耦合** | 模块间通过窄接口交互 | 头文件互相 `#include` 成环 |
| **单一职责** | 一个模块只有一个变更理由 | `misc.c` 3000 行万能文件 |

**依赖方向**（与 **ch08 8.1** 一致）：

```
app → middleware → driver → platform
         ↓            ↓
       utils ←────────┘
```

- **单向依赖**：下层不知道上层存在；`driver` 不 `#include "app_config.h"`
- **接口抽象**：上层依赖 `uart_ops`（**ch08 vtable**），不依赖 `uart_pl011.c`
- **可替换性**：换 BSP 只换 `platform/`，app 零改动（**demo04_weak**）

**模块粒度经验**：

- 一个 `.c` 对应一个**对外头文件**（`mod_uart.h` ↔ `mod_uart.c`）
- 单文件超过 ~500 行考虑拆分；但避免「一个函数一个文件」的过度碎片化
- 团队边界：每人负责 1–3 个模块目录，减少 merge 冲突
