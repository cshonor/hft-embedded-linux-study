# 9.2.1 模块划分方法

## 本节讲什么

「按什么切」没有唯一答案，但有三种在嵌入式项目中反复验证的方法：**按架构层次**、**按功能特性**、**按数据流**。本节给出每种方法的适用场景、划分步骤与反例，帮助你在立项或重构时快速定稿模块清单。

## 要点

**方法一：按架构层次划分**（最常用，**9.2** 五层模型）

1. 列出系统对外能力（采集、上报、OTA、本地存储）
2. 将能力映射到 app / middleware
3. 列出所需硬件（UART、Flash、RTC）→ driver / platform
4. 提取跨层复用逻辑 → utils

示例（温湿度采集器）：

| 模块 | 目录 | 职责 |
|------|------|------|
| 采集调度 | `app/sensor_app.c` | 定时读传感器、上报 |
| MQTT 客户端 | `middleware/mqtt_client.c` | 连接、发布 |
| SHT30 驱动 | `driver/sht30.c` | I2C 读写 |
| I2C 平台 | `platform/i2c_stm32.c` | 寄存器级 I2C |
| 日志 | `utils/log.c` | **demo05** |

**方法二：按功能特性（Feature）划分**

适合**产品线共用内核、差异化功能**的场景：

```
features/
├── ota/          # 可选编译
├── ble_mesh/
└── lora_wan/
core/
├── scheduler/
└── event_bus/
```

- Makefile / CMake 用 `CONFIG_FEATURE_OTA=y` 控制编译
- 优点：功能整块启停；缺点：需警惕 feature 间隐式耦合

**方法三：按数据流划分**

适合**管道式**系统（采集 → 处理 → 输出）：

```
sensor → filter → aggregator → uplink
   │        │          │           │
  src/    proc/      proc/       net/
```

- 每个阶段输入/输出 struct 明确（如 `sample_t`）
- 阶段间只传数据，不共享 mutable 全局状态（**9.7.1**）
- 便于单元测试：mock 上游、断言下游

**划分决策 checklist**：

| 问题 | 是 → 动作 |
|------|-----------|
| 两处以上会用到？ | 抽到 utils 或 middleware |
| 换芯片要改？ | 放 driver/platform |
| 换业务规则要改？ | 留 app |
| 需要运行时替换实现？ | 抽象 ops 接口（**ch08**） |
| 编译时间因改一行而全量重编？ | 拆小模块、减头文件依赖（**9.4**） |

**反例**：

- `common.c`：杂项垃圾桶，最终无人敢动
- `hal.c` 2000 行：所有外设混在一起
- `types.h` 包含全部 struct：任何改动触发全量编译
