# 9.2.2 面向对象编程的思维陷阱

## 本节讲什么

**ch08** 教你在 C 里模拟 OOP，但模块化划分时若照搬 C++/Java 思维，容易**过度抽象、层次过深、接口膨胀**。本节列出嵌入式 C 项目中最常见的 OOP 陷阱，并给出「够用就好」的 C 风格模块划分准则——继承与多态是工具，不是目的。

## 要点

**陷阱一：万物皆对象，强制继承树**

```
❌ Animal → Mammal → Dog → Husky → MyProjectDog
✅ uart_dev_t + uart_ops（一层抽象即可）
```

- C 没有真正的继承；首成员嵌入（**8.4.1**）有成本：类型转换、布局耦合
- 驱动层通常 **`struct + ops 表`** 足够，不必模拟完整类层次

**陷阱二：接口过多，ops 表爆炸**

| 症状 | 后果 |
|------|------|
| 10 个 ops 结构体 | 新人无法理清调用链 |
| 每个函数都经函数指针 | 调试困难、代码体积增大 |
| 纯虚函数式强制实现 | 简单 GPIO 也要写 5 个 stub |

- **准则**：仅对**多实现、需运行时切换**的边界用 ops（UART 驱动、存储后端）
- 纯工具函数直接 `utils_crc32()`，不必 `struct crc_ops`

**陷阱三：把「模块」等同于「类」**

| C++ 思维 | C 模块化思维 |
|----------|--------------|
| 一个 class 一个文件 | 一个**职责域**一个模块，可含多个相关 struct |
| private 成员 | `static` 函数 + 不透明类型 |
| 构造函数链 | `module_init()` + 明确错误码 |
| 友元 | 同目录 `internal.h`（仅本模块 `.c` include） |

**陷阱四：过早抽象平台层**

- 只部署一种 MCU 时，完整 `platform_ops` + weak 覆盖（**demo04_weak**）可能过度
- **YAGNI**：第一版可 `#ifdef BOARD_X`，第二次换板再抽 platform
- 但头文件边界、`.h`/`.c` 分离从第一天就要做（**9.3**）

**陷阱五：用多态掩盖模块边界混乱**

```c
/* 坏：app 直接持有具体 driver 内部 struct */
void app_run(struct pl011_regs *regs);

/* 好：app 只认抽象句柄 */
void app_run(uart_handle_t *uart);
```

- 多态是为**解耦**，不是为绕过模块化纪律
- 若 app 仍 `#include "pl011_regs.h"`，ops 表形同虚设

**实用平衡表**：

| 场景 | 推荐 |
|------|------|
| 单一硬件、小项目 | 扁平模块 + 直接函数调用 |
| 多 BSP、可测 | ops + weak / 链接时替换 |
| 多协议栈 | middleware 层 + 回调（**9.7.2**） |
| 内核/DPDK 级 | 完整 ops 模型（**ch08 8.5**） |
