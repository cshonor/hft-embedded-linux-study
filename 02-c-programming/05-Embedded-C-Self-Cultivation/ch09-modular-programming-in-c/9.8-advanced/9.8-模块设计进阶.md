# 9.8 模块设计进阶

## 本节讲什么

掌握 **9.1–9.7** 的基础模块化后，真实产品还面临**多平台移植**、**第三方框架集成**、**长期演进**。本节作为 **9.8.1 跨平台** 与 **9.8.2 框架** 的导读，建立进阶主题地图：何时引入抽象层、如何对接 RTOS/中间件而不破坏模块边界。

## 要点

**进阶主题矩阵**：

| 主题 | 核心问题 | 小节 |
|------|----------|------|
| **跨平台** | 同一 app 如何跑 STM32 / Linux / PC 仿真 | **9.8.1** |
| **框架** | 模块如何接入已有 OS/协议栈骨架 | **9.8.2** |
| **插件化** | 可选 `.a`、动态加载（Linux） | weak + CMake option |
| **版本与 ABI** | 模块升级不破坏调用方 |  opaque API、semver |
| **文档与 install** | 他人如何用你的 `lib*.a` | **9.2.3** install 目标 |

**进阶不等于更复杂**：

```
基础版：app → driver → 寄存器          （9.1–9.3 足够）
进阶版：app → framework → HAL → BSP   （多平台时才有价值）
```

- 评估标准：**第二块板子/第二个 OS** 是否 2 周内能 port
- 若否，检查 platform 层与 weak/ops 是否到位（**demo04_weak**）

**与 AIoT 生态**（**9.9**）：

- 云 SDK、蜂窝模组 AT 层、TLS 库——均以**独立 middleware 模块**接入
- 不把 MQTT 代码散进 driver

**质量门禁**：

| 检查 | 工具 |
|------|------|
| 模块间循环依赖 | include-what-you-use、自定义脚本 |
| 符号泄露 | `nm -g lib*.a` 应只有公开 API |
| 单测 | mock platform + PC gcc 构建 |
| 静态分析 | cppcheck、clang-tidy |
