# 9.8.2 框架

## 本节讲什么

**框架**提供应用运行的「骨架」——任务调度、设备注册、配置系统——你的模块作为**插件**接入，而非重写 main。本节对比自研轻量框架与 Zephyr/FreeRTOS/Linux 驱动框架的模块化接入方式，强调 **注册回调 / ops / 设备树** 等扩展点，与 **ch08 device/bus** 模型呼应。

## 要点

**框架 vs 库**：

| | 库 (lib) | 框架 |
|---|----------|------|
| 控制流 | 你调它 | 它调你（回调/ops） |
| 示例 | `ringbuf_push()` | `device_register()` |
| 模块关系 | 平级 | 核心 + 插件 |

**自研轻量框架骨架**：

```c
/* framework.h */
typedef struct {
    const char *name;
    err_t (*init)(void);
    err_t (*start)(void);
    void  (*stop)(void);
} app_module_t;

#define APP_MODULE_REGISTER(mod) \
    static const app_module_t * const _mod_##mod __attribute__((used)) = &mod

err_t framework_run_all(void);  /* 遍历链接段或显式表 */
```

- 各业务模块提供 `app_module_t` 实例，main 只调 `framework_run_all()`
- 类似 Linux `module_init` / constructor 段（高级）

**RTOS 框架接入**：

```
FreeRTOS: 每个模块一个 Task，入口 module_task(void *)
Zephyr:   DEVICE_DEFINE / SYS_INIT 宏注册
```

| 原则 | 做法 |
|------|------|
| 模块不 include 内核头到 app | 经 `osal.h` 包装 |
| 栈/优先级配置 | `config.h` 或 Kconfig |
| 同步 | OSAL mutex/semaphore（**ch10**） |

**Linux 驱动框架**（**ch08 8.3.3–8.3.4**）：

```
platform_driver_register → probe → uart_ops 绑定
```

- 驱动是框架插件；**file_operations** 是 ops 接口
- 用户态模块化可模仿：`bus_register` + `match` + `probe`

**中间件框架（MQTT/HTTP）**：

- 协议栈当 **middleware 模块**；网络 I/O 经 **platform socket 抽象**
- TLS 作为可选 `libmbedtls.a`，CMake `option(WITH_TLS)`

**避免框架过度设计**：

| 信号 | 动作 |
|------|------|
| 框架代码 > 业务代码 | 简化为 module 表 + 顺序 init |
| 10 层回调才到硬件 |  flatten |
| 宏魔法不可调试 | 显式注册数组 |

**install 与第三方集成**：

```
out/
├── lib/libmod_uart.a
├── include/mod_uart.h
└── pkg-config/mod_uart.pc   /* 可选 */
```

- 框架消费方只需 `-lmod_uart -Iout/include`
- 版本号写在 `mod_uart.h`：`#define MOD_UART_API_VERSION 1`
