# 9.7.1 全局变量

## 本节讲什么

全局变量是最省事的「共享内存」通信，也是模块化的大敌：**隐式依赖、难测试、竞态、初始化顺序不确定**。本节说明何时可容忍、如何收敛为模块私有 `static`，以及用 getter/API 替代裸 `extern` 的 refactor 路径。

## 要点

**问题清单**：

| 问题 | 说明 |
|------|------|
| **隐式耦合** | 读 `g_foo` 的模块与写者无 compile-time 契约 |
| **命名污染** | `g_`/`s_` 前缀仍可能冲突 |
| **线程安全** | 多任务读写无锁 → 数据竞争（**ch10**） |
| **测试** | 单测需手动 reset 全局状态 |
| **初始化顺序** | C 静态初始化顺序跨 `.c` 未定义（复杂依赖时） |

**可接受的窄例外**：

```c
/* config.c — 启动后只读 */
static const board_config_t s_board_cfg = { .uart_baud = 115200 };

const board_config_t *board_config(void)
{
    return &s_board_cfg;
}
```

- **只读配置**：编译期或 `main()` 早期初始化后不变
- **单模块私有**：`static int s_refcount` 在 `mod_uart.c`
- **调试开关**：`volatile int g_debug` 仅开发版，量产关闭

**从全局到模块 API 的重构**：

```c
/* 之前 */
extern int g_packet_count;
void somewhere(void) { g_packet_count++; }

/* 之后 — stats 模块 */
/* stats.h */
int stats_packet_inc(void);
int stats_packet_get(void);

/* stats.c */
static int s_packet_count;
int stats_packet_inc(void) { return ++s_packet_count; }
```

**若必须跨模块共享 mutable 状态**：

1. **集中到一个 `.c`**，不暴露变量，只暴露函数
2. 文档说明锁策略；锁在模块内（**ch10**）
3. 禁止在头文件 `extern` 可变全局——头文件只声明 API

**与头文件规则**（**9.4.3**）：

```c
/* ❌ header */
int g_mode = 0;

/* ✅ */
/* mode.c */ static int s_mode;
/* mode.h */ int mode_get(void); void mode_set(int m);
```
