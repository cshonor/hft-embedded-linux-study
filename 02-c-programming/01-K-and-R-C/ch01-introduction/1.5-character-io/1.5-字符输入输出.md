# 1.5 字符输入输出

## 本节讲什么

承接 [1.4 符号常量](../1.4-符号常量.md)，基于 `<stdio.h>` 的 **`getchar()` / `putchar()`**（头文件总览：[附录 B.1](../../appendix-b-standard-library/B.1-输入与输出stdio.h.md)），**不依赖指针、不用整行读函数**，逐个字符扫描输入流，手工实现「复制 → 计字符 → 计行 → 计单词」四个经典 Unix 文本小工具。

这是 **流式 I/O** 的入门：先吃透「一个字节一个字节读、自己判 `\n`、自己切单词」，再去看 `fgets`、`read`、DPDK mbuf 解析。后续第 7、8 章会展开文件与系统接口。

## 要点

### 1. 为什么不用 `gets` / `fgets`，偏要用 `getchar`？

标准库里其实有 **`gets`**（读一整行）和 **`fgets`**（带长度限制的读行），但本节 **刻意不用**，而是用 `getchar` 手动实现行处理与单词切分：

| 方式 | 特点 |
|------|------|
| **`gets` / `fgets`** | 库函数一次读入一行到缓冲区，逻辑封装在 stdio 内部 |
| **`getchar` 循环** | 每次只拿一个字符，**行界、词界、状态机全由你自己写** |

**教学目的**：让你吃透 **流式 I/O 的底层逻辑** —— 输入是一串字节流，`\n` 只是流里的一个字符，「一行」是你用状态机 **定义** 出来的，不是库替你猜的。

**工程背景（HFT / 低延迟）**：

- 行情、FIX/二进制协议、共享内存 ring、DPDK mbuf 往往 **没有「一行」这种高层概念**，只有连续字节
- 低延迟路径常 **绕开封装好的库函数**，自己控制每一个字符/字节的读写与分支，避免多余拷贝和不可控的缓冲策略
- 先会用 `getchar` 写 `wc`，再读内核 `read()`、再写 **零拷贝解析**，是同一条能力链

> **`gets` 已从 C11 标准移除**（缓冲区溢出），永远不要用。`fgets` 安全，适合应用层读行；**本节练的是「不用 fgets 也能做行统计」**。

### 2. `getchar` / `putchar` 基础

```c
int c = getchar();   /* 从 stdin 读一个字符；返回值是 int，不是 char */
putchar(c);          /* 向 stdout 写一个字符 */
```

- **`getchar()`** 从 **stdin** 读 **一个** 字符，返回 **`int`**（为何不是 `char` 见下）
- **`putchar(c)`** 向 **stdout** 写 **一个** 字符
- **`EOF`**（End Of File）：输入结束标记，常与 **`-1`** 比较；**必须用 `int` 接返回值**，否则与 `unsigned char` 混淆时 EOF 可能判错

经典读循环（[1.5.1](./1.5.1-文件复制.md) 起贯穿全节）：

```c
int c;
while ((c = getchar()) != EOF) {
    /* 对每个字符 c 做点什么 */
}
```

### 4. 重定向与 stdin/stdout

程序 **不打开文件名**，只读写标准流：

```bash
./wc < input.txt > output.txt    # 等价于 cat / 管道
echo "hello" | ./wc
```

体会 **stdin/stdout** 与终端、文件、管道的关系 —— 与后面 `fopen`、`read/write` 系统调用是同一套抽象的不同层。

### 5. `getchar` vs `fgets` 对照（知道即可，本节不练 fgets）

```c
/* 应用层读一行 — 安全，但黑盒缓冲 */
char buf[256];
if (fgets(buf, sizeof buf, stdin) != NULL) { /* 处理 buf */ }

/* 本节风格 — 自己拼「行」的逻辑 */
int c, len = 0;
char line[256];
while ((c = getchar()) != EOF && c != '\n' && len < 255)
    line[len++] = (char)c;
line[len] = '\0';
```

前者一行搞定；后者让你看见 **换行符何时出现、缓冲区边界如何守**。协议解析里几乎都是后者思路。

### 6. 警惕宏 hijack `getchar`（与 [1.4 §5](../1.4-符号常量.md) 同源）

`<stdio.h>` 里 **`getchar` 本身常是宏**（展开成 `getc(stdin)` 一类）。若你在头文件里再写：

```c
#define getchar my_getchar
```

预处理器会在 **整个翻译单元** 里把每一处 `getchar` 换成 `my_getchar` —— **不管** 它在哪个函数、哪一层 `{}` 里。这不是 C/Rust 作用域能挡的。

- 1.5 所有 `while ((c = getchar()) != EOF)` 实际调的是 **你的** 实现
- 编译能通过，gdb 里函数名也可能对不上，HFT 里属于 **静默行为漂移**

排查：`gcc -E wc.c | grep getchar` 看预处理结果；规范是 **宏名全大写 + 前缀**，永不与库函数同名。

## HFT / 底层开发的用处

1. **按字节扫描**：Order book 增量、逐字段 FIX、自定义二进制头 — 与 1.5.4 状态机同源
2. **少依赖 stdio 缓冲**：热路径用 `read`/`recv`/mmap ring，逻辑仍是「读一字节、改状态、可能输出」
3. **可测试、可预测**：自己写的状态机比调库函数内部缓冲行为更容易 profiling 与优化

## 与 C++ 的差异

| 主题 | C（本节） | C++ |
|------|-----------|-----|
| 字符 I/O | `getchar` / `putchar` | 更常用 `std::cin.get()` / stream，但原理类似 |
| 读行 | `fgets`；**不用** `gets` | `std::getline` |
| EOF | `int` + `EOF` | stream 的 `eof()` / 返回值 |
