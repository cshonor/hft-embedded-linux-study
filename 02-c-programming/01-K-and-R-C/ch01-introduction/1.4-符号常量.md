# 1.4 符号常量

## 本节讲什么

承接 [1.3 for 语句](./1.3-for语句.md) 的温度表循环，把 `0`、`300`、`20` 等 **魔法数字** 提成 **符号常量**：用 `#define` 在 **预处理阶段** 做纯文本替换，让循环上下限、步长 **一眼可读**，改数值只动一处。

## 要点

### 1. 核心概念

`#define` 是 C 语言 **预处理器指令**，用来定义 **符号常量**，替代代码里零散、无注释的裸数字。

```c
#define 常量名 替换文本
```

- 编译器在 **编译前**（预处理阶段）做 **纯文本替换**
- **不是变量**：没有类型、不能取地址、不参与运行时赋值；**宏名本身不占用运行时内存空间**
- 行末 **不必加分号**（加了会把 `;` 一并替换进代码，可能引发语法错误）

### 2. 三大要点

**① 文本替换机制**

预处理阶段直接全文替换，不存在数据类型，和普通变量完全不同。

```c
#define UPPER 300
/* 源码里每一处 UPPER 在编译前都变成 300 */
```

**② 提升可维护性**

数值只定义一次；后续修改只改 `#define` 一行，整个程序全部生效，不用到处搜数字。

**③ 本节只讲无参宏**

带参数的复杂宏（函数式宏）留到 [第 4 章预处理器](../../03-C-Traps-and-Pitfalls/ch06-preprocessor/) 与 K&R ch4 再学。

### 3. 魔法数字 vs 符号常量

| 写法 | 问题 |
|------|------|
| `for (fahr = 0; fahr <= 300; fahr += 20)` | `0` / `300` / `20` 含义不明，改步长易漏改 |
| `for (fahr = LOWER; fahr <= UPPER; fahr = fahr + STEP)` | 名字自带语义，循环头自解释 |

### 4. 优点总结

1. **可读性**：`UPPER` 一看即上限，`STEP` 即步长
2. **易维护**：改数值只改一处
3. **统一规范**：避免相同数字写多份、改错一份

### 5. 宏不管作用域 —— 能「偷偷」覆盖标准库（HFT 致命坑）

`#define` 由 **预处理器** 处理，发生在编译之前；它 **不认** C 的块作用域、函数作用域，也 **不认** Rust 那套词法作用域 —— 只要 token 匹配，**全文替换**。

```c
#define getchar my_getchar

int c = getchar();   /* 预处理之后 → int c = my_getchar(); */
```

后果：

- `<stdio.h>` 里的 **`getchar` 宏/函数名被劫持**，后面所有 `getchar` 都变成 `my_getchar`
- 编译 **仍可能通过**，行为却在运行时 **静默跑偏** —— 读缓冲策略、锁、延迟全变了
- HFT 热路径里这种 bug **极难查**：源码看起来调的是标准库，预处理后根本不是

| 机制 | `#define` 宏 | `const` / 函数 |
|------|-------------|----------------|
| 处理阶段 | **预处理**（编译前） | 编译、链接 |
| 作用域 | **无** — 从定义点到 `#undef` 或文件末 | 块/文件/链接域 |
| 能否覆盖 `getchar` | **能**，纯文本替换 | 不能靠名字偷偷换 stdio |
| 运行时内存 | **宏名不占**；替换结果可能是立即数 | **`const` 对象占内存**（常在常量区） |

与 **`const` 只读变量、字符串字面量** 对比见 [2.3 §0](../ch02-types-operators-expressions/2.3-常量.md)、[4.3.1 常量区](../ch04-functions-and-program-structure/4.3.1-程序内存布局.md)。

**工程规范**：

- 宏名 **全大写 + 项目前缀**：`HFT_RING_SIZE`，不要叫 `getchar`、`max`、`min`
- **禁止** `#define` 标准库函数名、关键字、常见 typedef（`uint32_t`、`size_t`）
- 怀疑被宏污染时：`gcc -E file.c | less` 看预处理结果；或 `#undef getchar`（仅当确知误定义）

详见 [C 陷阱 6.5 宏名冲突](../../03-C-Traps-and-Pitfalls/ch06-preprocessor/6.5-宏名冲突.md)；与 [1.5 getchar 循环](./1.5-character-io/1.5-字符输入输出.md) 对照理解。

## 完整示例（温度表 + 符号常量）

```c
#include <stdio.h>

#define LOWER  0    /* 华氏下限 */
#define UPPER  300  /* 华氏上限 */
#define STEP   20   /* 步长 */

int main(void)
{
    int fahr;

    for (fahr = LOWER; fahr <= UPPER; fahr = fahr + STEP) {
        printf("%3d %6.1f\n", fahr, (5.0 / 9.0) * (fahr - 32));
    }

    return 0;
}
```

对比裸数字版：别人不必猜 `300` 是上限还是别的业务阈值；改表范围时只改 `#define UPPER`。

```bash
gcc -Wall -Wextra -std=c11 -o fahr_define fahr_define.c
./fahr_define
```

## 与 C++ 的差异

| 主题 | C（本节） | C++ |
|------|-----------|-----|
| 常量写法 | `#define NAME value` 仍常见 | 更推荐 **`const`** / **`constexpr`**（有类型、作用域、类型检查） |
| 是否宏 | 纯文本替换，无类型 | `const` 是具名对象，可调试、可取址（若需要） |
| 兼容性 | 内核、DPDK、老 C 库大量 `#define` | 仍可 `#include` C 头，混用 `#define` |

C++ 工程里新代码优先 `constexpr`；读 Linux 内核 / 协议头文件时 **`#define` 仍是基本功**。

## HFT / 底层开发的用处

1. **协议与配置常量**：端口号、缓冲区大小、tick 步长、最大档位 — 用宏名避免魔数散落
2. **条件编译开关**：`#define DEBUG 1` 配合 `#ifdef`（ch4 / ch6 展开）
3. **注意陷阱**：宏无类型，`#define SIZE 1024*1024` 若缺括号，在表达式里可能算错 — 复杂替换见预处理器章节
4. **禁止宏名撞库函数**：`#define getchar my_getchar` 会全文劫持 stdio，低延迟路径上属于 **隐藏致命 bug**（见上文 §5）
