# 4.10 递归

## 本节讲什么

函数 **直接调用自身**（直接递归）或 **通过其他函数间接调用自己**（间接递归）。写法简洁、贴近分治定义，但每次调用在 **程序栈** 上占用独立 **栈帧**；**必须有基线终止条件**，否则无限递归 → **栈溢出（Stack Overflow）**。

**前置**：[4.1 函数](./4.1-函数的基本知识.md)、[4.3.1 程序内存布局 §6](./4.3.1-程序内存布局.md)（代码区 vs 栈帧）。

## 一、递归两大必备组成

| 组成 | 作用 |
|------|------|
| **基线条件（base case）** | 递归 **终止出口**，不再调用自身 |
| **递归递推步骤** | 把问题 **缩小** / 向基线推进，逐步分解 |

```
递归 = base case（停） + recursive step（更小的问题）
缺 base case → 无限 call → stack overflow
```

经典场景：**快速排序**、树/链表遍历、**进制转换**、表达式求值、目录树遍历。

## 二、底层内存执行原理（关联 4.3.1）

1. 函数 **机器指令** 在 **代码段（`.text` / 代码区）只有一份**，所有递归层 **共用同一段机器码**；
2. **每一次** 递归调用在 **程序栈** 分配 **独立栈帧**，包含：
   - 当前层 **局部变量**
   - **返回地址**（`ret` 跳回哪）
   - **寄存器现场**、**形参副本**
3. 栈是 **有限固定大小** 内存：
   - 用户态进程栈通常 **几 MB**（可 `ulimit -s` 查看）；
   - **Linux 内核栈** 仅 **约 8KB**（每 CPU 一份），深递归极易崩溃；
   - 递归深度 ↑ → 栈帧数量 ↑ → 最终 **栈溢出**。

```
代码区：printd 的机器码 ── 全程一份
栈区：  printd(123) 栈帧 → printd(12) 栈帧 → printd(1) 栈帧 → 逐层 pop
```

详见 [4.3.1 §6](./4.3.1-程序内存布局.md)：溢出的是 **栈**，不是代码区。

## 三、示例拆解：`printd` 数字打印（含负号）

K&R 经典：先递归处理 **高位**，返回时再 `putchar` **低位**，自然得到正序输出。

```c
void printd(int n)
{
    if (n < 0) {           /* 处理负号 */
        putchar('-');
        n = -n;
    }
    if (n / 10)            /* 递推：还有高位 → 继续递归 */
        printd(n / 10);
    putchar(n % 10 + '0'); /* 基线触发后，逐层返回时打印当前位 */
}
```

调用 `printd(123)` 的执行流程：

| 步骤 | 动作 | 栈上状态 |
|------|------|----------|
| 1 | `printd(123)` → `123/10≠0` → 调 `printd(12)` | 栈帧 1（`n=123`） |
| 2 | `printd(12)` → 调 `printd(1)` | 栈帧 2（`n=12`） |
| 3 | `printd(1)` → `1/10=0`，**基线**，打印 `'1'` | 栈帧 3（`n=1`） |
| 4 | 返回栈帧 2，打印 `'2'` | pop |
| 5 | 返回栈帧 1，打印 `'3'` | pop |

输出：`123`。

> `putchar` 在 **递推之后** 执行，所以数字是 **正序** 而非逆序；这与「先压栈再弹栈」的 LIFO 语义一致。

## 四、递归 vs 迭代（工程取舍）

| 倾向 **递归** | 倾向 **迭代**（循环） |
|---------------|----------------------|
| 树、AST、文件系统目录、天然分治（快排、表达式） | **HFT 热路径**、**内核调度**、嵌入式实时 |
| 深度 **固定且很小**，可读性优先 | 栈深度 **不可控**、延迟敏感、硬栈受限 |
| 业务层解析、编译器前端 | 追求 **常数开销小**、栈占用 **恒定** |

### 底层开销差异

| | **递归** | **迭代** |
|---|---------|---------|
| 每次步骤 | `call` 压栈 + 保存寄存器；`ret` 弹栈恢复 | 寄存器运算 + 分支跳转 |
| 空间 | **O(深度)** × 栈帧大小 | 通常 **O(1)**（或手动栈在堆/静态区） |
| 内核/HFT | 8KB 内核栈，**危险** | **首选** |

### 深递归替代：手动显式栈

内核/底层 **禁止依赖深递归**。用 **数组模拟栈**，数据放在 **静态区/堆**，不受 8KB 程序栈限制：

```c
/* printd(123) 的迭代版：静态栈存待打印数字，再逆序输出 */
#define STK_SZ 32
void printd_iter(int n)
{
    static int stk[STK_SZ];   /* 静态区，非进程栈 */
    int top = 0;

    if (n < 0) {
        putchar('-');
        n = -n;
    }
    if (n == 0) {
        putchar('0');
        return;
    }
    while (n > 0) {
        stk[top++] = n % 10;
        n /= 10;
    }
    while (top > 0)
        putchar(stk[--top] + '0');
}
```

> 这是「递归改循环」的最简形态；树/图遍历则用 `stk[]` 存 **待访问节点指针**，逻辑与递归版一一对应。

## 五、关键考点

### 1. 复杂度

- **时间**：递归 **总调用次数**（与递推规模相关）；
- **空间**：**最大递归深度** × **单栈帧大小**（常被忽略的隐性成本）。

### 2. 递归 ↔ 循环等价改写

任何递归都可改成 **「循环 + 手动栈」**，没有 **不可替代** 的递归逻辑；差别在可读性与栈开销。

### 3. 尾递归（避坑）

**尾递归**：递归调用是函数 **最后一条** 可执行语句，理论上可 **复用当前栈帧**（尾调用优化 TCO），空间降为 O(1)。

```c
/* 尾递归形式：return 前一步就是递归调用 */
int fact_tail(int n, int acc)
{
    if (n <= 1)
        return acc;
    return fact_tail(n - 1, n * acc);   /* 最后一句话 */
}
```

⚠️ **C 标准不强制 TCO**；GCC 在 `-O2` 下 **可能** 优化，**不保证**。生产代码（内核、HFT）**绝对不能依赖尾递归**，仍按有栈帧开销评估。

## 六、工程实践

| 场景 | 建议 |
|------|------|
| 应用层（表达式解析、目录遍历） | 递归可读性优先；评估最大深度 |
| 内核、裸机、HFT 低延迟 | **迭代 + 手动静态栈** |
| 深度评估 | 超过 **~500 层** 建议改迭代（用户态亦稳妥） |
| 内核遍历不确定深度结构 | **严禁** 裸递归 |

1. 递归前 **人工估算** 最大深度与栈帧大小；
2. 内核开发：深度不确定的链表/树遍历 → **显式栈** 或 **队列**；
3. 崩溃调试：理解栈帧后，`backtrace` / 汇编 `call`/`ret` 即「递归路径的逆向弹出」。

## 背诵要点（精简版）

```
递归 = base case + 向 base 推进
代码区一份指令；每层调用一份栈帧
深递归 → stack overflow（内核栈 ~8KB 最危险）
HFT/内核 → 迭代 + 手动栈，别赌尾递归 TCO
时间看调用次数；空间看最大深度 × 栈帧
```
