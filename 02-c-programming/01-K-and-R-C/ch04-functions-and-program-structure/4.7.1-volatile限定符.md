# 4.7.1 `volatile` 限定符

## 本节讲什么

**`volatile`** 是修饰变量的关键字，告诉编译器：**这个变量的值可能在程序「不知情」的情况下被改变**（硬件、中断、信号处理函数、其他线程等）。因此编译器 **不能** 把它优化成「只读一次、一直用寄存器里的旧值」，**每次使用都应从内存（或映射的设备地址）重新读取**。

**前置**：[4.7 寄存器变量](./4.7-寄存器变量.md)、[2.4 声明](../ch02-types-operators-expressions/2.4-声明.md)、[2.9 按位运算符 §volatile](../ch02-types-operators-expressions/2.9-按位运算符.md)。

## 要点

### 0. 寄存器优化 vs `volatile`（一对相反方向）

**差不多就是你理解的那个意思**：

| | **寄存器优化**（编译器默认） | **`volatile` 的要求** |
|---|------------------------------|------------------------|
| **谁触发** | 编译器 `-O1` 及以上常 **自动** 做 | 程序员标 **`volatile`** |
| **做什么** | 频繁使用的变量 **尽量放寄存器**；或 **读一次内存后长期在寄存器里用旧值** | **禁止** 这样优化 |
| **每次用变量时** | 可以用寄存器里 **缓存的值** | **必须老老实实再访内存**（或 MMIO 地址）读 **最新值** |
| **哪怕源码没写赋值** | 编译器假设内存里的值 **不变**，可省略重复读 | **不能假设** —— 硬件/中断/别的线程可能 **背着你改了** |

```
普通 int flag:
  读一次 → 放进寄存器 → 后面一直用寄存器里的 flag（快，但可能过时）

volatile int flag:
  每次 if / while / 表达式用到 → 都要再从内存读 flag（慢，但是新的）
```

**`register` 关键字**（[4.7](./4.7-寄存器变量.md)）是 **人** 对编译器说「请尽量寄存器化」；**`volatile`** 是 **人** 对编译器说「**别** 对我寄存器化/别省略内存访问」。两者都作用在 **编译器优化** 这一层，方向 **相反**。

### 1. 编译器默认会做什么优化

对 **普通变量**，编译器可能：

```c
int flag = g_ready;
if (flag)
    do_work();
else
    wait_more(flag);   /* 可能仍用上面缓存的 flag，不再读 g_ready */
```

若 `g_ready` 其实在 **别的线程** 或 **中断里** 已变成 1，但本线程 **一直用寄存器里旧的 0**，逻辑就错了。

**`volatile`** 的作用：**禁止** 这类「把内存读搬进寄存器后长期复用」的 **寄存器优化** —— 每次表达式里用到，都应 **再访一次对象**；**哪怕本函数里没有任何一行给该变量赋值**，也不能假定它不变。

### 2. 语法与适用对象

```c
volatile int status;
volatile uint32_t *reg;
const volatile int hw_counter;   /* 只读设备计数：不能写，但每次读不能省 */
```

| 可修饰 | 不可单独靠 `volatile` 解决 |
|--------|---------------------------|
| 局部、全局、**指针指向的类型** | **原子性**、**内存序**（多线程要用 `atomic`、锁） |
| `volatile int *p` / `int volatile *p` | `volatile` **不保证** 读改写原子 |

`volatile` 是 **类型限定符**（type qualifier），和 `const` 类似，可修饰变量、指针类型（[5.12](../ch05-pointers-and-arrays/5.12-复杂声明.md)）。

### 3. 典型场景：谁在「背着你」改变值

| 场景 | 例子 |
|------|------|
| **硬件寄存器 / MMIO** | `volatile uint32_t *GPIO_DR` —— 每次读写都真访问设备 |
| **信号处理函数** | `volatile sig_atomic_t flag = 0;` —— handler 置位，主循环轮询 |
| **被中断修改的全局标志** | 嵌入式里主循环与 ISR 共享的 `volatile int done` |
| **内存映射 I/O** | 网卡/DMA 描述符环状态位 |

```c
#define REG (*(volatile uint32_t *)0x40001000u)

void poll_ready(void)
{
    while (REG == 0)   /* 每次循环都从地址读，不能用上次缓存 */
        ;
}
```

### 4. 与 `register` 对比 —— 语义几乎相反

| | **`register`** | **`volatile`** |
|---|----------------|----------------|
| **意图** | 「尽量放寄存器，**少访内存**」 | 「**每次都要访内存**，别用旧缓存」 |
| **优化** | 允许编译器常驻寄存器 | **禁止** 省略对内存的访问 |
| **典型场景** | 热循环计数器（历史用法） | 硬件、异步修改的变量 |
| **能否 `&` 取址** | ❌（`register` 变量） | ✅（`volatile` 变量一般有地址） |

```c
register int i;      /* 建议：别老从栈读 */
volatile int flag;   /* 要求：别假设 flag 不变 */
```

### 5. `volatile` 不是什么

| 误解 | 事实 |
|------|------|
| 「多线程共享变量加 `volatile` 就安全」 | ❌ **不保证原子性**，`i++` 仍可能竞态；要用 **mutex / `atomic`** |
| 「比 `const` 还只读」 | ❌ `volatile` 只说不优化访问；**可读写**；`const volatile` 才是只读设备寄存器 |
| 「所有全局都要 `volatile`」 | ❌ 仅 **可能被外部异步修改** 的才需要 |

C11 起多线程同步用 **`<stdatomic.h>`**；`volatile` **不能替代** `atomic` 或锁。

### 6. 和 `const` 组合

```c
const volatile uint32_t *rom_status;  /* 只读硬件寄存器：每次读都真读，不能写 */
```

- **`const`**：你不能通过该左值 **写入**
- **`volatile`**：编译器不能 **省略** 读操作

## 易混点

| # | 要点 |
|---|------|
| 1 | **`volatile` = 防编译器优化掉访问**，不是「变量在寄存器里」 |
| 2 | 每次使用应从 **内存/映射地址** 读，不用寄存器里 **过期的缓存值** |
| 3 | 硬件 MMIO、信号标志是经典用途 |
| 4 | **≠ 线程安全**；多线程用 `atomic`/锁 |
| 5 | **寄存器优化** = 编译器自动把热变量放寄存器/缓存读；**`volatile` 禁止** 对该变量这么做 |

## HFT / 内核提示

与 [4.7 `register`](./4.7-寄存器变量.md) **成对** 理解：内核 **`volatile` 刚需**，HFT **`register` 可选**。

### 内核开发：`volatile` 几乎少不了

硬件寄存器对应的 **内存映射地址** **必须** 用 **`volatile`** 修饰：

- 防止编译器做 **寄存器优化**（读一次后长期用寄存器里的旧值）
- 否则轮询 **状态位、FIFO、中断标志** 时，可能 **读不到硬件实时状态**，驱动 **假死** 或 **丢事件**

```c
#define GPIO_IDR (*(volatile uint16_t *)0x48000010u)
while ((GPIO_IDR & 0x1) == 0)   /* 每次循环都真读引脚 */
    ;
```

- **Linux 内核**：`readl`/`writel`、`ioremap` 后的访问封装 —— 不要裸指针还忘了 `volatile`
- **嵌入式 BSP**：MMIO 宏几乎清一色 `volatile` + 正确宽度类型

### 高频量化：`volatile` 少写，`register` 看情况

| 场景 | 建议 |
|------|------|
| **用户态策略热循环** | 计数器 **可** 写 `register int i`；**`-O3` 常已自动** 放进寄存器，不写也行 |
| **行情/共享内存标志** | 优先 **`atomic`、内存序、无锁队列**；不要指望 `volatile` 解决线程安全 |
| **网卡寄存器** | 走 **DPDK PMD / 驱动**，应用层不直接 cast MMIO |

**一句话**：内核里 **`volatile` 关乎正确性**；HFT 里 **`register` 顶多锦上添花**，瓶颈多在 **缓存、分支、系统调用**，编译器 **自动寄存器优化** 往往已够用。
