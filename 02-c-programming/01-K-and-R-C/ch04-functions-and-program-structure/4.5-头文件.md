# 4.5 头文件

## 本节讲什么

**头文件**（后缀 **`.h`**）专门放 **声明** 和可共享的 **类型/宏**，供多个 `.c` 通过 **`#include`** 复用。多文件工程不只是语法需要 —— 更为了 **开发维护效率**：**代码复用**、**增量编译**、**模块化协作**（§8）。每个 `.c` 单独编译成 `.o`，链接器合并；**`.h` 保证大家用的是同一套声明**。

**前置**：[4.1 函数原型](./4.1-函数的基本知识.md)、[4.3 外部变量](./4.3-外部变量.md)、[4.4 作用域与链接](./4.4-作用域规则.md)。

## 要点

### 1. 头文件是什么

**头文件** = 后缀为 **`.h`** 的文件，**专门用来放声明**（以及宏、类型定义），**一般不写函数体、不写全局变量定义**。

| 常放内容 | 示例 |
|----------|------|
| **变量的 `extern` 声明** | `extern int packet_count;` |
| **函数原型** | `int add(int a, int b);` |
| **宏定义** | `#define MAXLINE 1000` |
| **结构体 / 类型定义** | `struct node { ... };`、`typedef int bool_t;` |

**不放**（应放在 **某一个 `.c`**）：

- 全局变量 **定义** `int num = 0;`
- 函数 **实现** `{ ... }`
- `static` 模块私有符号（只在本 `.c`）

### 2. 为什么需要头文件：`#include` 复用声明

当 **多个 `.c`** 都要用 **同一组声明** 时：

- **不好**：每个 `.c` 手写一遍原型 → 易抄错、改一处漏改别处
- **好**：写在 **一个 `.h`**，各 `.c` **`#include`** 进来

```c
/* math_utils.h */
#ifndef MATH_UTILS_H
#define MATH_UTILS_H

extern int g_seed;
int add(int a, int b);
double square(double x);

#endif
```

```c
/* main.c */
#include "math_utils.h"

int main(void)
{
    int s = add(1, 2);
    return 0;
}
```

```c
/* math_utils.c */
#include "math_utils.h"

int g_seed = 0;              /* 唯一定义 */

int add(int a, int b)
{
    return a + b;
}

double square(double x)
{
    return x * x;
}
```

**好处**：

1. **避免重复写** 相同声明
2. **保证一致** —— 所有 `.c` include 同一份 `.h`，编译器看到的 **函数签名、类型、宏** 相同
3. **接口清晰** —— `.h` = 模块对外「菜单」，`.c` = 实现

预处理时 `#include "math_utils.h"` 等价于把 `.h` 内容 **粘贴** 进当前 `.c` 再编译（[4.11 预处理器](./4.11-c-preprocessor/4.11.1-文件包含.md)）。

### 2.1 完整例子：`global_num`（`a.c` 定义 + `a.h` 声明 + 其他 `.c`）

**① 在 `a.c` 里定义（全程序唯一定义）**

```c
/* a.c */
int global_num = 0;
```

**② 在 `a.h` 里写 `extern` 声明**

```c
/* a.h */
#ifndef A_H
#define A_H

extern int global_num;

#endif
```

**③ 其他 `.c` 包含 `a.h` 即可用，不必再手写 `extern`**

```c
/* b.c */
#include <stdio.h>
#include "a.h"

void show_in_b(void)
{
    printf("%d\n", global_num);
}
```

```c
/* c.c */
#include "a.h"

void bump_in_c(void)
{
    global_num++;    /* 改的是 a.c 里定义的那一份 */
}
```

```bash
gcc a.c b.c c.c -o demo
```

| 文件 | 写什么 |
|------|--------|
| **`a.c`** | `int global_num = 0;` —— **定义**（分配内存，**仅此一处**） |
| **`a.h`** | `extern int global_num;` —— **声明** |
| **`b.c` / `c.c`** | `#include "a.h"` —— 自动带上声明 |

#### 运行时：全工程 **唯一一份** 数据

链接完成后，**不管哪个 `.c` 包含 `a.h`**，大家操作的都是 **同一块内存** 里的 **同一个** `global_num`：

| 行为 | 结果 |
|------|------|
| **`a.c` 里** `global_num = 10` | 内存里那份变成 10 |
| **`b.c` 里** `printf("%d", global_num)` | 读到 **10** |
| **`c.c` 里** `global_num++` | 变成 11；**`a.c`、`b.c` 再读都是 11** |

**整个工程里只有这一份数据** —— 不是每个 `.c` 各拷贝一份。哪个文件 **修改** 了它，其他文件 **再读取** 时拿到的都是 **修改后的最新值**（进程运行期间一直有效，见 [4.3](./4.3-外部变量.md)）。

```
        a.c 定义
          │
          ▼
    ┌─────────────┐
    │ global_num  │  ← 全程序唯一实体（如地址 0x601000）
    └─────────────┘
       ▲    ▲    ▲
       │    │    │
    a.c读写 b.c读写 c.c读写   （链接后都指向同一地址）
```

> 头文件也可叫 `common.h` 等名字；关键是 **一个 `.c` 定义 + 一个 `.h` 声明 + 其余 `#include`**。

**改类型时如何同步**：若要把 `global_num` 从 `int` 改成 `float`：

1. **`a.c`** 改定义：`float global_num = 0.0f;`
2. **`a.h`** 改声明：`extern float global_num;`
3. 所有 `#include "a.h"` 的 `.c` **自动** 看到新类型 —— **不会出现** b.c 还当 `int`、a.c 已是 `float` 的 **声明/定义不一致**

若不用头文件、各 `.c` 自己抄 `extern`，极易漏改某一文件 → 链接通过或静默错行为。**头文件 = 单一真相来源**。

### 3. 编译链接流程：编译「认声明」，链接「绑定义」

```
a.c  ──→ a.o  ─┐
b.c  ──→ b.o  ─┼──→ 链接器 ──→ a.out
c.c  ──→ c.o  ─┘
```

**编译阶段**（每个 `.c` **单独** 编译成 `.o`）：

| 文件 | 编译器看到什么 | 结果 |
|------|----------------|------|
| **`a.c`** | `int global_num = 100;` —— **定义** | `a.o` 里 **导出** 符号 `global_num`（有实体、占内存） |
| **`b.c`** | `#include "a.h"` → `extern int global_num;` —— **仅声明** | `b.o` 里 **引用** 符号 `global_num`（知道「有这么个 `int`」，**不知道** 定义在哪） |
| **`c.c`** | 同上 | `c.o` 里同样是 **未解析的引用** |

要点：**`b.c` 包含 `a.h` 后，编译器只确认「存在一个外部链接的 `global_num`」**，类型检查、`printf("%d", global_num)` 都能过；**此时不必、也无法** 知道它在哪个 `.c` 定义 —— 那是 **链接器** 的工作。链接绑定后，**任一文件改写，他处读到的都是最新值**（§2.1）。

**链接阶段**（链接器汇总 **所有** `.o` 里的符号表）：

1. 在 **`a.o`** 找到 **`global_num` 的定义**（分配好的那一份内存）
2. 把 **`b.o`、`c.o`** 里对 `global_num` 的 **引用** 填成 **同一地址**
3. 生成最终可执行文件 —— 三文件访问的是 **同一块内存**

```
a.o:  global_num  ← 定义（实体）
b.o:  global_num  ← 引用 ──┐
c.o:  global_num  ← 引用 ──┼── 链接器绑定到 a.o 那份
```

若 **没有任何 `.o` 提供定义** → **`undefined reference to 'global_num'`**（未定义引用）。  
若 **多个 `.o` 都有定义** → **`multiple definition of 'global_num'`**（重复定义）。

- **编译单元** = **一个 `.c` 文件**（`#include` 展开后）单独编译 → **`.o`**
- **链接器** 合并 `.o`；**外部链接** 符号跨文件 **定义一次、引用多处**（[4.4 §3](./4.4-作用域规则.md)）

§2.1 的 `global_num` 例子走的就是这条路径：`common.h` 保证各 `.c` **编译时声明一致**；链接器保证 **运行时只有一份实体**。

### 4. 单定义规则：`.h` 只声明，`.c` 定义

| 放在 `.h` | 放在 **某一个** `.c` |
|-----------|---------------------|
| `extern int num;` | `int num = 0;` |
| `int foo(int x);` | `int foo(int x) { return x; }` |
| `struct point { int x, y; };` | （类型定义可放 `.h`；带静态数据的定义放 `.c`） |

- 每个 **外部链接** 符号全程序 **只能定义一次**
- **`.h` 被多个 `.c` include** 时，写 `int num = 0;` → 每个 `.c` 各一份定义 → 链接 **重复定义**
- 正确：`.h` 写 **`extern int num;`**，**一个** `.c` 写 **`int num = 0;`**

见 [4.3](./4.3-外部变量.md)。

### 5. include 守卫（防止重复展开）

```c
#ifndef MATH_UTILS_H
#define MATH_UTILS_H
/* 头文件内容 */
#endif
```

同一 `.c` 多次 include 或链式 include 时，避免重复定义类型/宏。

### 6. `#include "..."` vs `#include <...>`

| 写法 | 查找路径 | 典型 |
|------|----------|------|
| **`#include "foo.h"`** | **当前目录**、工程路径优先 | 项目自己的 `.h` |
| **`#include <stdio.h>`** | **系统/编译器** 标准路径 | 标准库、第三方库 |

### 7. 组织方式：头文件 **不必** 与 `.c` 一一对应

**没有**「每个 `.c` 必须配一个同名 `.h`」的硬性规定。怎么拆头文件，**完全看项目需求** —— **怎么方便维护就怎么来**。

| 做法 | 适用场景 | 示例 |
|------|----------|------|
| **公共头文件** | 多个 `.c` **共用** 的声明、类型、宏 | `common.h` 放 `extern int global_num;`，`a.c` / `b.c` / `c.c` 都 include |
| **模块头文件**（常与 `.c` 成对） | 某个 `.c` 实现一个模块，对外暴露 **API** | `stack.h` 声明 `push`/`pop`，`stack.c` 实现；`main.c` 只 include `stack.h` |
| **混合** | 真实工程最常见 | 全局配置/错误码放 `common.h`；各子系统各有 `xxx.h` + `xxx.c` |

```
方式 A —— 公共头：
  common.h  ←── a.c, b.c, c.c 都 include（共享 extern、宏、类型）

方式 B —— 模块头：
  stack.h  ←── stack.c（实现）+ main.c（调用方）
  parser.h ←── parser.c + main.c

方式 C —— 混合：
  common.h + stack.h + parser.h + main.c
```

**原则**：

- **谁需要这份声明，谁就 `#include` 对应 `.h`** —— 不必为了「对称」硬造没用到的头文件
- **`.h` 描述的是「接口/契约」**，不是「`.c` 的镜像」；一个 `.h` 可被很多 `.c` include，一个 `.c` 也可 include 多个 `.h`
- 小项目：`common.h` 一把梭；变大后：按 **功能模块** 拆 `net.h`、`file.h` 等，**边界更清晰**

#### 按功能模块划分头文件（常见工程做法）

通常会 **按功能模块** 来划分头文件 —— 同一子系统的 **函数声明、相关类型、`extern` 全局** 放在 **同一个 `.h`**：

| 头文件 | 放什么 | 谁 include |
|--------|--------|------------|
| **`net.h`** | 网络相关：套接字 API、`extern int net_errno;`、协议结构体 | 只做网络的 `.c` |
| **`file.h`** | 文件 I/O：`open_log`、`read_config`、`extern FILE *log_fp;` | 只做文件操作的 `.c` |
| **`common.h`** | 全工程共用的错误码、基础类型 | 多数 `.c` |

```c
/* net.h */
#ifndef NET_H
#define NET_H
extern int packets_sent;
int net_connect(const char *host, int port);
void net_close(int fd);
#endif

/* file.h */
#ifndef FILE_H
#define FILE_H
int load_config(const char *path);
void write_log(const char *msg);
#endif
```

```c
/* server.c —— 只做网络 */
#include "net.h"

/* config_loader.c —— 只做文件 */
#include "file.h"

/* main.c —— 两者都要 */
#include "net.h"
#include "file.h"
```

**好处**（你的归纳）：

1. **按需包含** —— 哪个 `.c` 需要哪块功能，**只 `#include` 对应头文件**，不引入 **多余声明**
2. **减少耦合** —— `config_loader.c` 不必看见 `net_connect` 的原型，编译依赖更干净
3. **缓解命名冲突** —— 各模块内部状态用 **`static` 藏在 `net.c` / `file.c`**；对外名字集中在各自 `.h`，比一个大杂烩 `all.h` 不易撞名

```
❌ 不好：everything.h 塞进所有声明 → 每个 .c 都 include → 改一行牵动全员重编

✅ 较好：net.h / file.h / ui.h 分模块 → 谁用谁 include
```

模块实现仍常见 **`net.c` + `net.h`** 成对；私有 helper 用 **`static` 函数** 不写进 `net.h`（[4.6](./4.6-静态变量.md)）。

下面栈模块示例是 **方式 B**（`stack.h` + `stack.c`）；§2.1 的 `global_num` 是 **方式 A**（`a.h` 跨多文件）。

### 8. 工程价值：提高开发与维护效率

多文件 + 头文件，主要不是为了「显得专业」，而是实实在在 **省时间、好协作**：

| 维度 | 怎么做 | 得到什么 |
|------|--------|----------|
| **① 代码复用** | 通用 **函数原型、类型、宏、`extern` 声明** 放进 **`.h`**；多个 `.c` **`#include` 同一份** | 不用每个源文件抄一遍；改接口 **只改一处** `.h` |
| **② 编译效率** | 每个 `.c` **单独** 编译成 `.o`；改一个源文件后 **只重编该 `.c`**，再 **链接** 其余已有 `.o` | 大工程改一行 **不必全盘重编** |
| **③ 模块化封装** | 按功能拆 **`stack.c` / `parser.c` / `io.c`**，各配 **`stack.h` 等对外 API**；实现细节 **`static` 藏在 `.c`** | **逻辑清晰**；**多人并行** 改不同模块，减少冲突 |

#### ① 代码复用

```c
/* common.h —— 一份声明，多处 include */
extern int errno_code;
int parse_line(const char *s);
```

`main.c`、`parser.c`、`test.c` 都 `#include "common.h"` —— **同一套** 函数签名与类型，避免抄错、漏改。

#### ② 编译效率（增量编译）

```
首次：  gcc -c main.c parser.c stack.c   → main.o parser.o stack.o
        gcc -o app main.o parser.o stack.o

只改了 parser.c：
        gcc -c parser.c                    → 新的 parser.o
        gcc -o app main.o parser.o stack.o  /* main.o stack.o 复用，不重编 */
```

| 改了什么 | 通常要重编谁 |
|----------|-------------|
| **某个 `.c` 实现** | **该 `.c`** + 重新链接 |
| **某个 `.h` 声明/结构体布局** | **所有 include 该 `.h` 的 `.c`**（接口变了） |

所以 **`.h` 要稳定**：频繁改结构体成员会让大量 `.c` 跟着重编（[HFT/内核提示](#hft--内核提示)）。

#### ③ 模块化与多人协作

```
project/
  common.h          ← 共享类型、错误码
  stack.h  stack.c  ← 同学 A 负责栈
  parser.h parser.c ← 同学 B 负责解析
  main.c            ← 主控逻辑
```

- **`.h` = 模块合同**：别人只 include 你的头文件，就能调用 **对外 API**
- **`.c` = 黑盒实现**：内部 `static` 函数/变量 **不暴露**，各改各的文件，**边界清楚**

Linux 内核、DPDK、策略引擎都是同一套路：**头文件管接口，源文件管实现，链接拼成整体**。

## 示例：栈模块（K&R 风格）

```c
/* stack.h —— 声明集中处 */
#ifndef STACK_H
#define STACK_H

#define MAXVAL 100
void push(double f);
double pop(void);

#endif
```

```c
/* stack.c —— 实现 + 本文件私有 */
#include "stack.h"

static int sp = 0;           /* static：不放进 .h，仅本 .c */
static double val[MAXVAL];

void push(double f) { /* ... */ }
double pop(void)    { /* ... */ }
```

```c
/* main.c */
#include "stack.h"

int main(void)
{
    push(1.0);
    return 0;
}
```

## 易混点

| # | 要点 |
|---|------|
| 1 | **`.h` = 声明**；**`.c` = 定义/实现** |
| 2 | 多 `.c` 共用声明 → 写 **一个 `.h`** + **`#include`** |
| 3 | `.h` 里 **`extern` 声明**，**一个** `.c` **定义** |
| 4 | `.h` 里写 `int x = 0;` → 多 include → **链接炸** |
| 5 | **`static` 私有** 放 `.c`，**不要** 写进对外 `.h` |
| 6 | include 守卫防重复展开 |
| 7 | **按功能模块拆 `.h`**（`net.h`、`file.h`）—— 按需 include，少多余声明、少撞名 |
| 8 | 多文件价值：**复用声明**、**增量编译**、**模块化协作** |
| 9 | **`.h` 不必与 `.c` 一一对应** —— 可按 `net.h` / `file.h` 等功能模块拆 |

## HFT / 内核提示

- 驱动：**`xxx.h` 对外 API**，**`xxx.c` 实现** + `static` 私有 —— 改驱动 `.c` 常 **只重编该对象**
- 策略引擎：公共类型/错误码放 `.h`，热路径实现藏 `.c`；**多人** 分模块开发
- 改 `.h` 中结构体布局 → **所有 include 的 `.c` 重编**（接口稳定性）；内核里 ABI 头尤其慎重
- DPDK/大仓库：**增量编译** 是日常；头文件依赖管理（include 守卫、少 include 大 `.h`）直接影响编速
