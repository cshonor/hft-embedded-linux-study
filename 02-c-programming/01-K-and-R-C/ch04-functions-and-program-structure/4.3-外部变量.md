# 4.3 外部变量

## 本节讲什么

**外部变量**在 **所有函数之外** 定义（K&R 语境下的「外部」= **函数外**）。其中 **无 `static` 的全局变量** 带 **外部链接**，可被他文件 `extern` 引用；**`static` 文件级变量** 虽也在 **全局/静态存储区**、寿命贯穿全程，但 **不能跨文件访问**，不算严格意义上「可跨文件共享」的外部变量。理解 **存储期**、**链接** 与 **工程风险**，是读内核全局状态、多文件共享数据的前提。

**前置**：[1.10 外部变量与作用域](../ch01-introduction/1.10-外部变量与作用域.md)、[4.1 函数](./4.1-函数的基本知识.md)、[2.4 声明](../ch02-types-operators-expressions/2.4-声明.md)。

## 要点

### 1. 什么是外部变量

**外部变量**（K&R 用语）= 在 **函数外面**（文件顶层）定义的变量。**日常说的「全局变量」** 通常指其中 **无 `static`**、可跨文件共享的那一类 —— **全局变量本身就是外部变量**；但 **不能** 把「外部变量」和「所有放在全局区的变量」画等号。

```c
int bufpos = 0;        /* 外部变量 / 全局变量 —— 定义，外部链接 */
char buf[BUFSIZ];

void append(int c)
{
    if (bufpos < BUFSIZ)
        buf[bufpos++] = c;
}
```

| 对比 | **局部变量** | **外部变量（无 static 全局）** |
|------|-------------|-------------------------------|
| **定义位置** | 函数或 `{}` 块 **内** | 所有函数 **外** |
| **何时创建** | 每次进入函数/块 | 程序 **启动时** 分配一次 |
| **何时销毁** | 离开函数/块 | 程序 **结束** |
| **谁可见** | 当前块内 | 本文件从定义行到文件末；**他文件** 可 `extern` |
| **初值** | 未初始化 → **垃圾值** | 未初始化 → **清零**（0、`'\0'`） |

**核心**：内存从 **程序启动** 分配到 **程序结束** 才释放 —— **一直在**，不像局部变量函数结束就销毁；也 **不用** 每次调用函数重新创建。

### 1.1 术语辨析：外部变量 ≠ 全局区里的所有变量

口语里常把「函数外定义的变量」都叫 **全局变量**；和 **链接** 相关的 **外部变量** 则 narrower —— 指 **能跨翻译单元（`.c`）共享** 的那类。

| 种类 | 定义示例 | 存储区 | 存储期 | 链接 | 他文件 `extern` |
|------|----------|--------|--------|------|-----------------|
| **全局变量**（无 `static`） | `int g;` | 全局/静态区 | **静态**（全程） | **外部链接** | ✅ **能** 引用 |
| **文件级 `const`**（默认） | `const int lim = 1500;` | **常量区** | **静态**（全程） | **内部链接** | ❌ **不能**（须 `extern const` 定义） |
| **`static` 全局变量** | `static int s;` | 全局/静态区 | **静态**（全程） | **内部链接** | ❌ **不能** |
| **`static` 局部变量** | 函数内 `static int c;` | 全局/静态区 | **静态**（全程） | —（非文件级） | ❌ 名仅函数内 |

要点（你的归纳）：

1. **全局变量（无 `static`）** 本身就是 **外部变量**：作用域可扩展到 **整个程序**（多文件工程下经 `extern` / 头文件），**全程序一份内存**。
2. **`static` 全局变量** 也占 **全局/静态存储区**、也是 **静态存储期**，但被 **`static` 锁在本 `.c`** —— 他文件 **就算写 `extern` 也访问不到**；它是 **静态存储期变量**，但 **不算** 严格意义上 **可跨文件** 的外部变量（[4.6](./4.6-静态变量.md)）。
3. **外部变量不完全等同于「全局变量 + static 变量」这一笼统说法** —— 要看 **链接属性**：能不能被别的 `.c` 绑定到同一份定义。

```
int packet_count;          /* 外部变量 + 全局变量：可跨文件 */
static int file_private;   /* 全局区 + 静态存储期，但非跨文件外部变量 */
```

### 1.2 完整对照：`num` / `s_num` / `c_num`（`a.c` + `b.c`）

三种 **文件顶层** 定义，一张表分清：

| `a.c` 定义 | 存储区 | 链接 | `b.c` 能否用 | `b.c` 能否改 |
|------------|--------|------|-------------|-------------|
| `int num = 10;` | 全局/静态区 | **外部** | ✅ `extern int num;` | ✅ `num++` |
| `static int s_num = 20;` | 全局/静态区 | **内部** | ❌ `extern` 也无效 | ✅ **`a.c` 内可改** |
| `static const int sc = 40;` | **常量区** | **内部** | ❌ | ❌ **只读** |
| `const int c_num = 30;` | **常量区** | **内部**（默认） | ❌ 默认接不上 | ❌ 只读 |
| `extern const int c_num = 30;` | **常量区** | **外部** | ✅ `extern const int c_num;` | ❌ **仍不能改** |

```c
/* ========== a.c ========== */
int num = 10;                    /* 非 static 全局：外部链接 */
static int s_num = 20;           /* static 全局：仅 a.c */
const int c_num = 30;            /* const 默认内部：仅 a.c，常量区 */

void use_in_a(void)
{
    num++;           /* OK */
    s_num++;         /* OK：只有 a.c 能碰 s_num */
    /* c_num = 40; */  /* 编译错误：只读 */
}

/* 若要跨文件共享只读常量，a.c 应写成： */
/* extern const int c_num = 30; */
```

```c
/* ========== b.c ========== */
extern int num;                  /* ✅ 绑定 a.c 的 num，可读可写 */
/* extern int s_num; */          /* ❌ 链接不到 a.c 的 static s_num */
/* extern const int c_num; */    /* ❌ 若 a.c 是 const int c_num=30（无 extern） */

void use_in_b(void)
{
    printf("%d\n", num);         /* 输出 10；改 num 会影响 a.c 同一份 */
    num = 99;                    /* OK */
    /* s_num++; */               /* 编译错误：b.c 根本看不见 s_num */
    /* c_num = 40; */            /* 即使能 extern 引用，也不能改 const 值 */
}
```

**跨文件共享 `c_num` 的写法**（只读、可引用、不可改）：

```c
/* a.c —— 唯一定义 */
extern const int c_num = 30;

/* common.h */
extern const int c_num;

/* b.c */
#include "common.h"
printf("%d\n", c_num);   /* OK：读可以 */
/* c_num = 40; */        /* 编译错误：const 不可写 */
```

```bash
gcc a.c b.c -o demo
```

**一句话**：`num` = 普通全局，**extern 就能共享**；`s_num` = **static 锁死本文件、但在 `a.c` 内可改**；`c_num` = **const 默认文件私有只读**；**`static const`** = **常量区 + 仅本文件 + 不可改**；跨文件只读共享须 **`extern const` 定义**。

**`static const` 叠加**：`const` → 常量区、不可改；`static` → 他文件 `extern` 无效（[4.6 §2](./4.6-静态变量.md)）。

见 [2.3 §0.1](../ch02-types-operators-expressions/2.3-常量.md)、[4.4 §3](./4.4-作用域规则.md)。

关键不是「只在某一个 `.c` 里用」，而是：

> 只要变量定义在 **所有函数外面**，不管工程有 **多少个 `.c` 文件**，在 **其他文件** 里写 **`extern` 声明**，就能 **跨文件** 访问 **同一块内存**。

```c
/* === shared.c：定义一次（造仓库） === */
int packet_count = 0;    /* 全程序只有这一份内存 */

void on_packet(void)
{
    packet_count++;
}

/* === main.c：extern 贴标签，引用 shared.c 里的对象 === */
extern int packet_count;   /* 不分配内存，只说「别处有一个 int」 */

int main(void)
{
    printf("%d\n", packet_count);
    return 0;
}

/* === stats.c：同样可以 extern === */
extern int packet_count;

void print_stats(void)
{
    printf("total: %d\n", packet_count);
}
```

编译链接（以 `global_num` 为例，见 [4.5 §3](./4.5-头文件.md)）：

```bash
gcc -o app main.c shared.c stats.c
```

| 阶段 | 发生什么 |
|------|----------|
| **编译** | 每个 `.c` 单独变 `.o`；含 `extern` 的 `.c` **只知道符号存在**，不知道定义在哪 |
| **链接** | 链接器汇总符号表，把 **引用** 绑到 **唯一定义** 那份内存上 |

`packet_count` 在 `shared.c` **定义**，`main.c` / `stats.c` 经 `extern`（或头文件）**引用** —— 链接后 **只有一份**，三文件共享。

| 要点 | 说明 |
|------|------|
| **定义** | **某一个** `.c` 文件顶层 `int x;` —— **全程序只能一次** |
| **`extern` 声明** | **其他** `.c`（或本文件定义之前）`extern int x;` —— **可多次** |
| **内存** | **一份**，从启动到结束；所有 `.c` 里的函数改的都是 **同一个 `x`** |
| **头文件** | 常把 `extern int packet_count;` 放进 `.h`，各 `.c` `#include`（[4.5](./4.5-头文件.md)） |

**与局部变量对比**：局部变量在 **栈** 上，函数返回就销毁；外部变量在 **静态存储区**（见 [4.3.1 五区总览](./4.3.1-程序内存布局.md)），**跨文件、跨函数调用** 都存活。

### 3. 好处：函数间（跨文件）共享状态

- **所有函数** 都能 **直接读写**（同一翻译单元内）
- 适合 **跨多次调用** 累积的状态：缓冲区位、计数器、配置标志
- 比层层传参省事（小项目、K&R 最长行示例）

```c
int max;               /* 当前最长行长度 */
char longest[1000];    /* 当前最长行内容 */

/* main、mgetline、copy 都直接访问 max、longest */
```

详见 [1.10 最长行全局版](../ch01-introduction/1.10-外部变量与作用域.md)。

### 4. 坏处：改的地方多，难追 bug

| 风险 | 说明 |
|------|------|
| **耦合高** | 任意函数都能改，**接口不清晰** |
| **难追踪** | 出 bug 时不知道 **谁最后写了** 这个全局 |
| **难测试** | 函数依赖隐式全局状态，单测要 mock 全局 |
| **线程安全** | 多线程同时改 → 数据竞争（内核/HFT 要特别小心） |
| **命名污染** | 全局名与局部名冲突时 **遮蔽** 规则易混 |

**工程原则**：

- **首选** 参数 + 返回值传递数据（[1.8](../ch01-introduction/1.8-参数传值调用.md)）
- 必须全局时：用 **`static` 限制本文件可见**（[4.6](./4.6-静态变量.md)），或封装 **访问函数**
- 大型项目：状态收进 **`struct` + 显式传指针**，少裸全局

### 5. 与局部自动变量对照

```c
void counter(void)
{
    int local = 0;           /* 每次调用重新创建，每次从 0 开始（若未赋别的） */
    local++;
}

int global_count = 0;        /* 只分配一次 */

void bump(void)
{
    global_count++;          /* 多次调用累加，值一直保留 */
}
```

- **局部 `auto`**：栈上，作用域块内，调用结束销毁（[2.4](../ch02-types-operators-expressions/2.4-声明.md)）
- **外部全局**：静态存储区，程序全程（除非 `static` 改链接属性，见 4.6）

### 6. 定义 vs `extern` 声明（再归纳）

| | **定义** | **`extern` 声明** |
|--|---------|-------------------|
| 写法 | `int bufpos = 0;` | `extern int bufpos;` |
| 内存 | **分配** | **不分配**，引用别处定义 |
| 次数 | 全程序 **一次** | 可 **多次** |

其他 `.c` 通过 **`extern`** + 头文件访问本文件定义的全局 —— 见 [4.4 作用域](./4.4-作用域规则.md)、[4.5 头文件](./4.5-头文件.md)。

**准则**：**Define once; declare many times**（[1.10](../ch01-introduction/1.10-外部变量与作用域.md)）。

### 7. `static` 限制外部变量：只能本 `.c` 用

**无 `static` 的全局变量**：其他源文件里 **`extern` 声明一下就能直接用**（通常经头文件 `#include`）。  
**`static` 全局变量**：**只能在定义它的那个 `.c` 里生效**；别的文件 **哪怕写 `extern` 也访问不到**。

这样区分的目的：**控制变量的可见范围，避免不同 `.c` 之间的命名冲突** —— 多个文件可以各自拥有 `static int count;`，互不干扰；若都叫无 `static` 的 `int count;`，链接器会报 **重复定义**。

默认外部变量（无 `static`）带 **外部链接** —— 他文件 `extern` 能访问。若在 **定义时** 加 **`static`**，则 **内部链接**：**仅定义所在的那一个 `.c` 可见**。

#### 无 `static`：跨文件共享

```c
/* === a.c：定义 === */
int num;                 /* 或 int num = 0; */

/* === b.c === */
extern int num;          /* 声明后可直接用 num */
void f(void) { num++; }

/* === c.c === */
extern int num;          /* b.c、c.c 访问的是 a.c 里同一份 num */
void g(void) { num--; }
```

链接：`gcc a.c b.c c.c` → **一份** `num` 内存，三文件共享。

#### 有 `static`：锁在本文件（防命名冲突）

```c
/* === a.c === */
static int num;          /* 仅 a.c 可见；符号不导出给链接器 */

void bump(void) { num++; }

/* === b.c === */
static int num;          /* ✅ 合法：另一份 num，与 a.c 的 static num 无关 */

extern int num;          /* ❌ 想引用 a.c 的 static num → 链接失败 */
```

| 场景 | 结果 |
|------|------|
| `a.c` 与 `b.c` 各 `static int helper_state;` | ✅ **允许** —— 各文件私有，**名字可重复** |
| `a.c` 与 `b.c` 各 `int packet_count;`（无 static） | ❌ **重复定义** —— 全程序只能一处定义 |
| `b.c` 对 `a.c` 的 `static int x` 写 `extern int x;` | ❌ **无效** —— 链接器找不到可绑定的定义 |

| 定义写法 | 链接属性 | 本文件 | 其他 `.c` + `extern` |
|----------|----------|--------|----------------------|
| `int num;` | **外部链接** | ✅ | ✅ 可 `extern` 共享 |
| `static int num;` | **内部链接** | ✅ | ❌ **访问不到** |

**作用**：把模块私有状态 **藏在 `.c` 里**，避免全局命名冲突、缩小可被乱改的入口 —— 见 [4.6](./4.6-静态变量.md)。

#### 两个 `.c` 都定义 `int num;` → 链接 **重复定义**

```c
/* a.c */
int num;          /* 定义 ① */

/* b.c */
int num;          /* 定义 ② —— 与 ① 冲突 */
```

编译各自 `.o` 可能通过，**链接** 时报 **`multiple definition of 'num'`**，程序 **过不了**。

C **没有** `public` / `private`，靠 **`static` + 链接属性** 控制谁能跨文件看见：

| 手段 | 效果 |
|------|------|
| 无 `static` 全局 | **外部链接** —— 全程序 **只能定义一次** |
| `static` 全局 | **内部链接** —— 每个 `.c` 可各有自己的 `static int num;`，互不冲突 |
| `extern` 声明 | 不分配内存，引用 **那一个** 外部定义 |

若 `a.c`、`b.c` **都想** 有自己的 `num`，又想让 `c.c` 的 `extern` 只指向 **某一个**：

```c
/* a.c：对外共享的那份 */
int num;

/* b.c：本文件私有的那份，与 a.c 的 num 无关 */
static int num;

/* c.c */
extern int num;    /* 只可能指向 a.c 的非 static num */
```

`extern` **永远** 找 **外部链接** 的那个定义；`static` 的那份 **链接器不导出**，`c.c` 看不见。

### 8. 规范做法：头文件 `extern`，**只在一个 `.c` 定义**

多文件工程 **推荐** 写法（[4.5 头文件 §2.1](./4.5-头文件.md) 有 **`global_num`** 完整 walkthrough：`b.c` `#include` 后直接 `printf("%d", global_num)`，改类型只动 `a.c` + `common.h`）：

```c
/* common.h */
#ifndef COMMON_H
#define COMMON_H
extern int num;      /* 声明：所有 .c 包含后都知道「有这么个 int」 */
#endif

/* a.c —— 唯一定义处 */
#include "common.h"
int num = 0;         /* 定义 + 初值；全程序仅此一处 */

/* b.c */
#include "common.h"
void f(void) { num++; }   /* 无需再写 extern，头文件已声明 */

/* c.c */
#include "common.h"
void g(void) { num--; }
```

| 规则 | 说明 |
|------|------|
| **`.h`** | 只放 `extern int num;`（**声明**） |
| **某一个 `.c`** | 只放 **一次** `int num = 0;`（**定义**） |
| **其他 `.c`** | `#include` 头文件即可，**不要** 再写 `int num;` 或重复 `extern` |

**改类型**：定义处（如 `a.c`）与 `.h` 里的 `extern` **两处** 同步改即可；凡 include 该头的 `.c` 自动一致，不必逐个改声明。

**禁止** 在头文件里写 `int num = 0;` —— 多个 `.c` include 会变成 **多份定义**，链接同样炸。

### 9. 函数间通信方式选型

| 方式 | 特点 |
|------|------|
| **参数 + 返回值** | 接口清晰，**首选** |
| **外部全局变量** | 简单，**耦合高、难维护** |
| **指针形参**（ch05） | 改调用方数据，接口仍较清晰 |
| **文件 I/O** | 进程间或持久化 |

## 易混点

| # | 要点 |
|---|------|
| 1 | **无 `static` 全局** = 外部变量（可跨文件 `extern`）；**`static` 全局** 在全局区但 **非** 跨文件外部变量 |
| 2 | **全程有效**，不像局部变量每次调用重建 |
| 3 | 未初始化全局 → **0**；局部 auto → **垃圾值** |
| 4 | 所有函数能访问 → **改的地方多**，bug 难追源头 |
| 5 | **多 `.c` 工程**：一处 **定义**，他处 **`extern`** → **同一块内存** 跨文件共享 |
| 6 | `a.c`、`b.c` 都 `int num;`（无 static）→ 链接 **重复定义** |
| 7 | `static` 全局 → 仅本 `.c`；他文件 `extern` **无效** |
| 8 | **规范**：`.h` 里 `extern`，**一个** `.c` 定义，其余 `#include` |

## HFT / 内核提示

- **Linux 内核**：大量全局（`jiffies`、子系统状态）—— 读驱动必先认全局
- **HFT**：全局行情队列、ring buffer 索引常见；热路径要 **原子/无锁** 或减全局写
- **飞控**：传感器融合结果、模式标志 —— 优先模块内 `static` + API，少公开可变全局
