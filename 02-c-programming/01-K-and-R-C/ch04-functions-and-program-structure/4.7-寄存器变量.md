# 4.7 寄存器变量

## 本节讲什么

**`register`** 修饰的 **局部变量**（含形参），**建议** 编译器把它放在 CPU **寄存器** 里而不是 **栈/内存**，以加快 **频繁读写**。现代编译器多自动分配寄存器，手写 `register` 已少见，但读 K&R、旧代码、理解「热变量」优化时仍会碰到。

**前置**：[4.1 函数](./4.1-函数的基本知识.md)、[4.3.1 内存布局 §1 栈区](./4.3.1-程序内存布局.md)、[2.4 声明](../ch02-types-operators-expressions/2.4-声明.md)。

## 要点

### 1. 什么是寄存器变量

**寄存器变量** = 用 **`register`** 关键字修饰的 **局部变量**（或 **形参**）：

```c
void sum_array(int n, register int *p)
{
    register int i;
    register int sum = 0;

    for (i = 0; i < n; i++)
        sum += p[i];
}
```

| 对比 | **普通局部 `int i`** | **`register int i`** |
|------|---------------------|----------------------|
| **典型存放** | **栈**（[4.3.1 §1](./4.3.1-程序内存布局.md)） | **尽量** 放 CPU **寄存器** |
| **访问速度** | 需访栈（可能再进缓存） | 寄存器读写 **更快** |
| **适用** | 一般局部量 | **频繁读写** 的循环计数器、累加器等 |
| **取地址 `&i`** | ✅ 合法 | ❌ **非法**（未必有内存地址） |

**意图**：把最热、最常用的变量放在 **离 CPU 最近** 的寄存器里，减少访栈/访内存次数。

### 2. 只是「建议」—— 编译器可以不照办

- **`register` 不是命令**，是 **给编译器的提示**（hint）
- CPU **寄存器数量有限**；一个函数里 `register` 太多，或全局优化需要寄存器时，编译器会 **忽略** 部分 `register`
- **寄存器不够** 时，变量 **退化成普通栈变量** —— 程序仍正确，只是未必更快

```c
void f(void)
{
    register int a, b, c, d, e, f, g;  /* 未必都能进寄存器 */
    /* 编译器按优化策略挑选，其余放栈 */
}
```

**现代实践**：`-O2`/`-O3` 下编译器 **自动** 做 **寄存器优化**（见 [4.7.1 §0](./4.7.1-volatile限定符.md)），往往 **不必** 手写 `register`；写了也可能被忽略。

> **C17** 起块作用域 `register` 已 **弃用**；**C23** 移除该存储类提示。K&R 时代语法仍大量出现在教材与旧代码中。

### 2.1 「寄存器优化」≠ 只指手写 `register`

口语里的 **寄存器优化**，多半指 **编译器自己的优化**，不全是程序员写 `register`：

| 编译器常做的事 | 目的 |
|----------------|------|
| 把 **频繁使用** 的局部变量 **放进 CPU 寄存器** | 少访栈/少访内存，函数 **跑得更快** |
| 某变量从内存 **只读一次**，之后在寄存器里 **反复用旧值** | 省掉重复 load |

```c
/* 源码 */
for (int i = 0; i < n; i++)
    sum += a[i];
/* 优化后概念上：i、sum 可能一直在寄存器里转 */

int x = g_flag;
foo(x);
bar(x);   /* 编译器可能不再读 g_flag，仍用寄存器里的 x */
```

手写 **`register`** 只是 **提示**「请优先把我放进寄存器」；真正干活的是 **编译器优化器**。与 **`volatile`** 的冲突见 [4.7.1](./4.7.1-volatile限定符.md)。

### 3. 不能取地址：`&` 与 `register` 互斥

寄存器里的值 **不一定** 对应某块固定内存地址，因此：

```c
register int i = 0;
int *p = &i;    /* ❌ 编译错误：不能对 register 变量取地址 */
```

| 原因 | 说明 |
|------|------|
| **可能没有内存地址** | 变量只在寄存器中，链接器/栈帧里没有稳定槽位 |
| **与指针语义冲突** | `&` 要求对象有 **可寻址** 的存储（ch05） |

**记忆**：需要 **传指针** 给别的函数修改、或 **`&` 取址** 的变量，**不要** 声明成 `register`。

### 4. 只能是局部变量 / 形参 —— 不能写在函数外

**`register` 只能修饰函数内部的局部变量，或函数的形参。** 不能用于文件顶层的全局变量。

```c
register int g;          /* ❌ 非法：不能在函数外 */

int main(void)
{
    register int i;      /* ✅ */
    ...
}
```

| 为什么全局不行 | 说明 |
|----------------|------|
| **寄存器与调用栈绑定** | 寄存器为 **当前函数执行** 服务；`return` 后寄存器内容 **不再保留** 给该变量 |
| **全局要长期存内存** | 全局/static 对象从 **程序启动到结束** 都在（[4.3](./4.3-外部变量.md)），必须落在 **全局/静态区** 或链接器管理的内存，**不能** 占着易失的 CPU 寄存器 |

**一句话**：`register` 变量本质是 **局部变量** —— 生命周期跟 **这一次函数调用** 走，和栈上 `auto` 局部同类；只是 **存放位置** 优先尝试寄存器而非栈。

### 5. 与 `auto`、`static` 局部对比

| 存储类 | 作用域 | 典型存放 | 函数返回后 | `&` 取址 |
|--------|--------|----------|------------|----------|
| **`auto`（默认）** | 块/函数 | **栈** | 销毁 | ✅ |
| **`register`** | 块/函数 | **寄存器**（或退化为栈） | 销毁 | ❌ |
| **`static`（函数内）** | 块/函数 | **静态区** | **保留值** | ✅ |

`register` **不改变作用域**，只影响 **尽量放哪、能否取址**；与 [4.6 `static` 局部](./4.6-静态变量.md)（跨调用保留）是不同维度的优化。

## 示例

```c
unsigned strlen_reg(register char *s)
{
    register char *p = s;
    while (*p)
        p++;
    return (unsigned)(p - s);
}
```

循环里 `p` 频繁自增、解引用 —— 典型 K&R 风格「建议放寄存器」的写法。

## 易混点

| # | 要点 |
|---|------|
| 1 | **`register` = 建议**，编译器可忽略；寄存器不够 → **普通栈变量** |
| 2 | **不能 `&`** —— 未必有内存地址 |
| 3 | **仅局部/形参**；**不能** 修饰全局 —— 全局要长期占内存 |
| 4 | 仍是 **局部变量**：函数结束即失效，与栈局部同类生命周期 |
| 5 | **`register` ≠ `volatile`**：`volatile` 禁止优化省略访问；语义相反 |
| 6 | 现代靠 **编译器优化** 多于手写 `register` |

## HFT / 内核提示

两个关键字在 **高频量化** 和 **内核** 里都有用，但 **分工不同**：

| 关键字 | 内核开发 | 高频量化（HFT） |
|--------|----------|-----------------|
| **`volatile`** | **必用**：硬件寄存器对应的 **MMIO 地址** 必须 `volatile`，否则编译器 **寄存器优化** 可能让你读不到硬件 **实时状态**（状态位、FIFO 空满等） | 用户态策略代码 **较少** 直接写；设备侧多在驱动/PMD；共享状态更常用 **`atomic`/内存屏障** |
| **`register`** | C 源码里 **几乎不写**；靠编译器 + 必要时 **汇编** | **可** 标在高频循环的 **计数变量** 上，**略** 提速；但 **`-O3` 下编译器常自动** 把合适变量放进寄存器，**手写往往多余** |

### 内核：`volatile` 是硬需求

```c
/* 设备寄存器映射 —— 必须 volatile */
volatile uint32_t * const uart_dr =
    (volatile uint32_t *)(uintptr_t)0x40001000u;

uint32_t status = *uart_dr;   /* 每次读都真访问硬件，不能用缓存旧值 */
```

- **Linux**：优先 `readl`/`writel`/`ioread32` 等 API（内部已带正确语义）
- **不用 `volatile`** 时，优化后的驱动可能 **卡在旧状态**，表现为 **轮询不到中断、DMA 不前进**

### HFT：`register` 可选，瓶颈 seldom 在这

```c
for (register int i = 0; i < n; i++)   /* 可写，但常不必写 */
    sum += book[i];
```

- 热路径更应关注：**缓存局部性**、**分支预测**、**少 syscall**、**对象池少 malloc**
- **DPDK**：寄存器访问走 PMD；策略循环靠 **编译器自动寄存器分配** + **SIMD/内联** 居多

**结论**：内核侧 **`volatile` 不能省**；HFT 侧 **`register` 可写可不写**，现代编译器 **优化能力很强**，自动寄存器分配往往已够。
