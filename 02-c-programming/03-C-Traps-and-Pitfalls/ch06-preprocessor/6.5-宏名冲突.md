# 6.5 宏与关键字、typedef、标准库名字冲突

## 本节讲什么

预处理 **先于** 编译；`#define` 会替换 **所有** 匹配的标识符 token，包括 typedef 名、**标准库函数名**、甚至关键字（若未防护），造成全工程类型/语法错乱或 **静默行为篡改**。

宏 **不管** C 块作用域 / 函数作用域，也不按 Rust 词法作用域规则 —— 从 `#define` 出现位置到文件末（或 `#undef`），**纯文本全文替换**。

## 要点

### 1. 篡改类型：`#define int long`

```c
#define int long
int a;           /* → long a;  整文件 int 语义被篡改 */
```

### 2. 劫持标准库：`#define getchar my_getchar`（HFT 致命隐藏 bug）

```c
#include <stdio.h>
#define getchar my_getchar

int c = getchar();   /* 预处理后 → int c = my_getchar(); */
```

- 源码里 **看起来** 调 `<stdio.h>` 的 `getchar`，编译 **可能仍通过**
- 实际走的是 `my_getchar`：缓冲、锁、syscall 次数、延迟全可能变
- **无报错、无 shadow 警告** —— 预处理器不做作用域分析
- HFT 热路径上：profile 对不上、replay 与实盘不一致，极难定位

`<stdio.h>` 中 `getchar` 等本身 **常为宏**；再 `#define` 一层等于 **二次劫持**。

### 3. 其他常见撞名

| 危险写法 | 后果 |
|----------|------|
| `#define max(a,b) ...` | 与 C++ `std::max`、内核 `max` 宏冲突 |
| `#define uint32_t my_u32` | 整文件类型语义崩坏 |
| `#define DEBUG` 空宏 | `#ifdef DEBUG` 误触发（见 6.7） |

### 4. 规范

- 宏名 **全大写** + 项目前缀：`HFT_RING_SIZE`、`MYDRV_MAX_BUF`
- **禁止** `#define` 标准关键字、常用 typedef（`uint32_t`、`size_t`）、**标准库函数名**（`getchar`、`malloc`、`printf`…）
- 取消误定义：`#undef getchar`（极少用，优先别定义）
- 排查：`gcc -E file.c | less`，或 `grep -r '#define getchar' include/`

### 5. 与链接衔接

[ch04](../ch04-linking/) 中 **链接期** 符号名冲突（重复定义 `foo`）；宏冲突发生在 **编译更早期** 的预处理，连类型检查都还没开始。
