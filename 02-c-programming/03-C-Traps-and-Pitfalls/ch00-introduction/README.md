# 第 0 章 导读

**Introduction** — Andrew Koenig, *C Traps and Pitfalls*

## 本章目标

说明 **C 语言为什么到处是隐藏陷阱**：C 诞生于早年小型机环境，为 **精简、灵活、贴近硬件** 牺牲了语法安全性——没有强制边界检查、没有自动内存管理、许多构造在标准里属于 **未定义行为（UB）** 却仍能编译通过。

建立 **三类错误** 的分级意识：

| 级别 | 表现 | 排查难度 |
|------|------|----------|
| **编译报错** | 语法/类型错误 | 低，编译器直接指出 |
| **运行崩溃** | 段错误、总线错误 | 中，gdb/valgrind 可定位 |
| **静默错误** | 程序「正常跑」但结果错 | **最高**，HFT、内核、驱动高频 |

理解 **全书逻辑链路**（由浅入深）：

```text
词法 → 语法 → 语义 → 连接 → 库函数 → 预处理器 → 可移植性 → 建议与答案
 ch01   ch02    ch03    ch04      ch05        ch06          ch07         ch08
```

编译器处理源码的第一步是 **词法分析（lex）**：把字符流拆成 **记号 token**。后续语法、语义、链接问题，很多根源在 **token 如何被切开**（见 **ch01**）。

## 为什么 C 特别容易踩坑

1. **历史包袱**：K&R 时代规则少，为兼容保留大量「看起来能写」的构造
2. **信任程序员**：数组无界、指针可算术、宏纯文本替换
3. **静默失败**：`=` 与 `==`、`unsigned` 回绕、未初始化变量——编译器常不警告
4. **实现定义/未定义**：同一源码不同编译器/优化等级结果不同（与 [Expert C ch08](../04-Expert-C-Programming/ch08-halloween-vs-christmas/) 序列点问题呼应）

## 底层开发适用场景

- **高频交易**：静默逻辑错误比崩溃更致命（分支颠倒、条件永真/永假）
- **内核/驱动**：注释屏蔽代码、寄存器字面量、地址比较——词法层错误无运行时提示
- **嵌入式**：八进制地址、`'0'` vs `"0"` 混淆导致硬件配置错误
