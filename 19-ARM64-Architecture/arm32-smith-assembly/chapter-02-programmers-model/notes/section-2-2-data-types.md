## §2.2 数据类型 — Byte · Halfword · Word · 标量与浮点

> **Ch 2 · 程序员模型** · [章导读](../README.md)

---

### 核心先分清：「32 位」≠「内存格子是 32 位」

**一句话：** 32 位 ARM = **通用寄存器 32 bit** + **地址常 32 bit**；内存最小单元 **永远是 1 字节（8 bit）**，不是 32 位。

#### ① ARM32 两点（运算块 = 寄存器宽）

1. **CPU 单次通用运算标准粒度 = 32 bit** — 加减、逻辑、普通寄存器运算，默认一块就是 32 比特。  
2. **通用寄存器最大宽度 = 32 bit** — `r0`–`r15` 每个最多存 32 比特；**放不下超过 32 位的数** 时，用 **两个寄存器拼接**（惯例：低 32 位在 `r0`、高 32 位在 `r1`，如 `long long` / `UMULL` 结果）。

| 概念 | 是什么 | ARM32 上多大 |
|------|--------|----------------|
| **寄存器位宽** | 临时容器；也是 ALU 一次算多宽 | **r0–r15 = 32 bit = 4 字节** |
| **地址位宽** | 给内存编号 → 寻址上限 | 典型 **32 bit** → \(2^{32}\) ≈ **4 GB** |
| **内存存储单元** | 每个地址对应的最小格子 | **永远 1 字节（8 bit）** — 与 32/64 无关 |

```
地址 0x00000000 → 1 字节
地址 0x00000001 → 下 1 字节
……连续 4 个地址拼起来 → 才是一个 32 位「字」进 r0
```

**不要说「32 位内存」** — 「32 位」指 **寄存器/运算块（+ 常含地址宽）**，不是每个地址存 32 bit。

```armasm
ldr  r0, [r1]   @ 读连续 4 字节 → 32 位数进 r0
ldrb r0, [r1]   @ 只读 1 字节到 r0 低 8 位
```

#### ② 「几位 CPU」怎么判定？

口语里 **「32 位 / 64 位 CPU」** 的核心标准：

> **通用寄存器位宽 = CPU 通用整数运算单次处理的数据宽度。**

| 架构 | 通用寄存器位宽 | 单次标准运算块 | 单寄存器最大比特 |
|------|----------------|----------------|------------------|
| **ARM32（AArch32）** | 32 | 32 | 32 |
| **x86-32（IA-32）** | 32 | 32 | 32 |
| **x86-64** | 64 | 64 | 64 |
| **ARM64（AArch64）** | 64 | 64 | 64 |

→ 64 位分段细节（`rax`/`eax`/`ax`/`al`）：[CSAPP §3.3](../../../../01-CSAPP-3rd/chapter-03-machine-level-programs/notes/section-3.3-3.4-数据格式与访问信息.md)

---

### 32 位架构的「自然字长」

| 对象 | 宽度 |
|------|------|
| **基础整数标量上限** | **32 bit（1 word）** |
| **通用寄存器 r0–r12**（及 r13–r15） | **32 bit** |
| **典型 ALU / `LDR`/`STR` 主通路** | **32 bit** |

**口述：** 开发者默认 **「一个 r 寄存器 = 一个字的数据」**；更窄类型用 **`LDRB`/`LDRH`** 读入后再 **零扩展/符号扩展** 到 32 bit 参与运算（→ **Ch5**）。

---

### 三种基本长度（byte / halfword / word）

| 类型 | 一次访问占多少 **bit** | 占多少 **byte** | 典型 C 对应 |
|------|------------------------|-----------------|-------------|
| **Byte（字节）** | **8 bit** | **1 byte** | `char` / `uint8_t` |
| **Halfword（半字）** | **16 bit** | **2 byte** | `short` / `uint16_t` |
| **Word（字）** | **32 bit** | **4 byte** | `int` / `uint32_t` · 指针 |

**易混点（口述必清）：** 说「**8 位字节**」= **1 个 byte** = **8 个 bit** — **不是 8 个 byte**。  
「8 / 16 / 32」在 ARM 书里通常指 **bit 宽度**，不是 byte 个数。

**Ch1 回顾：** 内存与寄存器里都是比特；类型只约定 **读多少位、如何解释**（有符号/无符号 → [Ch1 §1.5](../../chapter-01-overview-computing-systems/notes/section-1-5-representation.md)）。

---

### 开发者标量命名：s8/u8 … s32/u32

Smith / 嵌入式资料常用 **`s` = signed、`u` = unsigned** 前缀 + **bit 宽度**：

| 类型 | bit | byte | 存储粒度 | C 典型 | 整数 Load/Store |
|------|-----|------|----------|--------|-----------------|
| **s8** | 8 | 1 | 字节 | `int8_t` | `LDRSB` / `LDRB` + 扩展 |
| **u8** | 8 | 1 | 字节 | `uint8_t` | `LDRB` |
| **s16** | 16 | 2 | 半字 | `int16_t` | `LDRSH` / `LDRH` + 扩展 |
| **u16** | 16 | 2 | 半字 | `uint16_t` | `LDRH` |
| **s32** | 32 | 4 | 字 | `int32_t` | `LDR` |
| **u32** | 32 | 4 | 字 | `uint32_t` | `LDR` |

**要点：**

- **s/u 不改变占用的 byte 数** — 同 **8/16/32 bit 比特**，解释不同（补码 vs 无符号，→ [Ch1 §1.5](../../chapter-01-overview-computing-systems/notes/section-1-5-representation.md)）。
- **对齐规则看访问宽度，不看 s/u** — `u32` 与 `s32` 一样要 **4 字节对齐** 才能 `LDR`。
- **Q 定点**（[Ch7 §7.7](../../chapter-07-integer-logic-arithmetic/notes/section-7-7-fractional.md)）常把 **s32** 解释成 **[-1, 1)** 小数 — 仍是 **32 bit 字**，无 FPU 时的 **f32 替代**。

---

### 对齐 (Alignment)

**字对齐（Word alignment）口述：** 32 bit 的 **字** 用 **`LDR`/`STR`** 访问时，**起始地址必须是 4 的倍数** — 如 **`0x0000`、`0x0004`、`0x0008`**；**不能**放在 **`0x0001`、`0x0002`、`0x0003`**（ARM7 上常 **Data Abort**）。

| 访问宽度 | 对齐要求 | 合法地址例 | 非法地址例 |
|----------|----------|------------|------------|
| **字节** | 无 | `0x0001` 任意 | — |
| **半字（16 bit）** | **2 的倍数**（偶地址） | `0x0000`、`0x0002`、`0x0006` | `0x0001`、`0x0003` |
| **字（32 bit）** | **4 的倍数**（地址低 2 bit = 0） | `0x0000`、`0x0004`、`0x0008` | `0x0001`、`0x0002`、`0x0003` |

**关系：** **半字 = 字的一半**（2 byte / 16 bit vs 4 byte / 32 bit）。  
**规定原因：** 匹配 CPU **单次处理宽度** — **`LDRH` 一次 16 bit、`LDR` 一次 32 bit**，对齐则 **一条指令读满**，非对齐则 **多次访问或 fault**。

**ARM7TDMI：** 半字/字 **非对齐** → 常触发 **Data Abort**。  
**Cortex-M4：** 部分配置下可容忍非对齐访问（`CCR` 等），仍建议 **默认按上表对齐写**。

### 为何要对齐（效率逻辑）

32 位 CPU 一次总线/Load 操作目标是 **32 bit（4 byte）**；内存 **每个地址 = 1 byte**。

```
字对齐地址 0x0004：
  0x0004 │ 0x0005 │ 0x0006 │ 0x0007  →  一条 LDR 进寄存器（完整一字）

非对齐 0x0003：
  0x0003 │ 0x0004 │ 0x0005 │ 0x0006  →  硬件要么拒绝（Abort），要么拆成多次访问（慢）
```

**口述：** 从 **4 的倍数** 起读，**一次** 拉满 32 bit 进寄存器；非对齐 = 零散、多周期或 fault — 所以 ARM7 **硬要求对齐**，M4 虽可放宽，**习惯仍按对齐写**。

**与 C 对照：** [02 C](../../../../02-c-programming/) 里结构体 `padding` 就是为对齐；`uint32_t *p = (uint32_t*)0x1001` 在严格 ARM7 上可能 fault。

### 内存地址存放示例（Smith §2.2 思路 + 权衡）

原书 §2.2 会用 **具体地址** 说明：何种 **`LDR`/`LDRH`/`LDRB`** 合法、何种会 **Abort**（与上表 `0x0004` / `0x0003` 同类）。通常 **不会** 单独算「char 浪费 75%」—— 那是把 **访问对齐** 和 **变量占坑** 合在一起时的推论；下面补上这一层，方便和 C 结构体对照。

**关键区分：**

| 概念 | 含义 |
|------|------|
| **存放（存储）** | `char` **占 1 byte**，可以紧挨着放 — **不要求** 每个变量都从 4 的倍数起 |
| **访问（Load/Store）** | 用 **`LDR` 读 32 bit** 时，**起始地址** 必须 4 对齐；用 **`LDRB`** 读字节则 **任意地址** |

**紧凑存放示例**（字节可连续，不浪费）：

```
地址     内容              说明
0x1000   'A'  (char)       LDRB [0x1000] ✓
0x1001   'B'  (char)       LDRB [0x1001] ✓
0x1002   'C'  (char)       LDRB [0x1002] ✓
0x1003   (填充/padding)    若下一项是 int，编译器常留 1 字节 pad
0x1004   0x12345678 (int)  LDR  [0x1004] ✓  ← 字从 4 的倍数起
```

三个 `char` 只占 **3 byte**；`int` 从 **`0x1004`** 做 **`LDR`** — **效率与密度兼顾**。

**若错误地「所有数据都按 4 byte 槽位存 char」**（仅作对比，实际不会这样设计）：

```
0x1000   'A'   + 3 byte 空洞
0x1004   'B'   + 3 byte 空洞
0x1008   'C'   + 3 byte 空洞
→ 3 个 char 占 12 byte，有效数据 3 byte ≈ 利用率 25%（浪费约 75%）
```

**口述：** ARM 按 **访问宽度** 定对齐，不是「每个 char 变量都必须占 4 byte」；**`LDRB` 让字节能密排，`LDR` 让字能一次读满** — 正是你说的 **权衡**。

**Smith 后续：** [Ch5 §5.1](../chapter-05-loads-stores-addressing/) 用 **`LDRB/H/`STR`** 在 **连续地址** 上操作；[Ch16](../chapter-16-memory-mapped-peripherals/) DAC 用 **`STRH`** 写 16 bit 寄存器域 — 宽度与对齐跟硬件手册一致。

### 浮点类型：f32 · f64（VFP / M4F）

**§2.2 正文** 以 **整数 byte/halfword/word** 为主；**浮点不在 Ch2 展开**，而在 **Ch9–11**（本仓库 [OUTLINE](../../OUTLINE.md) 标 **跳过**，但类型映射在此写全）。

| 类型 | bit | byte | 内存布局 | 计算寄存器 | 全书位置 |
|------|-----|------|----------|------------|----------|
| **f32** | 32 | 4 | **1 word**，4 字节对齐 | **s0–s31**（单精度） | [Ch9 §9.4](../../chapter-09-floating-point-basics/notes/section-9-4-data-types.md) |
| **f64** | 64 | 8 | **双字**，起始 **4 对齐** | **d0–d15**（= 两 s 拼 64 bit） | 同上 · [§9.7](../../chapter-09-floating-point-basics/notes/section-9-7-fp-registers.md) |

**和整数的关系：**

```
内存里 f32 占 4 byte — 和 u32/s32 同宽
        ↓
Load：VLDR s0, [addr]     （不是 LDR r0 — 浮点在 FPU 银行）
        ↓
运算：VADD.F s0, s0, s1   （Ch11）
        ↓
与整数互转：VCVT（Ch9 §9.11）— Ch18 可封装成 C 库函数
```

| | **整数 s32/u32** | **f32** |
|---|------------------|---------|
| 比特宽度 | 32 | 32 |
| 解释 | 补码/无符号整数 | **IEEE 754** 符号+指数+尾数 |
| 主寄存器 | **r0–r12** | **s0–s31** |
| 内存指令 | `LDR`/`STR` | **`VLDR`/`VSTR`**（需 **CPACR 开 FPU**，Ch9 §9.8） |

**f64 / VFP：**

- **Cortex-M4F**：**f32 硬件算** 为主；**f64** 常 **软件 libm** 或弱支持 — **d*** 寄存器是 **s 的 64 bit 视图**（s0+s1 → d0）。
- **ARM7 + 外部 VFP** 或 **Cortex-A**：**f64 双精度** 更完整 — 仍占 **8 byte**，`VLDR` 进 **d***。
- **无 FPU**：浮点 **Undefined 陷阱 + 软件仿真**（[Ch14 §14.3](../../chapter-14-exception-handling-arm7tdmi/notes/section-14-3-error-conditions.md)）或 **整数 Q / 查表**（Ch7/Ch12/Ch16 DAC）。

**Ch2 §2.4 预告：** M4F 除 **r0–r15** 外另有 **s0–s31** — 见 [section-2-4-cortex-m4.md](./section-2-4-cortex-m4.md)。  
**Ch3** 五个示例之一含 **浮点寄存器传数**（热身，非系统讲 IEEE）。

**阅读路径：**

| 目标 | 读哪 |
|------|------|
| 只懂 **类型宽度对应** | 本节表格即可 |
| 抠 **f32 位域 / VLDR / VCVT** | **Ch9**（精读或选读） |
| 飞控/Linux 应用层 | **C `float`/`double` + libm**，Ch9–11 可压缩 |

### Load/Store 指令关联（预告 Ch5）

- **`LDRB` / `STRB`** — 字节  
- **`LDRH` / `STRH`** — 半字  
- **`LDR` / `STR`** — 字  

**汇编习惯：** 宽度选对指令，比「用字访问再移位掩码」更清晰；MMIO 必须按硬件手册宽度访问（→ **Ch16**）。

---

### 可复述要点

1. **「32 位」= 寄存器宽 + 运算块（+ 常含地址宽）**；内存格子永远 **1 字节** — 别说「32 位内存」。  
2. **64 位整数在 ARM32 上要两个 r 寄存器拼接**；判定「几位 CPU」看 **GPR/通用运算宽度**。  
3. **s8/u8 … s32/u32** 映射 **byte / halfword / word**；**s/u 只改解释，不改宽度**。  
4. **32 bit = 字 = r 寄存器自然宽度**；更窄用 **`LDRB/H`** + 扩展；`LDR` 一次拉连续 4 字节。  
5. **f32** 内存占 **4 byte**，算在 **s***；**f64** 占 **8 byte**，算在 **d***（要 VFP/M4F）。  
6. **对齐** 按 **访问指令宽度**；char 可密排，**`LDR` 字访问** 须 4 对齐。  
7. 浮点细节 → **Ch9–11**；无 FPU → **Q 定点 / 查表**（Ch7/12/16）。
