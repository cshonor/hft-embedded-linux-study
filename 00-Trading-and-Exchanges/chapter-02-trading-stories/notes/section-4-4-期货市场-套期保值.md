# Ch 2 §4 期货市场：套期保值 · A Futures Market Trade

> **Trading and Exchanges** · Larry Harris · **精读**

## 4. 期货市场：套期保值 (A Futures Market Trade)

Harris 用 **大豆加工商** 在 **CBOT** 的故事，说明 **效用型交易 (utilitarian trading)**：不是为了赌方向，而是为了 **把商业风险转出去**。

---

## 故事在讲什么

加工商 **未来要在现货市场买进大豆**（或已持有待加工的库存）→ 怕 **大豆涨价** 侵蚀利润。

| 腿 | 市场 | 典型动作 | 价格涨时 |
|----|------|----------|----------|
| **现货敞口** | 物理/现货 | 将来 **买入** 大豆 | 采购成本 **↑**（亏） |
| **期货对冲** | CBOT 大豆期货 | **卖出** 期货 | 空头 **浮亏** |

两笔交易 **不在同一个 pit、同一个订单簿**，但 **经济效果** 是：涨跌大致 **相互抵消** → **套期保值 (Hedging)**。

**场内细节（历史）：**

- **Pit**：喊价 (Shouting) + 手势（手心向外 = 卖，向内 = 买）
- **清算**：**BOTCC** 作 **中央对手方 (CCP)**，担保履约 + 管理 **保证金 (margin)**

→ 今天 Pit 已被 **CME Globex** 等电子系统取代；**中央清算 + 保证金** 仍是期货核心。

---

## 套期保值的逻辑（三要素匹配）

套保的 **数学意图** 很清晰：

> **一个市场的盈利覆盖另一个市场的亏损** —— 只要 **头寸数量、方向、期限（合约/交割月）** 完全匹配，无论标的 **涨价还是跌价**，组合 **净值** 都锁在 **目标区间** 附近。

| 要素 | 示例（加工商） |
|------|----------------|
| **头寸** | 现货要买 1000 吨 ≈ 期货卖 N 手（按合约乘数换算） |
| **方向** | 现货 **多头敞口** ↔ 期货 **空头** |
| **期限** | 期货合约 **交割月** 贴近 **实际采购/加工** 时间 |

不匹配时，剩下来的是 **基差风险 (basis risk)** 或 **比例错配** —— 仍是 hedge，只是 **锁得不干净**。

**基差 (Basis)** = 现货价 − 期货价。对冲 **近似** 锁价，不是两价永远相等。

---

## 核心问题：同步性 (Synchrony)

**理论成立** 不等于 **执行瞬间已 hedge**。

Harris 故事里，加工商可以 **计划好** 先卖期货、三个月后再买现货 —— 这是 **商业节奏上的分时对冲**，中间窗口 **公司 knowingly 承担**。

但 **电子平台上一键套保** 的场景不同：用户点击「对冲」的 **那一刻**，若只有一腿成交：

```
t₀  现货买入 ✅（大豆多头已上身）
t₁  还没来得及在期货做空…
    价格暴跌 → 现货腿裸亏 —— 这就是「敞口窗口风险」
```

| 原因 | 说明 |
|------|------|
| **两本独立订单簿** | 现货、期货 **各自撮合** |
| **流动性不对称** | 一腿深、一腿薄 → 先成快腿、慢腿挂单 |
| **撮合速度差** | 一所在 colo、一所在公网 —— 毫秒级就是风险 |

所以：**对冲的「稳态逻辑」**（matched legs → 涨跌相抵）与 **「建仓瞬间是否已对冲」** 是两层问题。**后者** 才是撮合引擎要解决的 **同步性**。

| 层次 | 是否要求同步 |
|------|--------------|
| **经济学稳态** | 两腿最终都到位即可；允许 **计划内** 分时（加工商） |
| **平台即时套保** | 必须 **消灭或极缩短** 单腿窗口 —— 否则用户 **名义上在套保、实际上在裸奔** |

---

## 两个市场：为什么「知道要对冲」还不够？

现货市场与衍生品市场是 **独立 venue**：

| | 现货 / 物理 | 期货 |
|---|-------------|------|
| 订单簿 | 各自撮合 | 各自撮合 |
| 参与者 | 贸易商、加工商… | 投机者、套保者、HFT… |
| 清算 | 交货、信用条款 | **CCP + 逐日盯市保证金** |

**系统不会自动** 帮你在卖完期货后「补上」现货买入。只成交一边时，你得到的不是 hedge，而是 **单边赌注**：

| 只成交了… | 实际头寸 | 风险 |
|-----------|----------|------|
| 现货买入 ✅，期货卖出 ❌ | **纯做多** 现货 | 仍在赌涨价 — **没对冲** |
| 现货买入 ❌，期货卖出 ✅ | **纯做空** 期货 | 赌跌 + **保证金/追保** — 往往更急 |

所以：**对冲是经济目标；「建仓瞬间是否已配对」是同步性问题。**

---

## 引擎设计：一键双挂 · 原子组合单

要对 **即时套保** 真正负责，不能把两腿当成用户 **两次独立点击** 就完事 —— 必须 **绑定成一个原子操作**：

```
用户发起：一键双挂（现货买 + 期货卖，数量/期限已配好）
           │
           ▼
    ┌──────────────────┐
    │  Spread / Combo  │  要么两腿都成交
    │  Matcher         │  要么都不成交（全滚回）
    └──────────────────┘
           │
     ┌─────┴─────┐
  现货簿       期货簿
```

| 能力 | 作用 |
|------|------|
| **一键双挂** | UI/API 一次提交 **已配好比例** 的两腿 |
| **原子性 (atomic)** | **all-or-nothing** — 杜绝「一腿 orphan」 |
| **预检** | 下单前同时验：**现货资金/库存** + **期货保证金** |
| **组合风控** | 按 **净 delta** 算风险，而非单腿裸 futures |

这才符合用户心里「我已经套保了」的语义 —— **不是** 理论上说「迟早会对冲」，而是 **成交回报里就没有单腿窗口**。

Harris 加工商那种 **分月、分步** 套保，可以走 **普通单 + 人工/算法排程**；**平台型一键 hedge** 必须走 **组合单**。

## 与 go-dex 撮合引擎的对照

当前 [go-dex](../../00-practice-go-dex/code/orderbook.go) 是 **单交易对、单簿** 的 LOB — 对应 Harris 故事里 **「一个 pit / 一个合约」** 的一次撮合。

| Harris / 真实世界 | go-dex 现状 | 若做「现货 + 永续」套保 |
|-------------------|-------------|-------------------------|
| 现货簿 + 期货簿 **两个 engine** | 一个 `OrderBook` | 两个 `OrderBook`（两 symbol） |
| 加工商 **计划内分时** 对冲 | 两次独立 `AddOrder` | 适合 **老练机构**；窗口风险 **自担** |
| **一键双挂 / 组合单** | 未实现 | **`SpreadMatcher`**：原子两腿 or 全不成交 |
| **保证金** 只盯期货 | 未实现 | **RiskEngine**：组合预占 + 净 delta |

**读法：**

1. **Ch 2 §4** — 三要素匹配 + **同步性** 才是引擎题眼  
2. **Ch 2 §1 + go-dex `Match()`** — 单簿原子撮合是 **组合单的原子单元**  
3. **M4+ 扩展** — 在 **两簿之上** 加 **SpreadMatcher**，不是简单再写一个 `OrderBook`

→ leg risk / 部分成交：[Ch 17 §3.3](../../chapter-17-arbitrageurs/notes/section-3-3-四大核心风险.md)

## 敞口时间线：分时 vs 原子

**计划内分时（Harris 加工商）** — 窗口 **已知、可管理**：

```
时间 ──────────────────────────────────────────────→
     │ 卖期货 ( hedge )          │  后日现货采购
     ▼                           ▼
     ┌──┐                        ┌──┐
     │  │ 公司接受这段 basis 风险  │  │ 最终净敞口 ≈ 0
     └──┘                        └──┘
```

**即时套保（平台一键双挂）** — 窗口 **必须 ≈ 0**：

```
错误：  现货买 ✅ ──[价格可动]──> 期货卖 …
正确：  [现货买 + 期货卖] 原子成交 ──> 回报时已 matched
```

**Takeaway：**

| 场景 | 同步性要求 |
|------|------------|
| 商业 **排程型** 套保 | 低 — 最终三要素匹配即可 |
| 平台 **即时型** 套保 | 高 — **一键双挂 + 原子成交** 才能实现用户要的保值 |
