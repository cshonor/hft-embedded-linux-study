## 4. 贯穿全书的关键主题

Harris 说这些主题 **全书反复出现**。下面用 **go-dex 撮合引擎（造池子）** 和 **HFT（捕鱼）** 各举一例——你写代码时能直接摸到。

---

### 4.1 信息不对称 (Information asymmetries)

**书上：** 掌握更多价值信息或他人交易意图的人，相对不知情交易者有优势。

| 场景 | 例子 |
|------|------|
| **做市 / 深度接口** | 你接交易所 **深度 WebSocket**，比普通用户 **早 ~10ms** 看到卖一突然堆了 5000 手 → 提前调价或撤单 |
| **go-dex（引擎侧）** | **行情推送必须公平**——不能给 VIP 更快 feed；否则等于引擎帮 HFT **制造信息不对称**（§3 信息同步） |

---

### 4.2 期权与外部性 (Options and Externalities)

**书上：** 挂 **限价单** ≈ 给别人一个「随时按你的价成交」的 **免费期权**——对方 **不用付权利金**；你承担了 **外部性**（额外风险），市场 **不会额外补偿你**。

#### 摸得着的例子：10000 美元买 BTC 的限价单

你在 **10000 美元** 挂一笔 **限价买单**（买 BTC）：

| 概念 | 这是什么意思 |
|------|--------------|
| **免费期权** | 等于给全市场开了一张期权：**任何人** 都可以 **立刻按 10000 美元把 BTC 卖给你**，不用付手续费或权利金 |
| **外部性（你扛的风险）** | 单挂出去之后，行情会变，但 **你的价已经钉在簿上**——风险由你承担，不是由交易所或对手补偿 |

**外部性分两种方向（别搞混买/卖）：**

| 你挂的单 | 免费期权给了谁什么权利 | 典型「踩单」外部性 |
|----------|------------------------|-------------------|
| **限价买 @ 10000** | 任何人可随时 **按 10000 卖给你** | **利空**：价被打到 10000 吃满你的买单，然后 **继续下跌** — 你买在「还要跌」的路上 |
| **限价卖 @ 10000** | 任何人可随时 **按 10000 向你买** | **利好**：BTC 涨到 10500，大单仍可 **按 10000 买走** 你的货 — 你 **便宜卖了**，本可卖更高 |

> 口述里「利好涨到 10500 还得按 10000 **卖**」对应的是 **限价卖单** 被成交；**限价买单** 在利好时通常 **挂着不成交**，痛点在 **利空被吃单后继续跌**。

#### 撮合引擎里的成交条件（写 `Match()` 就按这个）

| 订单类型 | 挂在簿上 | 什么时候成交 |
|----------|----------|--------------|
| **限价卖 @ P** | Asks 的 P 档 |  incoming **买单** 的价格 **≥ P**（含 **市价买单** 从最低卖价一路 **扫卖盘** 扫上来） |
| **限价买 @ P** | Bids 的 P 档 |  incoming **卖单** 的价格 **≤ P**（含 **市价卖单** 从最高买价一路 **扫买盘** 扫下去） |

**要点：** 限价卖 **不需要** 把「市价维持在 10500」——只要来一笔 **足够高的市价买单**，按 **价格优先** 从卖一开始扫，**自然会扫到你 10500 那一档**。你在 go-dex 里实现的 walk-the-book，就是这套逻辑。

#### 被踩单完整场景：10500 限价卖 @ 市价 10000

1. 你在 **10500** 挂 **限价卖**，打算涨到了再出手。  
2. 大资金 **小单慢慢买**，把 Last Price 托到 **10490** — 让你以为马上要到 10500。  
3. 很多挂在 10500 卖的人 **撤单改挂 10600**；市场上 **10500 只剩你这一张卖单**。  
4. 大资金下一笔 **市价买单（最高愿付 10600）** — 引擎从 **卖一 → 卖二 → …** 依次吃 Asks，**扫到 10500 时按 10500 成交你的单**（中间怎么波动无所谓，只要 sweep 路径经过你的价位）。  
5. 大资金的 **吃单均价** 可能接近 10600（扫了多档），但它要的是 **你这张确定能成交的挂单**；转头可以在 **10490 以上** 再卖/对冲，赚的是 **你提供「在 10500 一定卖」的确定性** 与自身布局之间的差价。

```
小单托价 10490  →  别人撤 10500 改 10600  →  只剩你在 10500
       →  市价买扫到 10600  →  walk-the-book 必经过 10500  →  你的卖单被吃
```

**一句话：** 大资金 **不必** 真把价格「停」在 10500；**一笔更高价的市价单扫穿卖盘**，就能在 **你的固定价位** 完成收割。

**场景 B（镜像）：9900 限价买**

- 成交条件：incoming 卖价 **≤ 9900**（市价卖单 **扫买盘** 扫下来）。  
- 大资金 **砸盘触发你的买单 → 再拉涨**；逻辑与上面对称。

**本质：**

| | 你 | 大资金 |
|---|-----|--------|
| 单子 | **固定价位** 挂在簿上 | **主动单 + 推价** 控制 sweep 路径 |
| 角色 | 免费期权的卖方 / 确定性对手盘 | 用市价 sweep **精确吃你这一档** |

| 场景 | 例子 |
|------|------|
| **go-dex** | `Match()` 对市价单：**从最优对手档开始，逐档成交直到数量满足或扫完** — 规则写对了，上面场景才会在代码里 **可复现** |
| **你以后做市/HFT** | 挂单 = 暴露 **确定性**；别人可用 **市价 sweep** 专门吃你 — 想不想留在簿上、要不要快速撤单，是策略问题（§2 折页：手续费 / Maker 返佣） |

### 4.3 市场结构与竞争 (Market structure & Competition)

**书上：** 交易规则、费用、信息系统决定 **谁赚谁亏**；规则一改，利润分配就变。

| 场景 | 例子 |
|------|------|
| **Maker–Taker** | 交易所 **挂单返佣、吃单扣费** → 做市商拼命 **抢最优挂单位置**（卖一/买一队列前排）才能赚返佣 + 价差 |
| **go-dex（待做 M3）** | 你在引擎里实现 **费率表** = 你在 **设计市场结构**；费率设计错了，没人愿意挂单，池子干涸 |

---

### 4.4 通信与计算技术 (Communications and computing technologies)

**书上：** 市场是 **信息处理机制**；技术变，交易方式变，经济学原理不变。

| 场景 | 例子 |
|------|------|
| **HFT 基础设施** | 网卡 **硬中断**、**DPDK** 绕开内核协议栈 → 少 **几微秒** 延迟，抢先占最优价（§3：抢跑不是插队） |
| **go-dex（现阶段）** | 先用 Go 把 **撮合逻辑写对**；微秒级优化是 **HFT 层** 的事——但要知道：HFT 赚的部分利润，就是 **技术差距** 换出来的 |

---

### 4.5 价格相关性 (Price correlations)

**书上：** 相似工具、相关市场 **抢同一批订单流**，彼此联动。

| 场景 | 例子 |
|------|------|
| **跨币种套利** | BTC 急涨 → ETH 往往跟涨 → HFT **立刻联动改 ETH 挂单**，否则按旧价卖 ETH 会被 arbitrageur 吃光 |
| **go-dex（以后 M4+）** | 多 Symbol 多个 `Orderbook`；**价格联动是策略**，引擎只提供 **每个池子各自公平撮合** |

---

### 4.6 委托代理问题 (Principal-agent problems)

**书上：** 代理人（经纪、算法）有时按 **自己利益** 行事，而非委托人最优。

| 场景 | 例子 |
|------|------|
| **HFT 实盘** | 你写的策略说「最优价成交」，实盘执行模块却 **偷偷滑点**、没按设定价挂单 → **代码即代理人**，你要审计执行路径 |
| **go-dex** | 用户通过 API 下单，引擎 **不能** 暗改价格/数量；否则就是交易所版的 **代理背叛** |

---

### 4.7 可信度与信用度 (Trustworthiness and creditworthiness)

**书上：** 市场要让人相信 **对手会履约**、机构 **不会耍赖**。

| 场景 | 例子 |
|------|------|
| **交易所公信力** | **不能篡改订单簿**、**不能插单**、不能事后改成交记录 → 否则没人敢挂单 |
| **go-dex** | `Match()` 必须 **确定性、可复现**；成交日志可审计；推送与簿状态 **一致** — 这是在造 **可信度** |

---

### 4.8 零和博弈 (The zero-sum game)

**书上：** 有人赚，必有人亏；要问 **对手为什么愿意亏**。

| 场景 | 例子 |
|------|------|
| **HFT 赚价差** | 你早 10ms 在卖一成交 → 赚的每一分，往往来自 **慢半拍** 的市价买家多付的价 |
| **go-dex** | 引擎 **不创造也不销毁钱**——只 **重新分配**；五维里的 **交易利润** 在参与者之间切蛋糕，池子只负责 **切法公平**（§2） |

---

### 小结：八个主题 ↔ 你现在在哪一层

| 主题 | go-dex（基础设施） | HFT（以后） |
|------|---------------------|-------------|
| 信息不对称 | 公平推送、不偷跑行情 | 合法地 **读得快** |
| 期权与外部性 | 规则让限价单风险可预期 | 决定是否做市、挂哪档 |
| 市场结构 | Maker–Taker、优先级 | 在规则里 **抢位置** |
| 通信与计算 | 逻辑正确优先 | 微秒级优化 |
| 价格相关性 | 多 Symbol 独立撮合 | 跨池联动策略 |
| 代理问题 | 引擎不篡改订单 | 审计实盘执行 |
| 可信度 | 簿不可篡改、可审计 | 选可信 venue |
| 零和博弈 | 公平切蛋糕 | 做快的那一刀 |
