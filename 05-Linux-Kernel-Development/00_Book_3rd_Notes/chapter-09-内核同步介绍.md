### ① 临界区与竞态条件

**临界区（critical region / critical section）** = 访问、修改 **共享数据** 的代码段。

| 要求 | 含义 |
|------|------|
| **原子执行** | 如同 **一条不可分割指令** — 执行期间 **不被并发打断** |
| **同步（synchronization）** | 保证安全并发、消除不一致的过程 |

**竞态条件（race condition）** = 两个（或更多）执行线程 **可能同时** 进入 **同一临界区** 操作共享数据。

```
线程 A ──► 读 count=5 ──► +1 ──► 写 6
线程 B ──► 读 count=5 ──► +1 ──► 写 6   ← 应为 7，竞态丢失更新
```

**HFT 对照：** 用户态 **无锁队列、原子计数** 与内核 **「临界区必须互斥」** 是同一问题的两层 — tail 延迟常来自 **锁争用** 或 **伪共享**。

→ [02 SysPerf §5.2 mutex/spin](../../02-Systems-Performance-2nd/chapter-05-applications/notes/section-5.2-应用程序性能提升技术.md)

### ② 加锁 · Locking

多指令、复杂结构 — 靠 **锁** 保证临界区 **同一时刻只有一个线程** 进入。

| 比喻 | 说明 |
|------|------|
| **房间门锁** | 一次一人进房改共享数据 |

#### 锁争用时的两种表现

| 类型 | 行为 | 典型场景（Ch 10） |
|------|------|-------------------|
| **忙等（自旋）** | 线程 **旋转等待** 锁释放 | 临界区 **极短**、已持有锁者在另一 CPU 跑着 |
| **睡眠** | 当前任务 **阻塞**，让出 CPU | 临界区 **较长**、可能睡眠 |

#### 建议性与自愿性

| 事实 | 含义 |
|------|------|
| 锁是 **advisory & voluntary** | **编程约定** — 内核不会替你自动加锁 |
| 程序员责任 | **主动** `lock` / `unlock` 保护 **数据** |

> 错误：只给某条代码路径加锁，另一条仍裸访共享数据 → 锁 **无效**。

### ③ 并发的原因 · Causes of Concurrency

内核既要处理 **真并发**，也要处理 **伪并发**：

| 类型 | 说明 |
|------|------|
| **真并发（true concurrency）** | 多 CPU **同时** 执行 |
| **伪并发（pseudo-concurrency）** | 单 CPU 上 **交替** 执行 — 仍可能竞态 |

#### Linux 内核五类并发源

| # | 来源 | 说明 | 章节 |
|---|------|------|------|
| 1 | **中断** | 随时异步插入 | **Ch 7** |
| 2 | **软中断 / tasklet** | 中断返回后异步 | **Ch 8** |
| 3 | **内核抢占** | 内核态可被换下 | **Ch 4** |
| 4 | **睡眠** | 显式或缺页等导致 **进程切换** | **Ch 3–5** |
| 5 | **SMP** | 多处理器真并行 | **Ch 1** |

```
        ┌─ IRQ 上半部 ─────────────┐
        ├─ softirq / tasklet ─────┤
同一数据 ◄├─ 另一 CPU 上的进程 ────┼──► 必须同步
        ├─ 本 CPU 抢占后的另一任务 ─┤
        └─ sleep 唤醒后的竞争者 ────┘
```

**内核开发者：** 写每一行访问共享数据时，问：**五种来源里谁会同时碰到这里？**

→ [08-1 Day 14 临界区](../../08-system-low-level-hands-on/08-1-30days-os/notes/day-14-高分辨率及键盘输入.md)

### ④ 知道要保护什么 · Knowing What to Protect

| 数据 | 是否需要锁 |
|------|-------------|
| **局部自动变量** | **通常否** — 各线程独立栈 |
| **全局 / 共享内核结构** | **几乎都要** |

#### 基本原则

| 规则 | 说明 |
|------|------|
| 其他执行线程 **能访问** 的数据 | **需要同步** |
| 名言 | **「保护数据，而不是保护代码」**（**lock data, not code**） |

| 反例（保护代码） | 正例（保护数据） |
|------------------|------------------|
| 「给整个函数加一把大锁」 | 明确 **哪些字段** 是共享的，用 **合适粒度的锁** 护住 **那些字段的读写** |

#### per-CPU / 无锁（预告）

| 手段 | 思路 |
|------|------|
| **per-CPU 变量** | 每 CPU 一份 — 减少共享（Ch 8 softirq） |
| **RCU** | 读多写少（**Ch 10**） |

### ⑤ 死锁 · Deadlocks

**死锁** = 一个或多个线程 **互相等待** 对方已占有的资源 → **永远无法继续**。

#### 常见类型

| 类型 | 场景 |
|------|------|
| **自死锁（self-deadlock）** | 线程试图 **再次获取** 自己已持有的 **非递归锁** |
| **ABBA 死锁（deadly embrace）** | 线程1 持 **A** 等 **B**；线程2 持 **B** 等 **A** |

```
线程1:  lock(A) ──► 等待 lock(B)
线程2:  lock(B) ──► 等待 lock(A)     ← 环，死锁
```

#### 无死锁编码规则

| 规则 | 说明 |
|------|------|
| **嵌套锁固定顺序** | 全局约定：始终 **先 A 后 B** |
| **防止饥饿** | 避免某线程永远拿不到锁 |
| **不要重复获取同锁** | 除非明确使用 **递归锁** |
| **保持方案简单** | 锁越多、层次越深，ABBA 风险越大 |

→ **Ch 8** `spin_lock_irqsave` + `local_bh_disable` — 错误组合也会 **自死锁** 或 **活锁**

### ⑥ 争用和可扩展性 · Contention and Scalability

#### 锁争用（lock contention）

| 定义 | 影响 |
|------|------|
| 锁 **已被占用**，其他线程 **排队/自旋** 等待 | **瓶颈** — CPU 空转或睡眠唤醒开销 |

**HFT：** `perf lock`、延迟尖刺 — **高争用 mutex** 在热路径上是 **P99 杀手**。

#### 可扩展性（scalability）

| 定义 | 说明 |
|------|------|
| 增加 CPU 数量时，系统 **性能提升程度** | 理想线性扩展很少见 |

| 敌人 | 原因 |
|------|------|
| **粗粒度大锁** | 多 CPU **挤一扇门** — 扩展性极差 |
| **细粒度过多锁** | 无争用时 **加锁开销** 本身浪费 |

#### 锁粒度（granularity）

| 粒度 | 争用高时 | 争用低时 |
|------|----------|----------|
| **粗** | 扩展差 | 实现简单 |
| **细** | 扩展好 | 开销可能偏大 |

#### 工程建议（作者）

```
从简单锁开始 ──►  profiling 见争用 ──► 再细化粒度 / per-CPU / RCU
```

| 阶段 | 做法 |
|------|------|
| 初版 | **一把锁护结构** — 正确优先 |
| 优化 | 仅在有 **实测争用** 时拆锁、减临界区 |

→ **Ch 10** spinlock、mutex、seqlock、RCU 选型

### Ch 9 小结

| 问题 | 答案 |
|------|------|
| 临界区？ | 访问共享数据的代码 — 需 **原子** |
| 竞态？ | 多线程 **可能同时** 进同一临界区 |
| 锁的本质？ | **自愿** 保护 **数据** |
| 五类并发源？ | **中断 · BH · 抢占 · 睡眠 · SMP** |
| 死锁？ | **自死锁 · ABBA** — 固定加锁顺序 |
| 争用 vs 粒度？ | 粗锁害扩展；细锁有无谓开销 — **先简单后优化** |
