## 1.4–1.5 分析视角与性能挑战

### 两种分析视角

| 视角 | 方向 | 典型角色 | 问法 |
|------|------|----------|------|
| **工作负载分析** Workload analysis | **自上而下** | 应用开发者 | 「这个程序在**做什么**？」 |
| **资源分析** Resource analysis | **自下而上** | 系统管理员 / SRE | 「CPU/内存/磁盘/网络**够吗、忙吗**？」 |

**从分工来记（问题发起者 vs 问题解决者）：**

| | 工作负载（自上而下） | 资源（自下而上） |
|---|---------------------|------------------|
| **谁更常发起** | **应用开发者** — 写代码的人最清楚业务逻辑 | **运维 / 性能工程师** — 从硬件和 OS 指标反推 |
| **看什么** | 热路径、订单流、每 tick 干什么 | CPU 饱和、softirq、NUMA、丢包、page fault |
| **典型工具** | 火焰图（业务函数）、代码审查、tracing 业务 span | `top`/`perf stat`/`ethtool -S`/USE 法 |

两者**互补，不是二选一**：症状可能是 workload 设计问题（热路径多了 JSON），也可能是 resource 瓶颈（跨 NUMA 收包）— **先 workload 定位段，再 resource 找因**。

**HFT — 两种视角都要会：**

```
你写策略时  →  应用开发者  →  自上而下：解码→信号→风控→发单，各段逻辑是什么？
你调延迟时  →  性能工程师  →  自下而上：网卡/CPU/内存哪段扛不住、哪段在等？
```

小团队 **同一个人戴两顶帽子** — 不是「选应用还是选网络」，而是 **同一条 tick 链路上交替用两种问法**。网络专项（RSS、IRQ 亲和）是 resource 分析里的 **一块**，不是和「应用开发者」并列的二选一角色。

→ 方法论细节见 [Ch 2](../../chapter-02-methodologies/)（USE/RED 等）· tick 全链路清单见 [1.1–1.3](./section-1.1-1.3-系统性能角色与活动.md#hft--行情-tick-全链路-checklist)

### 性能工程为何难

| 挑战 | 说明 |
|------|------|
| **主观性** | 「够快」因用户/场景而异 |
| **复杂性** | 级联故障、子系统相互影响 |
| **多重原因** | 一个症状多个根因 |
| **多问题并存** | 需优先级，先修最大头 |

**HFT：** 抖动常是「多因叠加」（调度 + cache miss + 网络 burst）— 切忌只盯一个指标。
