## 2.6–2.7 建模与容量规划

> **定位：** 排队论 + 可扩展性模型 = **性能建模核心**。HFT 不是等 P99 爆了才加机器 — 用模型 **提前算拐点**，设限流/容量，再压测与实盘迭代。

```
     ↑___________________|___________________|
              别等出问题才被动优化
```

### 大白话：排队论是什么？

排队论研究的是 **「系统接请求的排队规律」**。

**餐厅类比：**

| 餐厅 | 排队论 | HFT |
|------|--------|-----|
| 进店的 **客人** | **请求（到达）** | **报单请求**、行情 tick |
| **服务员 + 后厨**（接待上限） | **服务节点** | **报单引擎**、撮合 loop |
| 等位多久、队排多长 | **排队延迟 L、逗留时间 W** | submit→ACK 变慢、网关对象积压 |
| 饭点会不会 **爆单** | 利用率 ρ 过高 → 系统失稳 | 开盘 burst → **超时 / reject** |

排队论要算的三件事，和开餐厅一样：

1. **客人多久能吃上** → 平均响应时间 **W**
2. **队排多长** → 队列里平均多少单 **L**
3. **高峰期会不会炸** → λ 接近接待上限时，延迟是否 **陡增**

**落到 HFT：** 报单请求是排队的「客人」，报单引擎的处理能力 μ 是餐厅的「接待上限」。用 **M/M/1**（只有一个服务台 — 单线程引擎）可以算出：**当报单到达率 λ 达到 μ 的约 70% 时，排队延迟开始陡增**。这时要在实盘炸单（超时拒单）之前 **提前限流或扩容**，而不是等 Grafana 上 reject 率已经飙了才反应。

```
饭点客人 → 70% 满座   ≈   报单 λ → 0.7μ
再进人就要排长队       ≈   排队延迟陡增、P99 爆
提前限流 / 加桌        ≈   限流 / 加核扩容
```

下面把类比换成 **模型命名**、公式和计算器。

---

### 模型命名：M/M/1 是什么意思？

排队论用 **Kendall 记号** `A/B/c` 描述一个队列：

| 位置 | 含义 | 常见取值 |
|------|------|----------|
| **A** | **到达间隔** 分布 | **M** = 负指数（→ 到达是 Poisson，间隔随机） |
| **B** | **服务时间** 分布 | **M** = 负指数；**G** = **General 任意分布**；**D** = 确定值 |
| **c** | **服务节点个数** | **1** = 单节点；**c** = 多节点并行 |

**M/M/1 — 排队论最基础模型：**

```
M  /  M  /  1
│     │     └── 只有 1 个服务节点（单线程报单引擎）
│     └──────── 服务时间 ~ 负指数分布（处理快慢有随机性）
└────────────── 请求到达间隔 ~ 负指数分布（报单到达随机、可 burst）
```

**HFT 相关的三个模型：**

| 模型 | 命名解读 | 对应场景 | 帮你算什么 |
|------|----------|----------|------------|
| **M/M/1** | 到达随机 + 服务随机 + **1 节点** | 单线程报单引擎、串行风控网关 | λ 接近 μ 时 **L、W** 何时陡增 → **限流阈值** |
| **M/M/c** | 到达随机 + 服务随机 + **c 节点** | 多核撮合集群、线程池、RSS 多队列 | **加几个核** 能把 ρ 压到 0.7 以下、**W 压进 SLO** |
| **M/G/1** | 到达随机 + **服务时间任意分布** + 1 节点 | 有的单 **快路径**、有的要走 **慢查**（风控 DB、合规校验） | 服务时间 **方差大** 时排队比 M/M/1 **更恶化** — 不能只用平均耗时估 |

**M/G/1 为何和 HFT 有关：** 报单处理往往不是「每单都一样快」— 缓存命中几 μs，触发慢查风控可能几百 μs。**G（General）** 允许你输入 **均值 + 方差**（或 P50/P99），比 M/M/1 更贴「快慢单混合」；方差越大，同样 ρ 下 **队列越长**（Pollaczek–Khinchine 结论 — **不必手推**，计算器会算）。

**选型速查：**

```
单线程、服务时间差不多     →  M/M/1（或更乐观的 M/D/1）
多核 / 线程池 / 多队列     →  M/M/c
快慢单混合、tail 重          →  M/G/1
```

### 排队论（Queueing Theory）— 性能建模核心

资源有队列时，**利用率未到 100%，响应时间和队列长度也会恶化** — 这就是「CPU 才 70% 怎么 P99 爆了？」的数学根因（餐厅 **七成满** 就开始排长队，同理）。

**HFT 常用模型（上表展开）：**

| 模型 | 对应系统 | 问什么 |
|------|----------|--------|
| **M/M/1** | 单线程报单引擎、单核撮合 loop、串行风控 | λ 接近 μ 时 **L、W** 多快恶化？ |
| **M/M/c** | 多核撮合集群、线程池、RSS 多队列 | **c 取多少** 能把排队延迟压进阈值？ |
| **M/G/1** | 快慢路径混合的报单/风控 | 慢单拉高 **方差** 后，队列是否比均值预估更糟？ |

符号（全书通用）：

| 符号 | 含义 | HFT 例子 |
|------|------|----------|
| **λ** | 到达率（requests/s） | 报单请求/s、tick/s |
| **μ** | 服务率（单服务台 capacity） | 单线程引擎 **稳定处理上限** orders/s |
| **ρ = λ/μ** | 利用率（须 **< 1** 才稳定） | 报单 λ=7000/s，μ=10000/s → ρ=0.7 |
| **L** | 系统中平均任务数（含服务中+排队） | 网关内存里积压的报单对象 |
| **W** | 平均逗留时间（排队+服务） | 报单 submit→ACK 均值 |

**Little 定律（永远成立）：** `L = λ × W` — 吞吐、延迟、队列长度三者绑在一起。

---

#### M/M/1 — 单线程报单引擎（一个服务台）

> 餐厅只有 **一位** 服务员接单 — 对应单线程报单引擎、串行风控。

假设：**M/M/1** — 到达间隔、服务时间均为负指数分布（见上节命名）；**一个**服务台。

**核心公式：**

```
ρ = λ / μ          （ρ → 1 时系统不稳定）
L = ρ / (1 - ρ)    （平均队列+服务中任务数）
W = 1 / (μ - λ)    （平均逗留时间 = 服务时间 + 排队等待）
W_q = ρ / (μ - λ)  （纯排队等待）
```

**为什么 70% 是拐点？— 单向车道 + 公式，不是拍脑袋**

把服务节点的处理能力想成 **一条单向车道**，请求是 **车**：

| 车流量（ρ） | 路上发生什么 |
|-------------|--------------|
| **~60%** | 大家都能匀速开；偶尔有一辆稍慢，后面很快散开，**几乎不堵** |
| **~70%** | 只要有一辆因突发情况慢下来，后面就开始 **排队**；车流的 **随机波动** 会把队尾 **快速拉长** |
| **→90%** | 路几乎满了，任何小扰动 → **长队、长时间等待** |

**HFT 报单同理：**

- 下单速率 **λ 随机波动**（开盘 burst、集中报单）— M/M/1 里第一个 **M**。
- 单订单处理耗时也有 **微小随机差**（缓存命中 vs 多一次分支）— 第二个 **M**。
- 当报单引擎 **使用率 ρ ≈ 70%** 时：来一小波集中报单，或几单处理稍慢 → 整条队列延迟从 **μs 级跳到 ms 级** → 实盘 **timeout / reject**。

**公式里藏着的「暴涨」：**

平均逗留时间相对空闲时（仅服务、不排队）的倍数：

```
W / W₀ = 1 / (1 - ρ)     （W₀ = 1/μ）
L      = ρ / (1 - ρ)     （队列长度同样含 1/(1-ρ) 因子）
W_q    ∝ ρ / (1 - ρ)     （纯排队等待）
```

关键在 **分母 (1 − ρ)**：ρ 越接近 1，分母越趋近 0，**等待时间不是线性涨，而是按 1/(1−ρ) 急剧放大** — 曲线在 60% 还算平，**70% 附近斜率明显变陡**，80% 以后近乎垂直。这就是 **70% 作为限流/扩容参考** 的来源：**M/M/1 推导出的核心结论**，不是经验拍脑袋。

```
等待倍数  W/W₀ = 1/(1-ρ)

  10× │                              ╭
   5× │                         ╭────
   3× │                    ╭────           ← ρ≈0.7 拐点区
   2× │              ╭─────
   1× │─────────────
      └────────────────────────────── ρ
        0.5   0.6  0.7  0.8  0.9   1.0
```

**利用率 vs 延迟（相对空闲时 W₀ = 1/μ）：**
| ρ（利用率） | L（平均任务数） | W / W₀（延迟倍数） | HFT 读法 |
|-------------|-----------------|---------------------|----------|
| 50% | 1.0 | **2×** | 尚可，留 headroom |
| 60% | 1.5 | **2.5×** | 开始吃 tail |
| **70%** | **2.33** | **~3.3×** | **拐点区 — 限流阈值常设在此附近** |
| 80% | 4.0 | **5×** | P99 极易爆 |
| 90% | 9.0 | **10×** | 实盘不可接受 |

**HFT 用法 · 报单引擎限流：**

```
已知：压测单线程网关 μ = 10,000 orders/s（稳定 P99 未恶化时的上限）
模型：ρ = λ/μ
若 SLO：平均逗留 W < 30 μs，服务时间 1/μ = 10 μs
  → W/W₀ = 1/(1-ρ) < 3  →  ρ < ~0.67
  → 限流阈值 λ_max ≈ 0.67 × μ ≈ 6,700 orders/s
```

不必等实盘 P99 飙了才砍 — **M/M/1 推导**：ρ 靠近 1 时 `1/(1−ρ)` 分母趋零，排队延迟陡增；**70% 是暴涨曲线的拐点**。限流、back-pressure 都应 **低于 knee**；来不及扩容时 **先限流**，避免 μs 变 ms 后大面积拒单。

**注意：**

- 真实报单服务时间往往 **更固定**（接近 **M/D/1**）— 同样 ρ 下排队往往 **比 M/M/1 还乐观一点**，但 **70% 附近变陡** 的定性结论不变。
- 单线程段：**锁、序列化、单连接发单** 都适用 M/M/1 心智模型。
- 模型给 **均值**；HFT 盯 **P99** — 压测时在 ρ=0.5/0.6/0.7 各扫一轮，画 **λ vs P99** 曲线找 **实测 knee**，与理论 ρ 对照。

→ 压测找 knee：[Ch 12.3](../../chapter-12-benchmarking/notes/section-12.3-基准测试方法论.md)

#### M/M/c — 多核撮合 / 并行服务台（多位服务员）

> 餐厅开了 **c 个窗口** 同时接单 — 对应多核撮合、RSS 多队列、线程池。

**c 个相同服务台**（多核各跑独立 shard、或线程池 c  workers）：

```
ρ = λ / (c × μ)     （ρ 须 < 1）
```

| 对比 M/M/1 | M/M/c 行为 |
|------------|------------|
| 单核 ρ=0.7 已陡 | **c=4, ρ=0.7** 等价每核 λ/(4μ)，总吞吐可近 **4× 单核**（理想） |
| 一个队列等一个 CPU | **共享队列** → 锁、order book 跨 shard 一致性 → 实际 **< 理想 c 倍** |

**实操：** 固定 λ 与目标 **W ≤ W_SLO**，在计算器里 **扫 c = 1, 2, 4, 8…** 直到 ρ = λ/(cμ) < 0.7 且 W 达标 — 得到 **最少核数**。再压测验证 **有效 c**（锁争用时 USL 会压低有效并行度）。

**HFT 映射：**

| 组件 | M/M/c 怎么读 |
|------|----------------|
| 多核 **撮合 / 定价** 集群 | c = 工作核数；μ = 单核 tick 处理率 |
| **RSS 多队列** 网卡 | c = 队列数；每队列一条 M/M/1 |
| 线程池 **报单序列化** | c = 线程数；争用大时退化为「有效 c < 物理 c」→ 用 USL 校正 |

**实操：** 压测时固定 λ，增加 c=1,2,4,8… 测吞吐与 P99；若加核 **吞吐不成比例、P99 反升** — 不是 M/M/c 理想区，是 **争用/一致性**（→ USL）。

---

#### M/G/1 — 快慢单混合（服务时间任意分布）

> 同一窗口接单，但有的菜 **现炒 30 秒**、有的要 **现炖 5 分钟** — 对应报单 **快路径 vs 慢查风控**。

**M/G/1 要点：**

- 到达仍为 **M**（Poisson / 负指数间隔）— burst 报单仍适用。
- 服务时间为 **G（General）**：只需 **均值 E[S]** 和 **二阶矩 E[S²]**（或方差 Var(S)），不必假设指数分布。
- 同样 ρ 下，**Var(S) 越大 → 排队越长** — 少数慢单就能把 tail 拖垮。

**HFT 映射：**

| 现象 | M/M/1 够用吗 | M/G/1 |
|------|--------------|-------|
| 每单处理时间差不多 | ✅ | 可选 |
| 90% 单 < 5 μs，10% 走 DB > 200 μs | ❌ 会 **低估队列** | ✅ 用 P50+P99 或实测方差 |
| 风控「抽样深查」随机触发 | ❌ | ✅ |

**输入从哪来：** 报单链路上 **histogram**（[2.5 埋点](./section-2.5-性能分析方法论.md)）导出 E[S]、P99，或压测回放统计 **服务时间样本** → 喂给计算器 — 仍 **不必手推 PK 公式**。

### 不用手推公式 — 排队论计算器

**日常用法：** 输入 **报单速率 λ**、**单订单平均处理时间 1/μ**（或 E[S]、方差），直接读 **L（队列长度）、W（逗留时间）、W_q（纯排队）**；M/M/c 再扫 **c** 找满足 SLO 的最小核数。

**输入 / 输出：**

| 输入（来自压测或 Grafana） | 输出（决策用） |
|----------------------------|----------------|
| λ — orders/s 或 ticks/s | **L** — 网关里平均积压多少单 |
| μ 或 E[S] — 单均处理时间 | **W** — 平均 submit→完成 |
| c — 候选核数（M/M/c） | **ρ** — 是否 > 0.7 → 该限流还是加核 |
| Var(S) 或 P99（M/G/1） | **W_q** — 纯排队等了多少 |

**工具示例（任选，不必自建）：**

| 类型 | 示例 | 说明 |
|------|------|------|
| **Python** | [`queueing-tool`](https://pypi.org/project/queueing-tool/) | M/M/c、M/G/1；`pip install queueing-tool` |
| **Python** | 上式 5 行 | M/M/1 快速心算、脚本嵌入 |
| **Web** | 搜 `M/M/1 calculator`、`M/M/c calculator` | 快速试 λ、μ、ρ |
| **仿真** | SimPy | 服务规则复杂、分布很怪时 |

**M/M/1 心算（5 行 Python，可与在线计算器对照）：**

```python
lam, mu = 7000, 10_000          # λ orders/s, μ orders/s
rho = lam / mu                  # 0.7
L = rho / (1 - rho)             # 平均队列+服务中 ≈ 2.33 单
W = 1 / (mu - lam)              # 平均逗留 ≈ 333 μs（若 1/μ=100μs 量级需按单位换算）
Wq = rho / (mu - lam)           # 纯排队等待
```

**M/M/c、M/G/1：** 用 [`queueing-tool`](https://pypi.org/project/queueing-tool/)（`pip install queueing-tool`）或 Web 搜 `M/M/c calculator` — 输入 λ、μ、c 或 E[S]、Var(S)，读 L、W。

**M/M/c 扫核数（伪代码逻辑）：**

```
for c in [1, 2, 4, 8, 16]:
    ρ = λ / (c * μ)
    W = calculator_mm_c(λ, μ, c)
    if ρ < 0.7 and W < W_SLO:
        print(f"最少 {c} 核"); break
```

→ 算完 **必须** 接 [Ch 12 压测](../../chapter-12-benchmarking/) 与实盘 histogram 校正 — 模型给 **量级与拐点**，不是替代实测。

### 可扩展性模型

排队论管 **单资源饱和**；可扩展性模型管 **加核 / 加机有没有用**。

#### 阿姆达尔定律（Amdahl's Law）

```
加速比 S(N) = 1 / ( f + (1-f)/N )
```

- **N** = 并行度（核数、实例数）
- **f** = **必须串行** 的比例（0~1）

| 串行比例 f | 8 核理论加速上限 | HFT 含义 |
|------------|------------------|----------|
| 5% | ~4.7× | 热路径有一点全局锁就封顶 |
| 10% | ~4.3× | 常见「看起来能并行」的策略引擎 |
| 20% | ~3.3× | **盲目堆 32 核可能只值 3×** |

**HFT 用法：**

- tick→parse→**串行风控**→send：先量 **f**（profile 串行段占比），再算加核值不值。
- **避免盲目堆机器**：8→32 核前，用 Amdahl 算 **理论上限**；若 f=15%，32 核上限 ~5.3×，不如 **砍串行**（减 f）。

→ 与 [Ch1.6 延迟与关键路径](../../chapter-01-intro/notes/section-1.6-延迟指标与读法.md)、[CSAPP 1.9 Amdahl](../../../01-CSAPP-3rd/chapter-01-tour-of-computer-systems/notes/section-1.9-重要主题-Amdahl与并发与抽象.md) 呼应。

---

#### 通用可扩展性模型（USL · Universal Scalability Law）

Amdahl 只考虑 **串行**；USL 再加两项实测可拟合参数：

```
X(N) = N / ( 1 + α(N-1) + βN(N-1) )
```

| 项 | 含义 | HFT 对应 |
|----|------|----------|
| **α · (N-1)** | **争用（contention）** | 锁、订单簿 mutex、单连接发单 |
| **β · N(N-1)** | **一致性（coherence）** | false sharing、跨 NUMA 订单簿、cache line 乒乓 |

**曲线形状：**

```
吞吐
  │     ╭──── 理想线性
  │    ╱  ╲
  │   ╱    ╲___  USL 峰值后 **反而下降**（加核更慢）
  │  ╱
  └──────────────── N（核数）
```

**HFT 用法 · tick 速率翻倍要加什么？**

USL 不直接输出「内存 GB」，而是告诉你 **扩展性拐点**；结合 workload 特征 **反推资源**：

```
1. 基线：N 核、tick_rate = R、P99 达标、ρ_net / ρ_cpu 记录
2. 目标：tick_rate → 2R
3. Amdahl：串行段是否已成为 f>10% 瓶颈？→ 先减 f，再加核
4. USL：拟合 α、β — 加核到 N' 时吞吐能否 ≥ 2R？
   - 若 N' 已到 USL 峰值 → 加核无效，要 **分片 / 无锁 / 多实例**
5. 资源因子（在扩展性 OK 的前提下）：
   - 内存：订单簿深度 × 标的数 × 2R 线性估（+ WSS 压测）
   - 网卡：2R tick × 平均包长 → 所需 pps / Gbps（留 30% headroom）
   - 每核 μ：M/M/c 算 ρ 是否仍 < 0.6~0.7
```

**例：** tick 从 50 万/s → 100 万/s

| 步骤 | 动作 |
|------|------|
| 模型 | M/M/c：现 4 核 ρ=0.55 → 2× tick 若仍 4 核则 ρ→1.1 **不稳定** → 至少 c=8 或 μ↑ |
| USL | 8 核历史压测若 α 大（锁争用）→ 有效吞吐 < 2× → 先 **shard 订单簿** 再扩核 |
| 网卡 | 100 万 pps × 平均 200 B ≈ 1.6 Gbps + header → **10G 口仍够**，但要查 **PPS 上限** 非仅带宽 |
| 内存 | 回放 2× tick 压 `pmap`/WSS — 订单簿 RSS 是否线性涨 |

---

### 容量规划 — 基准三步法

Gregg 容量规划 + HFT 落地：

```
① 模型算理论值     M/M/1·c 算 ρ 与 W；Amdahl/USL 算加核上限与 2× tick 是否可行
       ↓
② 压测 / 回放验证   扫 λ 找 knee；加核扫 N 拟合 USL；fio/打流验证网卡 pps
       ↓
③ 实盘数据迭代      Grafana：ticks/s、ρ_cpu、P99、drop — 校正 λ、μ、α、β
```

| 阶段 | 输入 | 输出 |
|------|------|------|
| **① 理论** | 目标 SLO、单核 μ、串行比 f | 限流阈值、最少核数 c、网卡/内存 **量级** |
| **② 压测** | 回放 trace、阶梯 λ | 实测 knee、USL 曲线、P99@ρ |
| **③ 实盘** | Prometheus 面板 | 修正模型；旺季前 **主动** 扩容 / 限流 |

**HFT 与云 Auto Scaling：**

- 共置低延迟：**固定容量 + 模型留 30~40% headroom**（ρ 目标 < 0.6~0.7），而非临时扩容 — 扩容本身有冷启动与 NUMA 抖动。
- 非关键路径（回放、风控离线）：可用弹性容量。

### 速查 · HFT 容量规划检查单

- [ ] **M/M/1**：单线程段 ρ=0.7 限流线（计算器验 L、W）
- [ ] **M/M/c**：扫 c，延迟压进 SLO 的 **最少核数**
- [ ] **M/G/1**：快慢单混合时用 **E[S]+方差/P99**，不单用均值
- [ ] 用 **queueing-tool / 在线计算器** 算理论值，再压测验证
- [ ] **Amdahl**：串行 f 是否 >10% — 加核前先减 f
- [ ] **USL**：拟合 α、β — 是否已过扩展峰值
- [ ] tick **2×** 场景：核数、**pps**、**RSS** 三因子估算 + headroom
- [ ] 理论 → **Ch12 压测** → **Grafana 实盘** 闭环，非事后救火
