# 2.6.4 友元与前置声明

## 本节讲什么

friend 含义、友元前置声明、有源前置声明、三概念辨析。

## 四、友元（friend）与友元前置声明

### 友元的含义

把**全局函数**声明为类的友元：

- **正常情况**：外部全局函数只能访问类的 **public** 成员，碰不到 **private** / **protected** 字段
- **写 `friend 函数名` 之后**：该全局函数可直接读、写类的私有成员，**不受访问权限限制**

```cpp
class Secret {
    friend void peek(const Secret&);   // peek 可以访问 private
private:
    int code = 42;
};
void peek(const Secret& s) { /* 可直接用 s.code */ }
```

完整版 `Sales_data` 中 `read` / `print` / `add` 是全局 IO 工具，通过 `friend` 访问成员（教材惯例；入门版成员本为 public）。

### 友元前置声明规则

**函数须先在 struct 定义前写函数原型（前置声明），再在 struct 内写 `friend`**，否则：

- struct **之前**的代码（如流构造函数）找不到 `read` 等名字
- 头文件分离时，其他 `.cpp` 看不到全局接口

完整版代码在 `struct Sales_data` **之前**有三行：

```cpp
// 友元前置声明
Sales_data add(const Sales_data&, const Sales_data&);
std::ostream& print(std::ostream&, const Sales_data&);
std::istream& read(std::istream&, Sales_data&);
```

随后在结构体内部用 `friend` 把上述**全局函数**指定为友元，允许它们访问非公有成员（本例成员虽为 public，但 `read` 需写 `item.revenue` 等，教材仍走友元 IO 模式）。

### 什么是友元前置声明

**前置声明**：在 `struct`/`class` 定义完成之前，先在**全局域**写出友元函数的原型。编译顺序：

1. 读到三行函数原型 → 记住签名；`Sales_data` 作为返回/参数类型被**前向声明**为不完整类型（见下节「类的前置声明」）
2. 读到 `struct Sales_data { ... friend add; ... }` → `friend` 与前面全局原型绑定，授予访问权限
3. 在 struct 之后写函数定义（函数体）

### 为什么必须有？（`Sales_data` 的两个核心原因）

**1. 函数原型写在 struct 定义之前，签名里已用到 `Sales_data`**

教材把 `add` / `read` / `print` 的声明放在 struct **上面**。没有这三行，就无法在定义 struct 之前写出合法函数原型；后面的

```cpp
Sales_data::Sales_data(std::istream& is) { read(is, *this); }
```

也要求 `read` 在此刻**已经声明**。

**2. 头文件 / 源文件分离：对外暴露 IO 工具接口**

- **`.h`**：三行前置声明 + struct 定义（含 `friend`）
- **`.cpp`**：`add` / `read` / `print` 的函数体

其他 `.cpp`（如 `main.cpp`）只 `#include "Sales_data.h"` 即可调用 `print(cout, item)`，不必关心实现细节。

### 两种友元写法对比

| 写法 | 是否需要类前前置声明 | 类外能否调用 |
|------|---------------------|--------------|
| **全局友元 + 类外定义**（`Sales_data`） | ✅ 需要 | ✅ |
| **类内 `friend` 后直接写函数体** | ❌ | ❌（几乎仅教学用） |

**情况 1：全局友元（教材用法）**

```cpp
Sales_data add(const Sales_data&, const Sales_data&);

struct Sales_data {
    friend Sales_data add(const Sales_data&, const Sales_data&);
};

Sales_data add(const Sales_data& lhs, const Sales_data& rhs) { /* ... */ }
```

函数属于**全局域**，不属于 `Sales_data`；业务代码、`main`、其他文件都要调用 → 必须类前声明。

**情况 2：类内定义友元（不需前置声明，类外不可用）**

```cpp
struct Test {
    friend void show(Test t) {
        std::cout << t.val;
    }
private:
    int val = 10;
};
// show(t);  // 类外调用：未声明
```

### 关键语法规则

1. `friend` 声明**授予**访问权，但**不等于**完整的工程接口布局；教材在 struct 前写全局原型，是为了**类型前向声明 + 分离编译**。
2. C++11 起，非定义的 `friend` 声明会把函数名**注入**外围命名空间；因此「只有 `friend`、没有类前原型」时，struct **之后**的 `main` 里调用 `add(a,b)` **有时能通过**。但 **`Sales_data(std::istream&)` 依赖的 `read` 必须先声明**，且头文件惯例仍要求三行写在 struct 前。
3. `friend` 写在 `public` / `private` **均可**，只影响类成员访问，不影响友元本身。
4. `print` / `read` 是全局 IO 工具，主程序与其他模块频繁调用 → 必须出现在头文件的全局声明区。

### 错误示范：删掉前置声明 + 流构造函数

下面代码删掉 struct 前的三行声明，并保留依赖 `read` 的构造函数实现 —— **必然编译失败**（`read` 未声明）：

```cpp
#include <iostream>
#include <string>

struct Sales_data
{
    Sales_data() = default;
    Sales_data(std::istream& is) { read(is, *this); }  // ❌ read 尚未声明

    friend Sales_data add(const Sales_data&, const Sales_data&);
    friend std::ostream& print(std::ostream&, const Sales_data&);
    friend std::istream& read(std::istream&, Sales_data&);

    std::string bookNo;
    unsigned units_sold = 0;
    double revenue = 0.0;
};

std::istream& read(std::istream& is, Sales_data& item) { /* ... */ return is; }
// 报错：'read' was not declared in this scope（或 equivalent）
```

**正确做法**：在 struct 前补上 `read`（及 `add`、`print`）的函数原型，或把 `Sales_data(std::istream&)` 的**定义**挪到 struct 之后、且放在 `read` 声明之后。

### 简化记忆

```
类外要调用的友元全局函数 → struct 定义前写函数原型 + struct 内 friend
只在类内用的友元 → friend 后直接写函数体，可不要前置声明
Sales_data 的 add/read/print 是全局 IO 工具 → 头文件前三行 + .cpp 实现
```

详见 ch7 [7.3 友元](../ch07-classes/7.3-友元（Friend）.md)。

### 类的前置声明：普通 vs 有源

> **易混提醒**：**有源** = 有无的「**有**」+ 源代码的「**源**」（implementation 在 `.cpp` 里）；与上一节 **友元**（friend，朋友的「友」）无关，勿写成「友源」。

这里的前置声明指**类型**（`class` / `struct`），与上文**友元函数**的前置声明是两层概念。

**普通前置声明**——头文件里只写类型名，不暴露成员：

```cpp
class Sales_data;   // 或 struct Sales_data;
```

此时 `Sales_data` 是**不完整类型**，编译器只知道「有这么个类型」，不知道大小和成员。因此只能用于：

- 指针 / 引用：`Sales_data* p;`、`Sales_data& r;`
- 函数参数 / 返回值（传引用或指针）

**不能**按值定义对象、不能访问成员、不能 `sizeof(Sales_data)`——因为定义尚未可见。

**有源前置声明**——语法同样是 `class Sales_data;`，但含义不同：

> **「有源」**：这个声明不是空泛的类型占位，而是已经和**具体实现源文件**绑定——类的完整定义在某个 `.cpp`（或静态库 / 动态库）里，且该实现**会被链接进当前工程**。

典型布局：

| 文件 | 内容 |
|------|------|
| `Sales_data.h` | `class Sales_data;` 或只含指针成员的接口；**不** `#include` 重依赖头 |
| `Sales_data.cpp` | `class Sales_data { ... };` 完整定义 + 成员函数实现 |
| `main.cpp` | `#include "Sales_data.h"`，通过指针/引用或 `.cpp` 提供的工厂函数使用 |

编译期：消费方 `.cpp` 只看到不完整类型 → **不必拉入** `Sales_data` 的全部成员与依赖头，**缩短编译时间**。  
链接期：链接器在 `Sales_data.o` / `.lib` 里找到完整实现 → 调用仍然正确。

```
OtherModule.h          Sales_data.h              Sales_data.cpp
  class Sales_data;  →  对外接口 / 指针成员   →   完整 class { ... }
  （有源：知道 .cpp      （编译 Other 时不          （单独编译、链入工程）
   会被链进来）           展开 Sales_data 内部）
```

HFT 大型工程里常用：**头文件尽量前置声明 + 指针/引用（或 PIMPL）**，把重实现藏在 `.cpp` 或库里，减少头文件依赖扇出（见 [Effective C++ 条款 31](../../02-Effective-C++/ch05-implementation/item31-降低文件之间的编译依赖.md)）。

**与 `Sales_data` 教材示例的关系**：Primer 2.6 的写法是在**同一头文件**里给出完整 `struct Sales_data { ... }`，属于「声明 + 定义都在头文件可见」的入门路径；工程化后会演进为「`.h` 前置声明或瘦接口 + `.cpp` 完整定义」的**有源**模式。

### 三概念辨析（之前易混的三件事）

| 概念 | 是什么 | 解决什么 | 彼此关系 |
|------|--------|----------|----------|
| **友元 `friend`** | 把全局函数变成类的「自己人」 | **访问权限**：外部函数可读 private 成员 | — |
| **友元前置声明** | struct **前**写 `add`/`read`/`print` 函数原型，struct **内**写 `friend` | 编译顺序 + 头文件分离；类外能调用友元函数 | 只和 **friend 函数**有关 |
| **有源前置声明** | `.h` 里 `class Sales_data;`，完整定义在已链入的 `.cpp` | **减头文件依赖**、缩短编译 | 和 **friend 无关**；**有**=有无的「有」，勿写成「友源」 |
| **类内初始值** | 成员写 `= 0` 或 `{0}` | 统一成员默认值 | **不破坏**聚合（见 [2.6.2 对比代码](./2.6.2-类内初始值.md)） |
| **聚合** | 纯数据 struct，无用户自定义构造等 | 支持 `{}` 按顺序 / 部分初始化 | 与类内初值**可共存**；自定义构造才破坏 |

**一句话总结**：

1. **`friend` 友元**：放开外部全局函数对 private 成员的访问；类外要用的友元函数须 **struct 前声明 + struct 内 friend**。
2. **类内 `变量{值}`**：给成员默认值；**不会**丧失聚合——真正破坏聚合的是**自定义构造函数**等。
3. **有源前置声明**：`class X;` + `.cpp` 有实现，减编译依赖，与 friend 无关。
