# 3.2.4 string 与 C 风格字符串

## 本节讲什么

C 过程式裸内存 vs `std::string` 封装；`char*` 声明陷阱；RAII 自动扩缩；HFT 固定缓冲折中。

## C 过程式 vs C++ 封装：两种思维

| | **C 风格字符串** | **`std::string`** |
|---|------------------|-------------------|
| 思维 | **裸内存 + 手动管理**（过程式） | **封装内存 + 自动管理**（面向对象 / RAII） |
| 长度 | 自己 `strlen` 或另记 | `size()` |
| 分配/释放 | `malloc` / `free`，易漏 | 构造/析构自动管 |
| 结束符 | 时刻记得 `'\0'`，防越界 | 类内部维护 |
| 操作 | `strcpy` / `strcat` | `append`、`+`（运算符重载，代码更简洁） |

C++ 相对 C 的升级：把「算长度、分配释放、加 `\0`」收成**成员函数 + 运算符**，你只调接口；业务代码更安全。

到了 **HFT 极致性能**场景，有时又会**绕回裸内存布局**——见下文「固定缓冲 + RAII」折中。

## C 风格：`char*` 是什么？

`char *`（或 `char[]`）本质是**普通指针**，指向字符数组的**首元素**；字符串以 **`'\0'`** 结束，长度靠扫到 `\0` 才知道——**没有**内建的 `size()`。

**声明风格**：很多人写 `char* p`，但 `char* p, q;` 只会把 **`p`** 声明成指针，**`q` 仍是 `char`**——经典坑。严谨写法 **`char *p, *q;`**（见 [2.4.2 声明语法](../ch02-variables-and-basic-types/2.4-const-qualifier/2.4.2-const口诀与声明语法.md)）。

```cpp
const char *msg = "FIX.4.2";   // 推荐：* 靠变量名
char buf[64] = "order";

// 拼接、变长都要手动 malloc/realloc、strcpy/strcat，极易越界或忘 '\0'
```

读网关旧 API 时常见 **`const char*`**（只读视图）——对应 const 里的 **指向常量的指针** `const char *p`：不能通过 `p` 改内容，但 `p` 可改指向别的缓冲区（见 [2.4.1 const 基础](../ch02-variables-and-basic-types/2.4-const-qualifier/2.4.1-const基础与constexpr.md)）。

## string 封装了什么？

`std::string` 把**指针、数组、长度、`'\0'`、内存分配**等细节收成类，你只调接口：

| 你关心的 | C 风格要自己管 | `string` |
|----------|---------------|----------|
| 长度 | 扫 `\0` 或另记计数 | `size()` / `empty()` |
| 追加 | `strcat` + 缓冲够大 | `append()` / `+=` |
| 内存 | `malloc` / `free` | 析构自动释放 |
| 结束符 | 手动保证 `\0` | 内部维护 |

```cpp
std::string fix = "8=FIX.4.2|";
fix.append("35=D|");           // 不用算 strlen、不用怕 '\0'
std::cout << fix.size();      // 字符个数，不含你操心的 '\0' 细节
```

**HFT 低 bug 率**：业务字符串优先 `string`，少碰裸 `char*` 的手动内存与 `\0`；必须对接 C API 时用 `c_str()` **临时**过渡。

## RAII、自动扩缩 vs Rust 所有权

**`string` 拥有**自己堆上的字符缓冲区（类似 Rust 的 `String` 拥有堆数据）：

- **变长**：`append` / `+=` / `+` 时，内容变长则**自动重新分配**更大缓冲；变短后**可能**缩小容量（实现相关，不保证立刻 shrink）
- **析构**：`string` 离开作用域 → 析构函数**自动 `delete[]` 内部缓冲** → 不用手写 `free`

这叫 **RAII**（资源获取即初始化）：**构造管分配，析构管释放**。

| | **`std::string`（C++）** | **Rust `String`** |
|---|-------------------------|-------------------|
| 谁管内存 | 对象**拥有**堆缓冲；析构释放 | **所有权**；`drop` 时释放 |
| 自动释放 | 作用域结束析构 | 离开作用域 `drop` |
| 编译期防错 | ❌ 无借用检查；仍有悬垂引用、`char*` 坑 | ✅ 借用检查器 |
| 拼接 | `+` 可能产生**临时** `string` | 移动/所有权，编译期优化更多 |

**相似点**：都是「对象活着就占资源，对象销毁就释放资源」，HFT 里比裸 `char*` 少泄漏。

**不同点**：C++ **没有** Rust 那套所有权 / 借用规则；`string` 仍可与裸指针、悬垂 `string&` 组合出 bug——RAII 只保证 **`string` 自己**析构时释放**自己**的缓冲，不保证你的指针仍有效。

```cpp
std::string s = "order";
const char* raw = s.c_str();
s += " extra";     // s 可能重分配 → raw 悬垂，危险
```

| | **`std::string`** | **C 风格 `char*` / `char[]`** |
|---|-------------------|-------------------------------|
| 长度 | **可变**，随内容自动增长/收缩 | 固定缓冲或裸指针，易越界 |
| 内存 | 构造时分配、析构时**自动释放**（RAII） | `malloc` / `new[]` 须配对 `free` / `delete[]` |
| 拼接 | `+` / `+=` 自动重新分配 | `strcat` 等须保证目标缓冲足够大 |
| 安全性 | 边界由 `size()` 管理；`at(i)` 越界检查 | 无长度信息，**缓冲区溢出**常见 |

```cpp
// ❌ C 风格：易忘释放或越界
char* p = new char[100];
// ... 若异常路径未 delete[] → 泄漏

// ✅ string：作用域结束自动析构
{
    std::string fix = "8=FIX.4.2|35=D|";
}   // 内存自动回收
```

**与 C 接口互操作**（读旧 API、写 C 函数时）：

```cpp
std::string s = "hello";
const char* c = s.c_str();   // 只读 C 风格视图；s 改内容后可能失效
// s[0] = 'H';               // 再调 c_str() 前勿假设 c 仍有效
```

HFT 新代码优先 `string`；底层网关若只给 `const char*`，可立刻包一层 `std::string msg(ptr);` 再处理。

## HFT 折中：固定长度缓冲 + C++ RAII

热路径（高频行情、合约码、短订单号）上，`string` 的**动态分配**可能带来延迟和碎片。常见折中：

- **内存布局用 C 的思路**：栈上或预分配好的 **固定长度 `char` 数组**，大小在编译期或启动时定死，**热路径零 heap 分配**
- **生命周期用 C++ 的思路**：用 **RAII 类/结构体** 包住这块缓冲，析构时自动清理（栈数组本身无需 free；若类内 `new[]` 则析构里 `delete[]`）

```cpp
// 栈上固定缓冲：类似 C，但放在 struct 里便于传递、封装
struct TickSymbol {
    char data[16]{};   // 合约码，如 "IF2406"，栈分配、无动态 alloc

    const char* c_str() const { return data; }
    // 写入时仍须自己保证不越界、保留 '\0'
};

// 或 std::array<char, 16> + 长度字段，仍无 string 的 realloc
#include <array>
struct FastSymbol {
    std::array<char, 16> buf{};
    uint8_t len = 0;
};
```

| 场景 | 推荐 |
|------|------|
| 一般业务、FIX 解析、日志 | `std::string`（安全、可变长） |
| 热路径、长度有上界、极高频 | 固定 `char[]` / `std::array<char,N>` |
| 既要固定缓冲又要异常安全释放堆内存 | 自定义 RAII 类（构造分配、析构释放） |

**结合两者优点**：**C 的内存布局保证性能**，**C++ 的 RAII/类封装保证释放逻辑不遗漏**——不是退回纯 C，而是在关键路径上**有控制地**少用 `string` 的动态性。
