## 5. UEFI 启动流程 · BIOS 传统启动对比

> **本章核心图：** 从通电到 `EfiMain` 跑起来 — MikanOS 走 **UEFI 线**，不是 BIOS 512 字节扇区线。

---

### 一、UEFI 完整启动流程（七步）

```
① 通电复位
      ↓
② 完整硬件初始化
      ↓
③ 枚举可启动设备
      ↓
④ FAT 分区 · 固定路径找 BOOTX64.EFI   ← 核心差异
      ↓
⑤ 加载 EFI 镜像到内存
      ↓
⑥ 移交执行权 → EfiMain
      ↓
⑦ 你的 C 代码运行（Hello World / 加载内核…）
```

#### 1. 通电复位

主板上电，CPU 自动复位，跳转主板 Flash ROM 里固化的 **UEFI 固件**。

**要点：** 现代 x86_64 PC 上 UEFI 通常 **直接进入 64 位长模式语境** 运行固件与应用 — **没有** 老式 BIOS 那套「先 16 位实模式 @ 低地址」的启动阶段（与 [02 30days-os Day1](../../02-30days-os/day-01-boot-asm/) 的 `0x7C00` 实模式 **不是一条路**）。

#### 2. 完整硬件初始化

UEFI 固件初始化：**CPU、芯片组、内存、显卡、键鼠、存储、网卡** 等；自带 **标准化驱动/协议**，比 Legacy BIOS 初始化更完整。

#### 3. 枚举所有可启动设备

扫描 **硬盘、U 盘、NVMe、网络启动** 等；读取 **GPT**（为主）或 MBR 分区表，识别各分区及 **文件系统类型**。

#### 4. 固定标准路径查找引导程序（**核心重点**）

在符合要求的 **EFI 系统分区（ESP）** 上 — 规范要求 **FAT/FAT32** — 按 **固定路径** 查找默认引导应用：

```
/EFI/BOOT/BOOTX64.EFI
```

| 项 | 说明 |
|----|------|
| **架构** | x86_64 → **`BOOTX64.EFI`**；ARM64 → **`BOOTAA64.EFI`** |
| **硬性要求** | 分区须 **FAT32**（常见 ESP 格式）；**路径/文件夹名/文件名** 按规范放置（大小写通常 **不敏感**） |
| **不是** | 磁盘 **第 0 扇区 512 字节**；也不是把 `.efi` 塞进引导扇区 |

→ 部署与排错 [§3 真机与 QEMU](./section-3-真机与QEMU测试.md) · 编译产出 [§2](./section-2-二进制编辑器与BOOTX64.md)

#### 5. 加载 EFI 镜像到内存

固件读取 **`BOOTX64.EFI`**（**PE32+** 格式可执行文件），分配物理内存，将整份镜像 **载入 RAM**。

#### 6. 移交执行权

CPU 跳转到镜像入口 — C 程序里即 **`EfiMain(ImageHandle, SystemTable)`** — **第一个「你的代码」** 开始执行；后续由 Loader **加载自制 64 位内核**（Ch3+）。

#### 7. 运行效果

执行你写的 C 逻辑：**初始化控制台、打印 Hello World**、读内存 map、加载 `kernel.elf` 等 — 本章仅到 **ConOut 打印**。

### 二、老式 BIOS 启动流程（对比用）

```
① 通电复位 → CPU 进入 16 位实模式，运行 BIOS 固件
      ↓
② 简易硬件初始化（仅基础设备）
      ↓
③ 读磁盘 0 磁道 0 扇区（前 512 字节），校验末尾魔数 0xAA55
      ↓
④ 512B 引导扇区（IPL）内极少量代码，再读「第二阶段」引导
      ↓
⑤ 第二阶段解析磁盘、读内核 — 无统一文件系统/路径规范
```

| 步骤 | 说明 |
|------|------|
| **512B 限制** | 引导扇区 **只能写很少汇编** — [02 30days-os](../../02-30days-os/day-01-boot-asm/) 启蒙即此 |
| **无 FAT 路径** | **不识别文件系统** — 只有 **LBA + 扇区** 裸读写 |
| **兼容性** | 老机器普遍支持；**效率低、流程不统一** |

→ 详表 [§1.四 BOOTX64 vs IPL](./section-1-本章定位.md#四核心区分bootx64efi--软盘启动两条线)

### 三、BIOS 与 UEFI 启动 · 四大核心区别

| # | 维度 | **BIOS（传统）** | **UEFI（MikanOS 主线）** |
|---|------|------------------|---------------------------|
| **1** | **CPU 运行模式** | 开机 **16 位实模式**，寻址 **≈1MB** 瓶颈 | 开机 **x86_64 长模式**，可访问 **全部物理内存**（仍受固件 map 约束） |
| **2** | **找引导程序** | **无视文件系统**，只读 **磁盘最前 512B 扇区** | **FAT 文件系统** + 固定路径 **`/EFI/BOOT/BOOTX64.EFI`** |
| **3** | **文件系统** | **无** — 裸扇区概念 | **强制 ESP 为 FAT32** — 有文件/文件夹 |
| **4** | **自制 OS 开发门槛** | 大量 **16 位汇编** 挤在 512B | **C 语言** + Clang/`lld-link` → `.efi`，**MikanOS 采用此方案** |

**HFT 读法：** 共置/服务器几乎全走 **UEFI → boot loader → 长模式内核**；BIOS 软盘链是 **历史启蒙**，不是生产主线。

### 四、配套关键名词

| 名词 | 一句话 |
|------|--------|
| **UEFI** | Unified Extensible Firmware Interface — **替代老旧 BIOS** 的可扩展固件；固化在主板 Flash，提供 **Boot/Runtime Services** 等标准接口 |
| **FAT / FAT32** | 简单跨平台文件系统；UEFI 规范要求 **EFI 系统分区** 常用 FAT32，固件才能按路径打开 **`BOOTX64.EFI`** |
| **BOOTX64.EFI** | x86_64 **默认 UEFI 引导应用** 文件名；你用 **C + `lld-link`** 编译的 **PE 产物**，放对路径才被加载 |
| **EfiMain** | UEFI 应用的 **C 入口** — 等价于用户态的 `main`，参数为 `ImageHandle` + **`SystemTable`** |
| **OVMF + QEMU** | **OVMF** = QEMU 用的开源 UEFI 固件；`-bios OVMF_CODE.fd` + `fat:rw:esp` **完全模拟** 真实 UEFI 启动，无需实体机 |

→ OVMF 命令 [SETUP.md](../../SETUP.md) · [§3](./section-3-真机与QEMU测试.md)

### 五、本章程序在链中的位置

```
[ 硬件 ]
[ UEFI 固件 ]              ← 主板 ROM，非你编写
[ BOOTX64.EFI / Hello ]    ← 本章 — EfiMain + ConOut
[ MikanLoader / 内核 ]     ← Ch2+
```

### 口述巩固 · 自测

1. **UEFI 去哪找引导文件？** — **FAT ESP** 上固定路径 **`/EFI/BOOT/BOOTX64.EFI`**，不是 512B 扇区。
2. **BIOS 传统启动认文件系统吗？** — **不认**，只执行 **0 扇区 IPL**（校验 **0xAA55**）。
3. **MikanOS 为何用 UEFI？** — **64 位长模式 + C/LLVM 开发**，跳过实模式扇区汇编。
4. **QEMU 如何模拟 UEFI？** — **OVMF 固件** + FAT 虚拟盘挂载含 `BOOTX64.EFI` 的目录。
