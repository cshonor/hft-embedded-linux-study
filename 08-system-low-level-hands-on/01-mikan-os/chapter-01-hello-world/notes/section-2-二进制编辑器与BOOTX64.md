## 2. 二进制编辑器构建 BOOTX64.EFI

> **最初尝试：** 不使用任何高级语言 — 直接用 **二进制编辑器** 编写含 **机器语言 + 数据** 的可执行文件。
>
> **不是软盘 IPL：** 这里编的是 **UEFI 规范下的 64 位 `.efi` 程序**，由固件从 FAT 分区 **`/EFI/BOOT/BOOTX64.EFI`** 路径加载执行 — 与 [02 Day 1](../../02-30days-os/day-01-boot-asm/) 软盘 **512 字节引导扇区** 是 **两条完全不同的启动机制**（详表 [§1.四](./section-1-本章定位.md#四核心区分bootx64efi--软盘启动两条线)）。

---

### 一、产出物

| 文件 | 说明 |
|------|------|
| **`BOOTX64.EFI`** | x86-64 UEFI **引导应用**（符合规范的 **64 位 PE/COFF 程序**） |
| **职责** | 固件加载后 **直接在你的代码里跑** — 后续章节由 Loader **加载 64 位内核**，无需先经历 16 位实模式 |
| **内容** | CPU 指令字节 + 字符串等 **数据段**（如 "Hello, world!"） |

编辑器中输入的 **十六进制** = 直接写入文件的 **机器码与常量** — 与 [01 Day 1](../../02-30days-os/day-01-boot-asm/notes/section-1.2-究竟做了些什么.md)「CPU 只认 0/1」同一原理。

---

### 二、部署与引导测试

```
FAT 格式 U 盘
└── EFI/
    └── BOOT/
        └── BOOTX64.EFI    ← UEFI 标准 x64 引导文件名
```

| 要点 | 说明 |
|------|------|
| **文件系统** | **FAT32**（常见 U 盘格式） |
| **路径固定** | UEFI 固件按规范查找 **`/EFI/BOOT/BOOTX64.EFI`** |
| **架构** | **x86-64** — 文件名中的 `X64` 表示 64 位 EFI |

插入 U 盘、从 USB 启动 → 若文件正确，固件加载并执行 → 屏幕出现 Hello World。

---

### 三、为何从二进制开始

| 原因 | 说明 |
|------|------|
| **消除抽象层** | 看见「程序 = 文件里的字节序列」 |
| **理解 EFI 体积** | 比 **512B IPL** 复杂 — 含 **PE 头** 等结构（见 [6. 文件格式](./section-6-C语言过渡与文件格式.md)）；**体积大换来的是规范化的 64 位入口** |
| **体会维护成本** | 手写机器码 **极难扩展** → 自然过渡到 C + 编译器 |

→ 十六进制对照可参考 [01 HELLOOS_HEX_REFERENCE](../../02-30days-os/HELLOOS_HEX_REFERENCE.md) 的读 hex 方法（内容不同，读法相同）

---

### 四、常见初坑

| 现象 | 可能原因 |
|------|----------|
| U 盘启动无输出 | 路径非 `/EFI/BOOT/BOOTX64.EFI`、非 FAT、或 **Secure Boot** 拦截 |
| 乱码 / 崩溃 | 机器码错误、**小端** 字节序写反（见 [4. 编码](./section-4-计算机结构与编码.md)） |
| 真机与 QEMU 不一致 | 固件版本、OVMF vs 实体 BIOS 差异 — 见 [3. 测试](./section-3-真机与QEMU测试.md) |
